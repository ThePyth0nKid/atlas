### V1.13 capabilities (shipped — full)

#### V1.13 Scope C Wave 1 — independent witness cosignature attestor (commits `a06e5ab` + `8410064`)

A second, independent attestor (the "witness") signs over the recomputed Atlas anchor-chain head with its own Ed25519 keypair. The verifier resolves the witness signature against `ATLAS_WITNESS_V1_ROSTER` (a pinned slice of `(witness_kid, [u8; 32])` pubkeys living in `crates/atlas-trust-core/src/witness.rs`). Trust-domain separation: the witness runs as the `atlas-witness` binary in a separate process from the `atlas-signer` — this is the entire point. A witness in the same process as the signer is "the same machine signed twice", not an independent attestation.

**New crate `atlas-witness`** (`crates/atlas-witness/`):
- `src/lib.rs` — `Witness` trait (dyn-safe, `Send + Sync`, `String` errors at boundary per V1.11 footgun #20).
- `src/ed25519_witness.rs` — `Ed25519Witness` impl. `Zeroizing<[u8; 32]>` secret wrapper; pubkey derived lazily on each call (no permanent in-memory copy of verifying-key bytes).
- `src/main.rs` — `atlas-witness` binary with two subcommands: `gen-key` and `sign-chain-head`. Secret file created **atomically** with `OpenOptions::create_new(true).mode(0o600)` on Unix (V1.13 footgun #36).
- `Cargo.toml` — depends on `atlas-trust-core` only (no cycle, V1.10 footgun #13).

**`atlas-trust-core` additions** (commit `a06e5ab`):
- `src/witness.rs` — `WitnessSig`, `ATLAS_WITNESS_DOMAIN`, `ATLAS_WITNESS_V1_ROSTER` (genesis-empty), `verify_witness_against_roster` + `verify_witnesses_against_roster` returning `WitnessVerifyOutcome { presented, verified, failures }`.
- `src/anchor.rs` — `AnchorBatch.witnesses: Vec<WitnessSig>` (backward-compat default empty). `ChainHeadInput` projection EXCLUDES `witnesses` from the head (prevents fixed-point regress). Compile-time enforcement via exhaustive destructure + `#[deny(unused_variables)]` (V1.13 footgun #33).

### V1.14 Scope I — HSM-backed witness deliverables

**I-a — `Pkcs11Witness` impl.**
- New module `crates/atlas-witness/src/hsm/` (sibling to `ed25519_witness.rs`).
- Single-feature gate: `--features hsm` on `atlas-witness` (mirrors `atlas-signer --features hsm`).
- Reuse `cryptoki` crate (already a dependency of `atlas-signer`). NOTE: V1.10 footgun #13 — Cargo no-cycles rule. The PKCS#11 module lives inside `atlas-witness`, NOT in a sibling crate. Reuse via direct `cryptoki` dep, not via `atlas-signer` re-export.
- Generation: `CKM_EC_EDWARDS_KEY_PAIR_GEN` per witness, single per-token pubkey (no per-workspace key derivation — witness is one-cosigner-per-host by design). Public-key handle pinned at session-open, scalar handle is `EXTRACTABLE=false`.
- Signing: `CKM_EDDSA(Ed25519)` over `ATLAS_WITNESS_DOMAIN || chain_head_bytes`. Same domain prefix as the file-backed witness — pinned by integration test (`hsm_witness_byte_equivalence`).
- Drop order matters (V1.10 footgun #15): `Mutex<Session>` BEFORE `_ctx: Pkcs11` so `C_CloseSession` runs before `C_Finalize`.
- Zeroize at every heap-resident copy of secret material (V1.10 footgun #17): PIN bytes (`Zeroizing<Vec<u8>>`), the (impossible) extracted scalar buffer if a future `C_Unwrap` path ever exists.

**I-b — CLI surface (`atlas-witness sign-chain-head --hsm`).**
- New flag `--hsm` mode-switch on the `sign-chain-head` subcommand. When set, requires the `ATLAS_WITNESS_HSM_*` trio (lib + slot + pin-file). When unset, the file-backed `--secret <path>` path is unchanged (V1.13 wave-C-1 behaviour preserved).
- Mirror the V1.11 `atlas-signer --features hsm` env-var pattern: `ATLAS_WITNESS_HSM_PKCS11_LIB`, `ATLAS_WITNESS_HSM_SLOT`, `ATLAS_WITNESS_HSM_PIN_FILE`. Fail-closed early diagnostic if any is missing — `(N/3 vars set)` shape (V1.12 footgun #26).
- The `Pkcs11Witness` impl ALSO requires a `witness_kid` parameter — same shape as the file-backed witness, the kid is a CLI argument, not derived from any HSM identifier.
- `gen-key` does NOT support `--hsm` — HSM key generation happens via `pkcs11-tool` in the operator runbook ceremony (matches the V1.11 wave-3 pattern: master seed import is a runbook step, not a CLI subcommand).

**I-c — Operator runbook ceremony (RUNBOOK §10 sub-section or §11).**
- Mirror the §wave-3 master-seed-import structure. Steps:
  1. Operator initialises a SoftHSM2 (or vendor HSM) token using `pkcs11-tool --init-token --pin-source env:ATLAS_WITNESS_HSM_PIN` on a hardened host (separate from the atlas-signer host AND separate from any existing master-seed token — the witness must be in a disjoint compromise surface).
  2. Generate the witness Ed25519 keypair: `pkcs11-tool --keypairgen --key-type EC:Ed25519 --label "atlas-witness:org-name:host-id"`.
  3. Read the public key out: `pkcs11-tool --read-object --type pubkey --label "atlas-witness:org-name:host-id" | atlas-witness extract-pubkey-hex`. (NEW subcommand — not the same as `gen-key`.)
  4. Operator pastes the hex pubkey into `crates/atlas-trust-core/src/witness.rs::ATLAS_WITNESS_V1_ROSTER` (same as V1.13 wave-C-2 step 3) and bumps the crate version (V1.7 boundary rule).
  5. Issuer's batch-emission code starts attaching `WitnessSig`s by calling `atlas-witness sign-chain-head --hsm` per batch.
- Caveats section: SoftHSM2 PIN-argv exposure (V1.13 footgun #23 carry-forward); HSM-witness rotation via `pkcs11-tool --delete-object`; co-existence of file-backed and HSM-backed witnesses in the same roster (different kids, both verify against `ATLAS_WITNESS_V1_ROSTER` — useful during migration).

      pub witness_kid: String,
      pub batch_index: Option<u64>,

      DuplicateKid,
      CrossBatchDuplicateKid,
      InvalidSignatureFormat,
      InvalidSignatureLength,

1. **`anchor-chain.jsonl` is append-only and irreversible** — sole writer is the issuer. (V1.7)
2. **Chain head canonicalisation must be byte-identical between issuer and verifier.** Pinned golden tests. (V1.7)
3. **Verifier may NOT trust `trace.anchor_chain.head` field** — recompute. (V1.7)

5. **Every signer-stdout boundary in TS uses `parseAnchorJson`** (lossless-json). (V1.8)
6. **`cargo` not in default PATH on Windows** — `/c/Users/nelso/.cargo/bin/cargo.exe`.
7. **Smoke at `apps/atlas-mcp-server`** — `pnpm --filter atlas-mcp-server smoke` from root.
8. **Single canonicalisation source — TS never re-implements anything cryptographic.** Always shell to `atlas-signer` (or `atlas-witness`). (V1.5+)
9. **HKDF info parameter is the trust boundary for per-tenant key separation.** (V1.9)
10. **Backward compat lenient mode is a trust trade-off, not a free win.** Strict mode is the security boundary. (V1.9) Now applies to `--require-witness` too. (V1.13 wave C-2)
11. **Don't re-derive a pubkey at verify time.** Verifier reads from the bundle (or roster). (V1.9 + V1.13)
12. **`std::env::set_var` in tests races on multi-core CI.** Use the V1.10 injection-form gates. (V1.10)
13. **Cargo no-cycles rule trumps clean module decomposition.** HSM lives in `crates/atlas-signer/src/hsm/` behind `--features hsm`, not in a sibling crate. **V1.14 Scope I: HSM lives in `crates/atlas-witness/src/hsm/`, NOT a sibling `atlas-witness-hsm` crate.** (V1.10 + V1.14 application)
14. **PKCS#11 is verbose; cryptoki abstracts the syntax but not the semantics.** Test against SoftHSM2 with pinned-pubkey goldens before trusting any vendor's HKDF/EDDSA. (V1.10)

16. **Absolute-path guard at config-parse time, not at use time.** Platform-conditional test data on Windows vs Unix. (V1.10)
17. **Zeroize at every heap-resident copy of secret material.** PIN bytes, HKDF output, per-tenant seed array — all wrapped. (V1.10) **V1.14 Scope I: PIN bytes for `ATLAS_WITNESS_HSM_PIN_FILE` get the same treatment.**
18. **Operator runbook ceremonies must use secrets-manager env vars, not literal PINs.** `umask 077`, source from secrets manager, `unset` at end. wave-3 ceremony (V1.11) further upgrades this to `--pin-source env:VAR` for `pkcs11-tool` so even argv exposure is closed. (V1.10/V1.11) **V1.14 Scope I: same rule for `ATLAS_WITNESS_HSM_PIN`.**
19. **Pre-flight test work pays for itself.** The V1.11 byte-equivalence test was instrumental in committing to wave-3's design. **V1.14 Scope I: write `hsm_witness_byte_equivalence` first.** (V1.11)
20. **Trait dyn-safety is non-trivial and worth pinning explicitly.** `WorkspaceSigner` is `Send + Sync` + dyn-safe — no `async fn`. Same is true of `Witness` (V1.13 wave C-1). (V1.11 + V1.13)
21. **Layered fail-closed dispatchers need explicit refusal text per layer.** Each refusal site names the env var to fix. (V1.11)
22. **`derive-key` style "structurally refused" subcommands are a real defensive pattern.** Refuse at the CLI surface, not at the FFI layer. (V1.11)
23. **PIN-on-argv is unavoidable on some legacy CLIs (softhsm2-util's init-token).** Document honestly + recommend host-hardening. Don't conflate shell history with `/proc/<pid>/cmdline`. (V1.11) Comment expansion in V1.13 docs-commit `01fa0a4` makes this explicit in the workflow files.

26. **Fail-closed early at the closest boundary to the misconfiguration.** (V1.12 Scope B2 — `smoke.ts` partial-trio guard) **V1.14 Scope I: `--hsm` without complete `ATLAS_WITNESS_HSM_*` trio fails-closed in the Rust CLI BEFORE any cryptoki context is opened.**

29. **Live-API CI lanes must be decoupled from PR triggers.** (V1.12 Scope B2 — `sigstore-rekor-nightly.yml`)

31. **One project-wide base64 dialect, no fallback.** `URL_SAFE_NO_PAD`. (V1.13 wave C-1)
32. **Inline crypto-primitive call to keep error semantics correct.** `verify_witness_against_roster` calls `VerifyingKey::verify_strict` directly, not via `crate::ed25519::verify_signature`. (V1.13 wave C-1)
33. **Compile-time enforcement against silent infinite-regress.** `ChainHeadInput` projection + exhaustive destructure + `#[deny(unused_variables)]`. (V1.13 wave C-1)

### NEW for V1.14 (V1.13 wave C-2 lessons)

37. **Newtype wrappers around byte-shaped strings with implicit invariants are worth the extra type.** `ChainHeadHex` wraps the 64-char lowercase-hex chain head; the strict constructor is the single source of truth for the head-shape contract; `decode_chain_head` delegates to it; the per-batch verifier construct-site has a `debug_assert` pinning the contract. Without the newtype, a future refactor that switches the underlying type from hex to raw bytes (or shortens to 32 chars) would compile-pass but be silently wrong everywhere it's consumed. (V1.13 wave C-2 — `crates/atlas-trust-core/src/anchor.rs::ChainHeadHex`)

38. **Encapsulation enforces invariants that documentation can only request.** `WitnessFailure.batch_index` was first-draft `pub`, which silently allowed external callers to read `None` from per-batch verifier output without any compile-time signal. Promotion to `pub(crate)` field + public `batch_index() -> Option<u64>` getter forces every external read through a single API surface where the absence-vs-presence semantic is doc-explicit. The HIGH-2 finding from the C-2 reviewer was that "the invariant is doc-only" — wave-C-2 fixed it structurally. (V1.13 wave C-2 — `crates/atlas-trust-core/src/witness.rs::WitnessFailure`)

39. **Sanitisation helpers are testable; raw inline guards are not.** `sanitize_kid_for_diagnostic` is a small `pub(crate)` helper that clamps oversize kids to a placeholder before they land in any `WitnessFailure`. The first-draft impl inlined the same logic at every error site, which made it impossible to test the boundary behaviour (where do we clamp? what's the placeholder shape?) in isolation. Extracting the helper let three boundary tests pin the behaviour: passthrough at-or-below cap, clamp above cap, off-by-one. SEC-MED-1 from the C-2 reviewer drove this extraction. (V1.13 wave C-2 — `crates/atlas-trust-core/src/witness.rs::sanitize_kid_for_diagnostic`)

40. **Test-only parameterisation is fine; production-pinned-roster discipline is doc-enforced.** `verify_witness_against_roster` and `verify_witnesses_against_roster` take a free `roster` parameter so unit tests can drive fail-closed branches with ad-hoc test rosters. The "production callers MUST pass `ATLAS_WITNESS_V1_ROSTER`" discipline is doc-only — there's no compile-time enforcement that production code can't pass a different roster. The wave-C-2 docs-commit (`01fa0a4` LOW #3 fix) makes the discipline doc-explicit; reviewers should reject any production call site passing a non-pinned roster. A future V1.x could enforce this at the type level by introducing a `ProductionRoster` newtype that only `ATLAS_WITNESS_V1_ROSTER` can construct — currently weighed as too heavy for the value, but the option is on the table. (V1.13 wave C-2 — `crates/atlas-trust-core/src/witness.rs` doc additions)

41. **Strict-mode threshold check site uses the named lenient helper, NOT inline-filter.** `outcome.has_witness_failures()` (shipped in wave C-1) is the named API for "did any witness evidence row have ok=false". The strict-mode wiring in wave C-2 calls this helper at the threshold check site; it does NOT re-walk the chain or write inline `evidence.iter().any(|e| e.check == "witnesses" && !e.ok)`. Single source of truth for the question, single place to change if the evidence row shape ever evolves. (V1.13 wave C-2 — `crates/atlas-trust-core/src/verify.rs::apply_witness_threshold`)

42. **Extract a pub(crate) helper for both production and unit tests when the production driver is hard to fake end-to-end.** `verify_trace_with` hard-wires `ATLAS_WITNESS_V1_ROSTER` (genesis-empty), making the strict-mode passing path untestable end-to-end. The C-2 MEDIUM-2 fix extracted `apply_witness_threshold(verified, threshold, evidence, errors)` as a `pub(crate)` helper. Production code calls it; the unit test uses `aggregate_witnesses_across_chain_with_roster` (also `pub(crate)`-extracted) to build a real chain that produces `verified=N` against a parameterised test roster, then drives the same helper. Both production code AND the test exercise the SAME function — no risk of test passing while production is broken. (V1.13 wave C-2 — `crates/atlas-trust-core/src/verify.rs`)

43. **Workflow-comment expansion has high security ROI when a defence is non-obvious from the code.** The V1.12 LOW carry-over closed in `01fa0a4` added a multi-line comment to two CI workflows explaining that `softhsm2-util --init-token --pin "$PIN"` exposes the PIN via `/proc/<pid>/cmdline` orthogonal to the existing `set +x` shell-trace defence. Without the expanded comment, a future editor who sees only `set +x` could remove it thinking it's the only PIN-leak path. The expanded comment names BOTH defences and explicitly states "removing `set +x` does NOT close this orthogonal hole — both defences are needed independently." Comments that name what is NOT defended are sometimes more load-bearing than comments naming what IS. (V1.13 docs-commit — `.github/workflows/hsm-byte-equivalence.yml` + `hsm-wave3-smoke.yml`)

44. **The `~Nms` window estimate is fragile; the architectural bound is durable.** Same workflow-comment expansion: first-draft said "exposure window typically <100ms in CI" — security-reviewer L-1 flagged this as an unsourced timing estimate that becomes wrong on heavily-loaded shared runners. The fix replaced the time estimate with the architectural bound: "window closes only when the one-shot process exits". True regardless of runner speed; doesn't decay if CI infra changes. Apply this rule to every "this is bounded" claim in security docs. (V1.13 docs-commit reviewer fix)

45. **Phantom function names in cross-reference docs are a real defect class.** Both reviewers (independently) caught the same HIGH in the V1.13 docs-commit: my new "Production callers MUST pass `ATLAS_WITNESS_V1_ROSTER`" doc on `verify_witness_against_roster` referenced a phantom function `witness_evidence_for_chain` (a name from the wave-C-1 handoff that never matched the actual code — the actual function is `aggregate_witnesses_across_chain` → `witness_evidence_from_aggregate`). The doc's security intent (explaining why a different roster would silently widen the trust surface) was sound, but a reviewer following the call-chain reference would have failed to find it, defeating the doc's entire purpose. **Always verify intra-doc names against the actual symbol table** — `cargo doc --no-deps` warnings catch broken intra-doc links; rely on them. (V1.13 docs-commit — `01fa0a4` reviewer fix)
