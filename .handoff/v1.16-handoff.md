**Design decision tree (key tradeoffs ‚Äî Welle C):**
- **Cloudflare Workers + Static Assets vs Pages vs Netlify vs GitHub Pages:** chose Workers + Static Assets. Pages is the older path (Cloudflare's static-only product); Workers + Static Assets is the 2026 first-class hosting model that allows arbitrary Worker code AHEAD of the static-asset binding via `experimental_serve_directly = false` (== `run_worker_first = true` in wrangler ‚â•4). This is what makes layered Worker-emitted security headers possible ‚Äî without it, the Cloudflare edge would serve assets directly and bypass the Worker entirely. GitHub Pages and Netlify don't expose a code path between the request and the static asset at all ‚Äî they would force the receiver to be on a separate origin, killing the same-origin `report-uri` advantage.
- **One Worker for everything vs separate Workers for hosting + receiver + cron:** chose single Worker. The receiver is ~250 lines, the security-header layer is ~80 lines, the cron handler is ~40 lines. Separating them would multiply auth/binding complexity (each Worker would need its own AE/R2/DO bindings + its own deploy lifecycle) while saving zero runtime cost (Workers cold-start is ~5ms regardless of code size up to ~1MB).
- **Workers Analytics Engine + R2 daily heartbeat vs raw R2 per-report vs KV vs D1:** chose AE+R2. Per-report R2 PUT is a financial-DoS vector ‚Äî R2 Class-A operations are billed per-request, and an attacker spamming `/csp-report` could rack up costs even with rate-limiting in place. AE `writeDataPoint` is billed per-datapoint not per-Class-A-op (‚âàfree) AND queryable via SQL from the dashboard. D1 is a SQLite store with row-level Class-A billing ‚Äî same financial-DoS concern. KV has eventual consistency that makes it inappropriate for ingestion. AE handles the hot path; R2 is only touched once per day for archive ‚Äî the cron writes one PUT/day with a heartbeat marker (Welle D candidate: extend the cron to query AE and batch the day's datapoints into the daily R2 object).
- **Durable Object rate-limit vs KV-counter vs CF Rate-Limiting Rules:** chose DO. KV-counter has eventual consistency ‚Äî a coordinated attacker hitting different POPs could massively over-shoot the limit before counters converge. CF Rate-Limiting Rules (the WAF feature) sits ABOVE the Worker and would silent-block requests before the receiver gets to silent-204 them ‚Äî splitting the silent-204 invariant across two layers. DO is single-isolate single-instance globally addressable by `idFromName("global")`, gives strict consistency, and the rate-limit logic stays inside the receiver's silent-204-or-accept boundary. Latency cost: ~5-15ms typical for the DO round-trip, which is fine for a `/csp-report` endpoint where the caller is already waiting on the browser's report POST anyway.
- **IPv6 /64 vs /128 prefix-based rate-limit key:** chose /64. A single attacker on a /64 has 18 quintillion addresses to rotate through ‚Äî /128-keying would let them defeat the per-prefix limit trivially. /64 matches the typical residential ISP allocation and the IPv6 RFC-recommended subnet-delegation boundary; an attacker who can rotate ACROSS /64s is by definition a multi-network botnet, which the global cap (1000/min total) defends against.
- **`safeJsonParse` DEFAULT_LIMITS vs per-receiver tighter limits:** chose to keep DEFAULT_LIMITS (10/100/8192) as the generic-utility default and pass tighter `RECEIVER_PARSE_LIMITS` (4/24/8192) at the receiver call site. Real CSP reports are flat ~12-key objects at depth-2 (legacy) or depth-3 (modern); 2√ó headroom is enough. Tighter caps reduce the parser's worst-case work per accepted payload (defence-in-depth on top of the body-size cap).
- **Method-enforcement at receiver vs router:** chose router-only. Defence-in-depth at the receiver function would mean a 405 reply for non-POST, which gives the caller an oracle to distinguish "wrong method" from "wrong Origin / wrong CT / wrong shape" ‚Äî breaking the silent-204 invariant. The router (`index.ts`) already only routes POST to the receiver, and a non-POST falls through to the static-asset binding (which 404s for `/csp-report`). Removed the unreachable 405 guard + the 3 dead method-enforcement tests in csp-receiver.test.ts.
- **Cron archive: heartbeat-marker only vs full AE‚ÜíR2 batch:** chose heartbeat-only for Welle C. The goal of getting the cron path in place is to lock in the daily-PUT cadence (R2 lifecycle rule operates on object-modification time, so once-per-day writes are required), the operator-side R2 lifecycle policy (delete > 365 days), and the binding wiring (R2 bucket + cron trigger). **Design decision tree (key tradeoffs):**
- **Hash-CSP vs nonce-CSP vs external-script + SRI:** chose external-script + SRI. Hash-CSP requires a build step to compute and inject the script hash on every edit (poor fit for static-hosting); nonce-CSP requires the hosting provider to mint a per-request nonce (forces server-side rendering); external-script + SRI works against any plain static host (`npx serve`, `python3 -m http.server`, S3, GitHub Pages, Cloudflare Pages, Netlify) with the hash maintained by `tools/playground-csp-check.sh --update-sri`.
- **`<meta http-equiv>` CSP delivery vs HTTP header:** chose meta-tag for the page-bytes-portability win; documented `frame-ancestors` is browser-ignored via meta-tag (header-only), and the validator emits a WARN to remind operators that clickjacking protection still needs a real header at deploy time. Defence-in-depth either way: a correctly-configured host SHOULD also send the same policy as a header.
- **`'wasm-unsafe-eval'` vs `'unsafe-eval'`:** `'wasm-unsafe-eval'` is the dedicated CSP3 keyword for `WebAssembly.compile`/`instantiate` and does NOT enable JavaScript `eval()`/`new Function()`. Hardening win without the JS-eval cost.
- **Strict TT (`require-trusted-types-for 'script' + trusted-types 'none'`) vs default-policy:** chose strict (no policy can be created). The application is sink-free by construction (textContent / className / style.display only), so the strictest setting is the load-bearing one ‚Äî any future regression that re-introduces a sink fails at the browser boundary, not at code-review time. This required confirming wasm-bindgen-emitted glue is also TT-compatible (see below).
- **wasm-bindgen TT-compatibility:** `pkg/atlas_verify_wasm.js` is gitignored, so the audit can't be a static file check. Solved by (a) building once at audit time (`wasm-pack build crates/atlas-verify-wasm --target web --release --out-dir ../../apps/wasm-playground/pkg`), (b) auditing the emitted glue for TT-protected sinks (eval / new Function / innerHTML / outerHTML / document.write / insertAdjacentHTML / setTimeout(string) / setInterval(string) / *.text =), (c) confirming clean at wasm-bindgen 0.2.118, (d) codifying the audit as part of the validator (`PKG_GLUE` check runs whenever `pkg/` is present). Future wasm-bindgen upgrades MUST re-run the validator to confirm continued TT-compat.
