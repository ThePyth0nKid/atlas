**Why this matters (the operational-resilience argument):** `resolveSignerBinary` resolves the `atlas-signer` Rust binary path on first call and caches the result. Pre-Welle-4 the cache was process-long: a long-running consumer (Claude Desktop's MCP host that lives across days, the production atlas-web Next.js server, a Vercel/Cloudflare worker tier) would pin whatever path was valid at startup. If the operator ran `cargo build --release` to ship a new signer build between consumer restarts (and the new binary replaced the old one in-place, OR the prior cached path was the debug build that got `cargo clean`'d), every subsequent sign call ENOENT'd until an operator noticed and bounced the host. Atlas's whole trust contract assumes the signer is reachable; a stale cache silently broke this for hours. Welle 4 bounds the drift window to 60s ŌĆö within that window the cache wins and there are no syscalls; after expiry resolution rebuilds from scratch. The amortised cost is at most one `existsSync` per minute per process regardless of write rate, preserving the "no per-write existsSync" property the original cache existed to provide.

**What landed in `packages/atlas-bridge/src/paths.ts`:**
- New constant `SIGNER_BINARY_CACHE_TTL_MS = 60_000` documenting the operational rationale (long-running consumers, hot-path syscall budget, both-direction TTL).
- New interface `SignerBinaryCacheEntry { readonly path: string | null; readonly expiresAt: number }` ŌĆö readonly fields, immutable assignment via object-literal construction.
- `signerBinaryCache: SignerBinaryCacheEntry | null` replaces the old `cachedSignerBinary: string | null | undefined`. Same module-level binding, but each cache hit/miss creates a new entry object rather than mutating.
- New module-level `signerCacheNow: () => number = Date.now` ŌĆö clock-injection seam for the test. Production callers always observe `Date.now`.
- `resolveSignerBinary` rewritten: reads `now = signerCacheNow()`, returns `signerBinaryCache.path` if `expiresAt > now`, else calls `doResolveSignerBinary()` and stores `{ path: resolved, expiresAt: now + TTL_MS }`.
- New private `doResolveSignerBinary()` extracted from the old body ŌĆö same env-var-then-release-then-debug preference order, same `existsSync` guard on env-var override (Welle 1 web hardening, preserved).
- **Security-reviewer follow-up (closed in-PR):** `ATLAS_SIGNER_PATH` is now `path.resolve()`'d to absolute BEFORE `existsSync` + cache. The TTL means the resolved path is reused for up to 60s; if a relative path were stored and the long-running process changed cwd in the meantime (`process.chdir`, or a worker thread with a different inherited cwd), the cached probe and the later spawn would resolve against different bases. existsSync itself is cwd-relative ŌĆö the absolute resolve closes that gap and makes the cache value carry the same meaning across the whole TTL window regardless of cwd drift.
- New test seam `__signerBinaryCacheForTest = { TTL_MS, reset, setClock, restoreClock }` ŌĆö same `__`-prefix convention as `__signerLimitsForTest` in `signer.ts`. Cache stores only path strings, not key material ŌĆö no privilege-escalation surface; worst-case abuse is forcing every call to do a fresh existsSync, which is a self-inflicted performance regression on the caller.

**What landed in `packages/atlas-bridge/src/index.ts`:**

**What landed in `apps/atlas-mcp-server/scripts/test-signer-binary-cache.ts` (NEW):**

## 4. V1.19 Welle 1 ŌĆö implementation context (rationale, threat model, hardenings)

**State summary:** first user-facing write path. After 18 waves of verify-side and distribution-side hardening (V1.0 ŌåÆ V1.18), V1.19 pivots to product surface. A Browser-User can now create an Atlas node via `POST /api/atlas/write-node` + `/write` page (Next.js 16 App Router, React 19, TS-strict). Trust-Inkrement structurally identical to MCP-host writes ŌĆö same `atlas-signer` Rust binary, same HKDF-SHA256 workspace-key derivation, same append-only `events.jsonl` format ŌåÆ the Verifier cannot distinguish web-emitted from MCP-emitted events because they're byte-identical at the AtlasEvent envelope layer.

**Threat model & deliberate non-goals (from `route.ts:8-51` source comment):**

1. **NO authentication.** Atlas's trust-model is key-based (every event records `signature.kid`), not user-based. Deployer gates exposure at network/proxy layer ŌĆö see `OPERATOR-RUNBOOK ┬¦17`. Surfacing this route on a public address without an upstream auth gate IS the operator's choice and responsibility. Adding half-baked auth would be worse than none ŌĆö operators would assume security exists when it's only a thin wrapper.

2. **Per-tenant kid auto-derived from `workspace_id`.** Route NEVER accepts a caller-supplied `kid`. Eliminates "sign as somebody else" affordance the MCP tool intentionally reserves for trusted MCP hosts (Cursor, Claude Desktop). Deterministic via `perTenantKidFor(workspace_id) ŌåÆ "atlas-anchor:{workspace_id}"`.

3. **Single-process locking only.** Per-workspace mutex in `lib/atlas/event.ts` serialises writes within ONE Node process. Running atlas-web AND atlas-mcp-server against the same workspace concurrently CAN fork the DAG. Verifier accepts forks (it's a DAG), but the user's mental model breaks. Deploy one writer per workspace until V2 ships an external lock.

4. **`ATLAS_DEV_MASTER_SEED=1` must be set** in the Next.js process environment (or the wave-3 HSM opt-in), otherwise per-tenant subcommands of `atlas-signer` refuse to start. Surfaced as 500 with a clear message when missing.

**Web-specific hardenings (vs MCP variant):**

- **Body-size + attributes cap (defence-in-depth):** `REQUEST_BODY_MAX_BYTES = 256 KB` (raw HTTP body, Content-Length pre-check before `req.json()` reads stream) + `ATTRIBUTES_MAX_BYTES = 64 KB` (Zod `.refine()` on JSON-serialised attributes). 64 KB is well within Linux ARG_MAX (~2 MB) ŌĆö payload passes to `atlas-signer` as `--payload <json>` argv string, so 1 MB attributes would E2BIG ŌåÆ generic 500. 64 KB gives clean Zod error at boundary instead.
- **Signer-subprocess timeout + stdout cap:** `SIGN_TIMEOUT_MS = 30_000`, `DERIVE_TIMEOUT_MS = 5_000`, `STDOUT_CAP_BYTES = 512 KB`. Settle-pattern with AbortController-style guard prevents double-resolve on timeout-vs-close race. clearTimeout in both close + error handler.
- **`redactPaths` on 500-responses:** strips absolute filesystem paths from error messages ŌĆö defends against path-disclosure via 500. Applied to `SignerError.message` and `unexpected: ${err.message}` paths. Defence-in-depth: signer should not write paths to stderr in the first place, but redacting here means we don't depend on that discipline. `StorageError` messages already path-sanitised at source (`storage.ts:sanitiseFsError`).
- **Per-workspace mutex Map-identity-race-closed (CRITICAL pre-resume fix):** `if (workspaceMutex.get(workspaceId) === next) workspaceMutex.delete(workspaceId)` ŌĆö identity comparison before deletion closes race where concurrent caller already set'd a newer tail and this callback would otherwise delete the newer tail from the map. **Applied to BOTH `apps/atlas-web/src/lib/atlas/event.ts` AND `apps/atlas-mcp-server/src/lib/event.ts`** for parity (and as preparation for V1.19 Welle 2 bridge-extraction).
- **Spread `attributes` FIRST, schema-checked fields win:** `{ ...args.attributes, kind: args.kind, id: args.id }` ŌĆö never let a caller-supplied attribute object overwrite the schema-checked node identity. Silent override would corrupt the signed payload.
- **`existsSync` guard on `ATLAS_SIGNER_PATH` cache:** if env-var path doesn't exist on disk, fall through to workspace candidates instead of caching an invalid path that would ENOENT-fail every subsequent sign.
