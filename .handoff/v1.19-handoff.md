**Why this matters (the two-surfaces-one-invariant argument):** prior to Welle 9, the `parents[0] === stored[i-1].event_hash` strict-linearity property — the structural invariant that a single-writer-per-workspace deployment MUST satisfy — was enforced only at the atlas-web HTTP test surface (`scripts/e2e-write-edge-cases.ts` [C.6], shipped in Welle 8). The Rust verifier `atlas-trust-core` accepted forked DAGs as valid wire shape. This was correct for the default Atlas threat model (Atlas is fundamentally a DAG; multi-process writers against the same workspace produce valid forks), but it left a gap: an auditor verifying an already-anchored release could not at the verifier surface detect that a deployment claimed to be single-writer but had a fork-DAG in its trace history. Welle 9 lifts the invariant into the verifier as an opt-in `--require-strict-chain` flag. Default behaviour is unchanged (lenient DAG); deployments that contractually guarantee single-writer (e.g. hosted SaaS with a workspace-mutex enforcement layer) can turn on strict mode and surface a structured `TrustError::StrictChainViolation { msg }` if the contract is ever silently breached. The flag's CLI doc names the trust assumption explicitly so an operator reading `--help` understands what they are pinning before they enable it.

**The five pinned properties (in implementation order):**
1. **Trace non-empty** — empty trace fails immediately. A linear chain requires at least 1 genesis event; an empty trace cannot satisfy that. SR-H-1 fix: prior implementation `if events.len() != 1 ...` style would have silently passed empty traces.
2. **No self-reference** — checked FIRST among per-event checks. An event whose `parent_hashes` contains its own `event_hash` is a 1-cycle. SR-H-2 fix: positioned ahead of the genesis-count check because a 1-event self-ref has `parent_hashes.len() == 1` (so `parent_count = 1`, NOT 0), which would have fired the genesis check first with the misleading "expected exactly 1 genesis event ... found 0" message instead of naming the actual cycle.
3. **Exactly one genesis** — exactly one event satisfies `parent_hashes.is_empty()`. Two genesis events ⇒ unparented sibling roots, not a chain. Zero genesis events ⇒ a cycle somewhere (every event refers back).
4. **Single-parent for non-genesis** — every non-genesis event has `parent_hashes.len() == 1`. Multi-parent events are valid DAG merges but break linearity.
5. **No sibling-fork** — no event hash is referenced as a parent by more than one other event. This is the "linear" property: each parent is referenced exactly once. Without this check, two non-genesis events could both list the same parent — a DAG fork that property (4) alone does not catch.

**What landed in `crates/atlas-trust-core/src/error.rs`:**
- New variant `StrictChainViolation { msg: String }` added to the `#[non_exhaustive]` `TrustError` enum after `BadAlg`. CR-2 fix: prior implementation used `Result<(), String>` which deviated from the module's `TrustResult<()>` convention. Now the strict-chain check returns a structured error like every other verifier failure, so auditor tooling can switch on `TrustError::StrictChainViolation { msg }` rather than parse the human-readable text. `Display` impl: `"strict-chain violation: {msg}"`. ## 13. V1.19 Welle 1 — implementation context (rationale, threat model, hardenings)

**State summary:** first user-facing write path. After 18 waves of verify-side and distribution-side hardening (V1.0 → V1.18), V1.19 pivots to product surface. A Browser-User can now create an Atlas node via `POST /api/atlas/write-node` + `/write` page (Next.js 16 App Router, React 19, TS-strict). Trust-Inkrement structurally identical to MCP-host writes — same `atlas-signer` Rust binary, same HKDF-SHA256 workspace-key derivation, same append-only `events.jsonl` format → the Verifier cannot distinguish web-emitted from MCP-emitted events because they're byte-identical at the AtlasEvent envelope layer.

**Threat model & deliberate non-goals (from `route.ts:8-51` source comment):**

1. **NO authentication.** Atlas's trust-model is key-based (every event records `signature.kid`), not user-based. Deployer gates exposure at network/proxy layer — see `OPERATOR-RUNBOOK §17`. Surfacing this route on a public address without an upstream auth gate IS the operator's choice and responsibility. Adding half-baked auth would be worse than none — operators would assume security exists when it's only a thin wrapper.

2. **Per-tenant kid auto-derived from `workspace_id`.** Route NEVER accepts a caller-supplied `kid`. Eliminates "sign as somebody else" affordance the MCP tool intentionally reserves for trusted MCP hosts (Cursor, Claude Desktop). Deterministic via `perTenantKidFor(workspace_id) → "atlas-anchor:{workspace_id}"`.

3. **Single-process locking only.** Per-workspace mutex in `lib/atlas/event.ts` serialises writes within ONE Node process. Running atlas-web AND atlas-mcp-server against the same workspace concurrently CAN fork the DAG. Verifier accepts forks (it's a DAG), but the user's mental model breaks. Deploy one writer per workspace until V2 ships an external lock.

4. **`ATLAS_DEV_MASTER_SEED=1` must be set** in the Next.js process environment (or the wave-3 HSM opt-in), otherwise per-tenant subcommands of `atlas-signer` refuse to start. Surfaced as 500 with a clear message when missing.

**Web-specific hardenings (vs MCP variant):**

- **Body-size + attributes cap (defence-in-depth):** `REQUEST_BODY_MAX_BYTES = 256 KB` (raw HTTP body, Content-Length pre-check before `req.json()` reads stream) + `ATTRIBUTES_MAX_BYTES = 64 KB` (Zod `.refine()` on JSON-serialised attributes). 64 KB is well within Linux ARG_MAX (~2 MB) — payload passes to `atlas-signer` as `--payload <json>` argv string, so 1 MB attributes would E2BIG → generic 500. 64 KB gives clean Zod error at boundary instead.
- **Signer-subprocess timeout + stdout cap:** `SIGN_TIMEOUT_MS = 30_000`, `DERIVE_TIMEOUT_MS = 5_000`, `STDOUT_CAP_BYTES = 512 KB`. Settle-pattern with AbortController-style guard prevents double-resolve on timeout-vs-close race. clearTimeout in both close + error handler.
- **`redactPaths` on 500-responses:** strips absolute filesystem paths from error messages — defends against path-disclosure via 500. Applied to `SignerError.message` and `unexpected: ${err.message}` paths. Defence-in-depth: signer should not write paths to stderr in the first place, but redacting here means we don't depend on that discipline. `StorageError` messages already path-sanitised at source (`storage.ts:sanitiseFsError`).
- **Per-workspace mutex Map-identity-race-closed (CRITICAL pre-resume fix):** `if (workspaceMutex.get(workspaceId) === next) workspaceMutex.delete(workspaceId)` — identity comparison before deletion closes race where concurrent caller already set'd a newer tail and this callback would otherwise delete the newer tail from the map. **Applied to BOTH `apps/atlas-web/src/lib/atlas/event.ts` AND `apps/atlas-mcp-server/src/lib/event.ts`** for parity (and as preparation for V1.19 Welle 2 bridge-extraction).
- **Spread `attributes` FIRST, schema-checked fields win:** `{ ...args.attributes, kind: args.kind, id: args.id }` — never let a caller-supplied attribute object overwrite the schema-checked node identity. Silent override would corrupt the signed payload.
- **`existsSync` guard on `ATLAS_SIGNER_PATH` cache:** if env-var path doesn't exist on disk, fall through to workspace candidates instead of caching an invalid path that would ENOENT-fail every subsequent sign.
