## 2. Prime directive — the trust property

> An auditor with the trace bundle and a fresh `atlas-verify-cli` binary can establish, **fully offline**, that every claim was bit-identical to what was signed and anchored.

This is the ONLY thing that matters. Every Phase 2 design decision filters through it:

- The verifier may NOT make a network call. Fixtures must be checked in.
- The signer (issuer) DOES make a network call to Rekor — but only at issue time, never at verify time.
- Whatever the signer captures from Rekor's response goes into `AnchorEntry` and is later replayed offline by the verifier.
- Any drift between issuer-derived bytes and verifier-recomputed bytes is a trust violation. Single-source the canonicalisation in Rust.

## 3. Phase 2 — `--rekor-url` issuer

### Goal

```
echo '<batch json>' | atlas-signer anchor --rekor-url https://rekor.sigstore.dev
```

…produces `AnchorEntry[]` whose entries verify under `default_trusted_logs()` exactly as the mock-issued entries do today, except `format = SigstoreRekorV1`.

When `--rekor-url` is **unset**, the existing mock path runs unchanged.

### Critical design question (settle FIRST before writing code)

Atlas event hashes are **BLAKE3** (`event_hash`, `pubkey_bundle_hash`). Rekor's `hashedrekord` schema only accepts `sha1/sha256/sha384/sha512`. The verifier already pins `algorithm == "sha256"` and `body.spec.data.hash.value == anchored_hash`. So `anchored_hash` for a Sigstore-format entry must be a **SHA-256 hex**, not a blake3 hex.

**Recommended design (A) — explicit prefix derivation:**

Add to `atlas-trust-core::anchor`:

```rust
/// Derive the SHA-256 anchored_hash for a Sigstore-format entry, from
/// the underlying Atlas blake3 hash and its kind. Domain-separated so
/// a dag_tip and a bundle_hash with the same blake3 value get distinct
/// Sigstore digests. Issuer and verifier MUST use this single function.
pub fn sigstore_anchored_hash_for(kind: &AnchorKind, blake3_hex: &str) -> String {
    use sha2::{Digest, Sha256};
    let prefix: &[u8] = match kind {
        AnchorKind::DagTip => b"atlas-dag-tip-v1:",
        AnchorKind::BundleHash => b"atlas-bundle-hash-v1:",
    };
    let mut h = Sha256::new();
    h.update(prefix);
    h.update(blake3_hex.as_bytes());
    hex::encode(h.finalize())
}
```

Then in `verify_anchors` (or `verify_anchor_entry`), for Sigstore-format entries, replace the simple `expected_hash == entry.anchored_hash` check with:

```rust
let expected_sigstore_hash = sigstore_anchored_hash_for(&entry.kind, expected_blake3_hash);
ct_eq_str(&entry.anchored_hash, &expected_sigstore_hash)
```

The issuer in `atlas-signer/src/anchor.rs` calls the same function before constructing the hashedrekord body. **Single source — drift is impossible.**

**Add a unit test** that pins the function output for both kinds against a known blake3 input. This is the bit-identical contract; pin it.

> Why prefix-derivation and not "sha256 of the canonical signing-input"? Because it requires the verifier to re-canonicalise the event/bundle content. That logic exists, but routing it through here doubles the surface for canonicalisation drift. Prefix-derivation is one tiny function shared by issuer + verifier. Choose simplicity unless there's a concrete reason not to.

### Wire format — Rekor REST API

#### POST /api/v1/log/entries

Request body — `hashedrekord` v0.0.1:

```json
{
  "apiVersion": "0.0.1",
  "kind": "hashedrekord",
  "spec": {
    "data": {
      "hash": {
        "algorithm": "sha256",
        "value": "<sigstore_anchored_hash hex>"
      }
    },
    "signature": {
      "content": "<base64-DER ECDSA signature OR base64 ed25519 signature>",
      "publicKey": {
        "content": "<base64 of PEM-encoded SPKI of the Atlas anchoring key>"
      }
    }
  }
}
```

> **Note on signature/publicKey**: hashedrekord requires both. The verifier does NOT validate the inner signature — Sigstore did at submit time. So the Atlas anchoring key can be any valid signing key. Recommendation: derive a deterministic dev key from a seed (`b"atlas-rekor-anchor-v1-dev-seed"`), document it, and make rotation a V1.7 concern. The signature signs the **artifact bytes**, which here = the prefixed-blake3 string `b"atlas-dag-tip-v1:" || blake3_hex` (or bundle-hash-v1). Sign with Ed25519 (matches existing `ed25519-dalek`); base64-encode the raw 64-byte signature for the `content` field.

Response — JSON object keyed by entry UUID (sha256 hex of canonical body):

```json
{
  "abc123...": {
    "body": "<base64 of canonical body>",
    "integratedTime": 1767784636,
    "logID": "c0d23d6a...",
    "logIndex": 800000000,
    "verification": {
      "inclusionProof": {
        "checkpoint": "<full multi-line C2SP signed-note text>",
        "hashes": ["<hex>", "<hex>", ...],
        "logIndex": 800000000,
        "rootHash": "<hex>",
        "treeSize": 1274638893
      },
      "signedEntryTimestamp": "<base64>"
    }
  }
}
```

#### Mapping response → `AnchorEntry`

```rust
AnchorEntry {
    kind,                                                  // from request
    anchored_hash: sigstore_anchored_hash,                 // from request
    log_id: response.logID,                                // verify equals SIGSTORE_REKOR_V1_LOG_ID
    log_index: response.logIndex,
    integrated_time: response.integratedTime,
    inclusion_proof: InclusionProof {
        tree_size: verification.inclusionProof.treeSize,
        root_hash: verification.inclusionProof.rootHash,
        hashes: verification.inclusionProof.hashes,
        checkpoint_sig: extract_signature_line(&verification.inclusionProof.checkpoint)?,
    },
    entry_body_b64: Some(response.body),                   // already STANDARD base64
    tree_id: Some(SIGSTORE_REKOR_V1_ACTIVE_TREE_ID),       // V1.6 only supports active shard
}
```

#### Parsing `checkpoint` (C2SP signed-note)

The `inclusionProof.checkpoint` field is the FULL signed-note. Format:

```
rekor.sigstore.dev - 1193050959916656506
1274638893
<base64(rootHash)>

— rekor.sigstore.dev <base64(4-byte keyID || DER ECDSA sig)>
```

The signature line starts with `— ` (em-dash + space, U+2014 followed by U+0020). After the em-dash + name, the third token is the base64 we need for `checkpoint_sig`. Extract it carefully:

```rust
fn extract_signature_line(checkpoint: &str) -> Result<String, String> {
    // Find the blank line separating signed body from signature lines.
    // Signature lines have the format: "— <name> <base64>".
    // We want the base64 of the signature line whose <name> is the log
    // origin (rekor.sigstore.dev). Per C2SP, the keyID embedded in the
    // base64 must match SIGSTORE_REKOR_V1_KEY_ID.
    // ...
}
```

**Defensive:** verify the extracted `checkpoint_sig` decodes to ≥ 4 bytes whose first 4 bytes equal `*SIGSTORE_REKOR_V1_KEY_ID`. The verifier will redo this check, but failing fast at issue time gives a clearer error than letting the verifier reject the trace later.

## 5. Footguns / hard-won learnings (don't re-discover)

1. **JS `parseInt` truncates `tree_id`** — `1193050959916656506` exceeds `Number.MAX_SAFE_INTEGER` (`9007199254740991`). When loading fixtures or response JSON in Node, parse the tree-ID as a STRING (or use `BigInt`). When writing fixture JSON, splice the tree-ID via regex string replacement, not `JSON.stringify({ tree_id: parseInt(...) })`.

2. **Base64 strictness on the Sigstore path** — Sigstore uses RFC 4648 §4 STANDARD base64 with `=` padding, period. The Phase 1 hardening removed the URL_SAFE_NO_PAD fallback. Don't re-add it. The mock path uses URL_SAFE_NO_PAD, also period.

3. **Tree-ID enforcement** — `verify_sigstore_rekor_v1` rejects any entry whose `tree_id != SIGSTORE_REKOR_V1_ACTIVE_TREE_ID`. V1.6 only trusts the active shard. If you need historical-shard support, that's V1.7 — add their origins to `default_trusted_logs` and lift the check.

4. **Origin separators** — `canonical_checkpoint_bytes_sigstore` rejects origins containing `\n` or ` - ` or empty. C2SP separators; bypass = signed-body injection.

5. **C2SP keyID is BIG-endian, prepended** — `base64_decoded_signature[..4] == SIGSTORE_REKOR_V1_KEY_ID = SHA-256(DER SPKI)[:4]`. Defends against signature lines authored by a different log.

6. **`cargo` not in default PATH on this Windows machine** — every shell invocation needs `export PATH="/c/Users/nelso/.cargo/bin:$PATH"`. The PowerShell tool inherits a different PATH than `bash`; prefer the `bash` Bash tool with the explicit export.

7. **Smoke is at `apps/atlas-mcp-server`, not root** — `pnpm --filter atlas-mcp-server smoke` from root, OR `cd apps/atlas-mcp-server && pnpm smoke`.

8. **Pinned hex values in tests** — three tests pin specific computed hashes (`sigstore_leaf_hash_is_rfc6962_sha256`, `sigstore_parent_hash_is_rfc6962_sha256`, `sigstore_canonical_checkpoint_bytes_pin`, `sigstore_log_pubkey_matches_pinned`). If a refactor changes these without changing the algorithm, you broke determinism — investigate before updating the pins.

9. **Single canonicalisation source** — TS never re-implements canonical hashing or canonical bytes. Always shell to `atlas-signer`. The `bundleHashViaSigner` and `anchorViaSigner` functions enforce this. Keep it that way.

10. **Atlas's anchoring key vs the log's signing key** — TWO separate keys:
    - **Log signing key** (Sigstore's `SIGSTORE_REKOR_V1_VERIFYING_KEY`) — pinned, signs checkpoints, defines the trusted log identity.
    - **Atlas anchoring key** (Phase 2 new) — used to sign the hashedrekord artifact, embedded in `signature.content` + `signature.publicKey.content` of the entry body. The verifier doesn't validate it; Sigstore validated it at submit time. Atlas just needs *some* signing identity.
