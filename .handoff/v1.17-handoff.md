## 0. V1.17 Welle C (2026-05-05) ÔÇö Trust-Root Mutation Defence (current)

**State summary:** in-repo cryptographic gate against trust-root mutation is shipped. Any commit modifying a trust-root-protected surface file MUST itself be signed by a key in the **prior** trust root ÔÇö `.github/allowed_signers` *as it existed at the PR base*. The "old trust root" semantic is the entire defence: it closes the bootstrap-attack vector where the malicious commit adds the attacker key in the same commit that relies on it. Welle C closes the F-5b gap that Welle B explicitly noted as out-of-scope and adjacent: "tag signing keeps the trust root in the repo, which is auditable but *mutable* by anyone with `contents: write`." Without Welle C, a single PAT compromise mints publish-lane trust in one commit. With Welle C, the attacker needs a commit signed by an *already-trusted* key ÔÇö i.e. the prior credential they don't have.

**Why `pull_request` (not `pull_request_target`):** `pull_request` runs the workflow file *from the base branch*. An attacker modifying the verify-trust-root-mutations workflow inside their PR cannot change which logic runs against their PR. `pull_request_target` would have the same base-branch behaviour but with write-permission tokens ÔÇö the famous "pwn request" footgun. Avoided. The third option, `push: master`, is NOT covered here ÔÇö a push event would load the workflow file *from the pushed commit itself*, so a malicious push modifying the workflow could disable verification on the same push. The push layer is closed by operator-side branch protection (no direct push to master incl. admins), documented in `docs/OPERATOR-RUNBOOK.md ┬ž14`.

**Code shipped:**
- `tools/verify-trust-root-mutations.sh` (NEW, ~310 lines) ÔÇö pure-bash verifier. For every commit in the `--base..--head` range, enumerates changed files via `git diff-tree --no-commit-id -r --name-only -m` (the `-m` flag is the merge-commit fix ÔÇö without it, `git show` returns empty for merge commits and a malicious merge could smuggle a trust-root change unnoticed). If any changed file matches the **PROTECTED_SURFACE** array (exact match) or **PROTECTED_PREFIXES** array (subtree match), runs `git -c gpg.format=ssh -c gpg.ssh.allowedSignersFile=<base-snapshot> verify-commit`. The base-snapshot is the `.github/allowed_signers` content extracted via `git show ${BASE_SHA}:.github/allowed_signers` ÔÇö i.e. the trust root at the PR base, NOT at the PR head. **CI containment guard**: rejects `ATLAS_TRUST_ROOT_BASE` / `ATLAS_TRUST_ROOT_HEAD` env-var overrides when `$GITHUB_ACTIONS=true`. **Anti-rewrite bootstrap defence**: even in apparent bootstrap mode (file absent at merge-base), if the file has *ever existed on master* (`git log --diff-filter=A --all` finds an add commit that's an ancestor of base), hard-fail with "anomalous bootstrap window" diagnostic ÔÇö closes admin-force-push-manufactures-fresh-bootstrap attack.

## 1. V1.17 Welle B (2026-05-05) ÔÇö Tag-Signing Enforcement

**State summary:** SSH-based tag-signing enforcement is shipped. Every `v*` tag in this repo MUST be cryptographically signed by an SSH key listed in the in-repo trust root `.github/allowed_signers`, OR (a) the standalone `verify-tag-signatures.yml` workflow fails red on push, **and** (b) the `wasm-publish.yml` workflow's first step (before any `actions/setup-node`, before any `npm publish`) fails closed. The two-layer gate is intentional defence-in-depth: a single bypass (e.g. someone deletes the standalone workflow file via a force-push to `master`, or someone disables required checks in branch-protection) cannot reopen the publish path. Welle B closes the F-5 tag-authenticity gap that became load-bearing the moment Welle A documented `uses: ThePyth0nKid/atlas/.github/actions/verify-wasm-pin-check@v1.17.0` as the recommended consumer pin form: a SHA-pinning consumer is only as authentic as the tagÔćĺcommit binding that produced the SHA.

**Why SSH (not Sigstore/gitsign, not GPG):** decision tree documented in `docs/SECURITY-NOTES.md` scope-l. SSH wins on (1) Windows-host friendliness ÔÇö `ssh-keygen -Y sign/verify` is in upstream OpenSSH and ships with both Git for Windows and modern Windows 10+, no extra installer; gitsign on Windows currently requires a manual cosign install + separate gitsign exe + Windows path quirks. (2) Trust root in-repo + auditable ÔÇö `.github/allowed_signers` is a plaintext file enumerating exactly which SSH keys can sign tags; review-able as a single artefact. Sigstore's trust root is the public Fulcio + Rekor instances, anchored in OIDC issuers ÔÇö strong but not auditable as a single committed file in the same way. (3) Keys already in maintainers' GitHub accounts ÔÇö `https://github.com/ThePyth0nKid.keys` already returns the two ed25519 keys nelson@ultranova.io uses for git push; the `.github/allowed_signers` entries are the same key bytes with a `nelson@ultranova.io` namespace prefix. Sigstore is documented as additive in ┬ž0 / V1.18 candidates: it can run **alongside** SSH (parallel signing of the same tag) for cross-channel verifiability without replacing SSH as primary trust root. GPG is rejected ÔÇö long key-management tail (rotation flow, expiry, web-of-trust mental model) for the same defence we get from SSH with first-class `ssh-keygen -Y` UX.

- `scripts/check-lockfile-integrity-pnpm.sh` (227 lines) ÔÇö handles `pnpm-lock.yaml` v6+. **AWK `index()`/`substr()` instead of `~`/`sub()`** for package-key matching ÔÇö defends against regex injection via `package-name` input (e.g. a name like `foo.*bar` would otherwise match unrelated keys via the `.` metachar). Reuses `awk -v pkg=...` literal substring matching, no regex compilation of attacker-controlled strings.
- `scripts/check-lockfile-integrity-bun.sh` (218 lines) ÔÇö handles binary `bun.lockb`. - `scripts/check-lockfile-integrity-bun-text.sh` (259 lines) ÔÇö handles JSONC `bun.lock` text format directly (no `bun` requirement). **String-aware JSONC tokenizer** (single-line `//` and block `/* */` comment stripping that respects string-literal boundaries ÔÇö naive regex `//` stripping would mangle URLs like `https://registry.npmjs.org/...`). Asserts every package entry's HTTPS resolved-URL + integrity-prefix invariants.
- `scripts/check-provenance.sh` (Layer 3, 213 lines) ÔÇö runs `npm audit signatures` and asserts the SLSA L3 OIDC-signed Sigstore Rekor attestation is present and valid. **`npm ls $PACKAGE --depth=0` precondition** before the audit (defends silent-pass when our package isn't in tree but others ARE signed ÔÇö `npm audit signatures` reports counts in the success case, so positive-grep alone would match an unrelated package's "verified attestation" line). **Retry-with-backoff** on transient classes (network/5xx/timeout); hard-fail classes (`attestation-failure`, `signature-failure`) skip retry. **Case-insensitive failure-context grep** (`grep -iF "$PACKAGE" | grep -iE "(missing|invalid|failed|error|untrusted)"`) defends against future npm-output casing changes.
