#### V1.10 wave 2 — sealed-seed loader (PKCS#11)

The master seed lives inside a PKCS#11 device. HKDF runs *inside* the device via `CKM_HKDF_DERIVE`; the master seed never enters Atlas address space. Production deployment surface:

| Env var | Purpose | Constraint |
|---|---|---|
| `ATLAS_HSM_PKCS11_LIB` | absolute path to the PKCS#11 module (`.so`/`.dylib`/`.dll`) | absolute (library-hijack defence) |
| `ATLAS_HSM_SLOT` | token slot id, decimal | non-negative integer |
| `ATLAS_HSM_PIN_FILE` | absolute path to the user-PIN file | absolute, mode 0400 |

Trio dispatch behaviour:
- **All-three-set** → HSM mode; `master_seed_loader` returns `Box<Pkcs11MasterSeedHkdf>`.
- **All-three-unset** → dev mode; loader returns `Box<DevMasterSeedHkdf>` (still gated by `ATLAS_DEV_MASTER_SEED=1`).
- **Partial** → refuse with actionable error (silent fallback would defeat the audit signal).
- **HSM init failure** → fatal; no fallback to dev seed.

PKCS#11 attribute roster (operator-side import ceremony, see `OPERATOR-RUNBOOK.md` §2):
- `CKO_SECRET_KEY` / `CKK_GENERIC_SECRET` / `CKA_VALUE_LEN=32`
- `CKA_LABEL=atlas-master-seed-v1` (hard-coded, multi-master-seed deployments are an explicit non-goal)
- `CKA_TOKEN=true` / `CKA_DERIVE=true`
- `CKA_EXTRACTABLE=false` / `CKA_SENSITIVE=true` / `CKA_PRIVATE=true` / `CKA_MODIFIABLE=false`

Memory-disclosure hardening:
- PIN bytes wrapped in `Zeroizing<Vec<u8>>` on read; `AuthPin` (`secrecy::SecretString`) zeroes on drop.
- 32-byte HKDF output wrapped in `Zeroizing<Vec<u8>>` at the cryptoki FFI boundary.
- Per-tenant seed array wrapped in `Zeroizing<[u8; 32]>` in both `derive_workspace_signing_key` (V1.9 leaf) and `derive_workspace_signing_key_via` (V1.10 trait-routed).

Ed25519 generation note: PKCS#11 has no standard "HKDF then Ed25519 keypair" mechanism, so wave 2 ships the 32 derived bytes out of the token. Wave 3 evaluates either `CKM_EC_EDWARDS_KEY_PAIR_GEN` (vendor-dependent) or signing inside the token via `CKM_EDDSA`.

## 2. V1.10 post-review findings — V1.11 warm-up sprint

The V1.10 wave 2 commit was reviewed by parallel `code-reviewer` + `security-reviewer` agents. Cheap fixes landed in `c9d2078`. The remaining findings below are **the warm-up work for V1.11**. None of them block production-equivalent deployment; all of them tighten the documented security story.

### HIGH — should fix in warm-up

1. **`secret_hex` in `PerTenantIdentity` is unprotected `String` (`crates/atlas-signer/src/keys.rs:670–679`).** `derive_workspace_signing_key_via` correctly zeroizes the 32-byte HKDF buffer, but the very next line copies via `signing_key.to_bytes()` (unprotected `[u8; 32]`) → `hex::encode(...)` (unprotected 64-byte `String`) → `PerTenantIdentity::secret_hex`. The `String` lives inside the struct for the lifetime of the `derive-key` subcommand and is only freed (not zeroized) on drop, leaving 64 hex chars in the allocator's freed-pool until reused. This is on the ceremony hot path. **Fix options:**
   - **(a)** Wrap the intermediate `Zeroizing::new(signing_key.to_bytes())` and let `secret_hex` itself stay a `Zeroizing<String>` field — requires changing `PerTenantIdentity::secret_hex` to a wrapper type.
   - **(b)** Eliminate the struct field entirely. The hex secret is only emitted on stdout for the `derive-key` ceremony; no consumer needs to hold it. Refactor `derive-key` to write the hex directly to stdout from a local `Zeroizing<String>` and drop the field from `PerTenantIdentity`.
   - **(b) is cleaner** and reduces the surface area; the field exists only because V1.9 originally returned it as part of a JSON envelope. Audit downstream consumers (TS `signer.ts`, `keys.ts`) — most read `pubkey_b64url` and `kid` only.

### MEDIUM — opportunistic

2. **PIN file mode-bit check at read time (`crates/atlas-signer/src/hsm/pkcs11.rs:175`).** Runbook §2 step 5 uses `install -m 0400` and pre-flight `umask 077` — the *write* side is correct. The *read* side has no permission guard: an operator who creates the PIN file with world-readable permissions (e.g. forgot the umask, or `echo "$PIN" > /run/atlas/hsm.pin`) gets silent acceptance. **Fix:** after `std::fs::read`, on Unix check `metadata().mode() & 0o077 == 0` and return `MasterSeedError::Locked("PIN file has world/group-readable permissions; chmod 400 required")`. Wrap in `#[cfg(unix)]`; Windows has no portable equivalent (deployments are Linux/container-first per the runbook anyway). Add an adversary test using `tempfile` + a 0644 PIN file.

3. **Mutex-poison ephemeral key cleanup (`crates/atlas-signer/src/hsm/pkcs11.rs:234–260`).** If `derive_for` panics between `derive_key` succeeding (ephemeral object created inside the token) and `destroy_object` running, the ephemeral key handle is never destroyed. CKA_TOKEN=false means it dies with the session, but while the Mutex is poisoned the session stays open until process exit. For the per-command subprocess spawn model, this is bounded; for any future long-lived embed (MCP server holding a `Box<dyn MasterSeedHkdf>`), accumulated orphaned objects are a token-storage concern on commercial HSMs. **Fix:** RAII guard around the derived handle so `destroy_object` is guaranteed even through panic. Stronger invariant if future code ever sets CKA_TOKEN=true.

4. **`master_seed_loader` re-init per subcommand (`crates/atlas-signer/src/main.rs`, all four dispatch arms).** Each subcommand calls `keys::master_seed_loader()` independently. In dev mode this is cheap; in HSM mode each call opens the PKCS#11 library, calls `C_Initialize`, opens an authenticated session, and walks `C_FindObjects`. Today the binary is a one-shot CLI (at most one subcommand per process) so the cost is bounded, but this **will become an audit-log footprint** on commercial HSMs where every `C_Login` is logged. **Fix:** call `master_seed_loader()` once in `main` and pass `&*hkdf` down. The trait object already supports it; the dispatch arms just need the borrow plumbed through.

5. **Mutex-poison error categorisation (`crates/atlas-signer/src/hsm/pkcs11.rs:234–239`).** Returns `MasterSeedError::Unavailable("…mutex poisoned…")`. Operator runbook describes `Unavailable` as "check HSM connectivity" (transient). A poisoned mutex is permanent for this process instance. **Fix:** either categorise as `DeriveFailed` or extend the message with "process restart required."

6. **Document TOCTOU on `dlopen` path (`docs/OPERATOR-RUNBOOK.md` §2 threat model).** The absolute-path guard closes the CWD-relative hijack vector. Between config-parse and `Pkcs11::new`, an attacker with filesystem write can replace the file at the absolute path. This is a standard `dlopen` TOCTOU and cannot be fully closed in userspace without kernel mechanisms (mount namespaces, sealed FDs). **Fix:** add a sentence to the threat-model section explicitly calling out filesystem ACLs on `ATLAS_HSM_PKCS11_LIB` as a required control, not just a nice-to-have.

### LOW — nits, drop in passing

7. **Runbook ceremony missing post-import attribute assertion (`docs/OPERATOR-RUNBOOK.md` §2 verification block).** Current verification runs `--list-objects --type secrkey` (label + class only). An operator whose HSM silently ignores `--usage-derive` or sets `CKA_EXTRACTABLE=true` will see a passing `derive-pubkey` smoke but have an extractable seed. **Fix:** add a `pkcs11-tool ... --read-object` command that emits the attributes, with explicit "expected output: `CKA_SENSITIVE: yes`, `CKA_EXTRACTABLE: no`" comment.

8. **Retire `ATLAS_PRODUCTION` after wave 2 stabilises.** The HSM trio is now the production audit signal. `ATLAS_PRODUCTION=1` is documented as belt-and-braces but carries the V1.9 footgun forward (literal-`"1"`-only recognition) for any operator who relies on it without the trio. **V1.11 task:** plan deprecation — emit a `tracing::warn` if `ATLAS_PRODUCTION` is set without the HSM trio in V1.11; remove in V1.12.

9. **`HsmConfig` field visibility.** Fields are `pub` with auto-derived `Debug`; `pin_file` path leaks in any `format!("{:?}", cfg)` site (the path itself is not sensitive, but operator logs would expose your secret-mount layout). Consider `pub(crate)` for `pin_file` if no external consumer needs it. Audit current callers first.

## 4. Footguns / hard-won learnings

### Carried forward (still load-bearing in V1.11)

1. **`anchor-chain.jsonl` is append-only and irreversible** — sole writer is the issuer. (V1.7)
2. **Chain head canonicalisation must be byte-identical between issuer and verifier.** Pinned golden tests. (V1.7)
3. **Verifier may NOT trust `trace.anchor_chain.head` field** — recompute. (V1.7)
4. **Sigstore Rekor v1 shard roster is a source change** — adding tree-IDs requires crate version bump. (V1.7)
5. **Every signer-stdout boundary in TS uses `parseAnchorJson`** (lossless-json). Don't add a new `JSON.parse(stdout)` site without re-deriving why. (V1.8)
6. **`cargo` not in default PATH on Windows** — `/c/Users/nelso/.cargo/bin/cargo.exe`. Use `export PATH="/c/Users/nelso/.cargo/bin:$PATH"` before any cargo invocation in bash.
7. **Smoke at `apps/atlas-mcp-server`** — `pnpm --filter atlas-mcp-server smoke` from root.
8. **Single canonicalisation source — TS never re-implements anything cryptographic.** Always shell to `atlas-signer`. (V1.5+)
9. **HKDF info parameter is the trust boundary for per-tenant key separation.** Always domain-prefix. (V1.9)
10. **Backward compat lenient mode is a trust trade-off, not a free win.** Strict mode is the security boundary. (V1.9)
11. **Don't re-derive a pubkey at verify time.** The verifier reads from the bundle; never knows the master seed. (V1.9)
12. **`std::env::set_var` in tests races on multi-core CI.** Use the V1.10 injection-form gates (`production_gate_with`, `master_seed_gate_with`, `master_seed_loader_with`, `HsmConfig::from_env`).

### NEW for V1.11 (V1.10 lessons)

13. **Cargo no-cycles rule trumps clean module decomposition.** Wave 2 originally planned a sibling `atlas-signer-hsm` crate; it would have cycled through `atlas-signer`'s `MasterSeedHkdf` trait. The in-tree `crates/atlas-signer/src/hsm/` behind the `hsm` feature flag is the corrected layout. If V1.11 wave 3 is tempted to extract HSM code into a sibling, **don't** — extend the in-tree module instead.

14. **PKCS#11 is verbose; cryptoki abstracts the syntax but not the semantics.** `bExtract=true` + `CKF_HKDF_SALT_NULL` + `CKM_HKDF_DERIVE` is the byte-equivalent of `Hkdf::<Sha256>::new(None, seed)`. **Test this with a pinned-pubkey golden against a known SoftHSM2 seed before trusting any vendor's HKDF.** The drift class is silent: the verifier produces "VALID" for whichever pubkey ends up in the bundle.

15. **Drop order matters for FFI resources.** `Pkcs11MasterSeedHkdf` declares `session: Mutex<Session>` *before* `_ctx: Pkcs11` so the session destructor (`C_CloseSession`) runs before the context destructor (`C_Finalize`). Inverting these fields produces a use-after-free in cryptoki on every drop. Document this when adding new FFI-resource structs.

16. **Absolute-path guard at config-parse time, not at use time.** `Path::is_absolute()` semantics differ between Unix (`/path` = absolute) and Windows (drive prefix required). V1.10 ships platform-conditional test data — when adding new path-bearing env vars, use the same `#[cfg(windows)]` / `#[cfg(not(windows))]` pattern in tests, not a single `/path` literal.

17. **Zeroize at every heap-resident copy of secret material.** Wave 2 hardening had to wrap PIN bytes (`Zeroizing<Vec<u8>>`), HKDF output (`Zeroizing<Vec<u8>>` at the cryptoki FFI), and per-tenant seed array (`Zeroizing<[u8; 32]>`) at three different sites. The pattern is: any time a secret crosses a heap or stack-arr ay boundary, wrap it. The H-1 finding deferred to V1.11 (see §2.1) is exactly the case where one site missed the wrap.

18. **Operator runbook ceremonies must use secrets-manager env vars, not literal PINs.** Wave 2 caught a `mktemp`+`chmod` race window AND a literal-PIN-on-command-line risk in the original ceremony. Standard pattern: `umask 077` upfront, source PINs from `your-secrets-manager get ...` into shell vars, `unset SO_PIN USER_PIN` at end. Apply this to any new ceremony added in V1.11.
