### V1.6 capability (shipped)

The verifier (`atlas-trust-core`) accepts two anchor formats; the issuer (`atlas-signer`) emits both:

- **`atlas-mock-rekor-v1`** (V1.5): blake3 RFC 6962 + Ed25519 over a 3-line atlas-mock checkpoint. Default for offline demos and `pnpm smoke`.
- **`sigstore-rekor-v1`** (V1.6): SHA-256 RFC 6962 + ECDSA P-256 over a C2SP signed-note checkpoint, against the public Sigstore Rekor v1 production log (`rekor.sigstore.dev`). Opt-in via `atlas-signer anchor --rekor-url <url>` or MCP `rekor_url` field / `ATLAS_REKOR_URL` env (precedence: field > env > mock).

Dispatch happens via a pinned `BTreeMap<log_id, TrustedLog>`. See `crates/atlas-trust-core/src/anchor.rs::default_trusted_logs`.

### What V1.6 cannot do yet (the gap V1.7 should close)

**The anchor history is not chained.** Each `atlas_anchor_bundle` invocation OVERWRITES `data/{workspace}/anchors.json` with a snapshot of the current state. That gives the auditor:

> ✓ "This `dag_tip` / `bundle_hash` was witnessed at time T against a pinned log."

But NOT:

> ✗ "The SEQUENCE of witnessed states is consistent. No prior state has been silently rewritten."

A server can issue an anchor at T₁, later corrupt the trace, then anchor a *different* state at T₂ — and the auditor only ever sees the most recent anchors.json. The trust property holds *per anchor*; it does not hold *across* anchors.

**That's V1.7.**

## 2. Prime directive — the trust property

> An auditor with the trace bundle and a fresh `atlas-verify-cli` binary can establish, **fully offline**, that every claim was bit-identical to what was signed and anchored.

V1.7 must extend this to:

> The auditor can additionally establish that the anchor history is **monotonic** — no past anchor batch has been silently rewritten or omitted between time T₀ and T_now.

Every V1.7 design decision filters through this:

- The verifier may NOT make a network call. Anchor-chain validation is fully local.
- The chain proof is in the trace bundle. Whatever the issuer captures at chain-extension time goes into `AtlasTrace` and is replayed offline.
- Any drift between issuer-derived chain state and verifier-recomputed chain state is a trust violation. Single-source the canonicalisation in Rust.
- **Backwards compatibility**: existing V1.5/V1.6 trace bundles must continue to verify. The chain proof is additive — older bundles without it pass under "lenient" mode; strict mode (`require_anchor_chain`) demands it.

## 3. V1.7 scope — recommendation

The strongest single V1.7 deliverable is **anchor-chain tip-rotation**: cross-link consecutive anchor batches so a server cannot rewrite past anchored state without breaking the chain. This closes the biggest remaining trust gap with modest scope (one new field on `AtlasTrace`, one new verifier check).

**Recommended primary scope**: anchor-chain tip-rotation.
**Recommended secondary scope** (cheap add-on): Sigstore log shard expansion (lift the strict-tree-id check, add 2–3 historical shard origins to `default_trusted_logs`).
**Deferred to V1.8+**: per-tenant Atlas anchoring keys, HSM/TPM sealing, multi-issuer redundancy.
**Deferred to V2** (unchanged): Cedar policy enforcement, full COSE_Sign1, SPIFFE attestation, graph-database integration.

## 4. V1.7 primary — anchor-chain tip-rotation

### Goal

Each `atlas_anchor_bundle` run produces an anchor batch B_n. V1.7 adds a chain field that binds B_n to B_{n-1}:

```
chain_head_n = blake3(
    b"atlas-anchor-chain-v1:" ||
    canonical_bytes_of(B_n.entries) ||
    chain_head_{n-1}    // 32 bytes; all-zero for the first batch
)
```

The chain head is stored as a new field on `AtlasTrace`:

```rust
pub struct AtlasTrace {
    // ... existing fields ...

    /// Hash-chain head over consecutive anchor batches issued for this
    /// workspace. Defends against post-hoc rewriting of past anchored
    /// state. The verifier recomputes this from `anchor_chain_history`
    /// (ordered list of past batches) and compares to the head a fresh
    /// anchor batch would produce. Mismatch → ✗ INVALID.
    ///
    /// Empty for V1.5/V1.6 trace bundles; lenient verifier accepts
    /// missing chain (matching the V1.5 "no claim is fine" rule).
    /// Strict mode (`require_anchor_chain`) demands a non-empty history
    /// and a verified head.
    pub anchor_chain: Option<AnchorChain>,
}

pub struct AnchorChain {
    /// All anchor batches issued for this workspace, in issuance order.
    /// Each batch is a (timestamp, root_hash) pair plus the entries
    /// themselves so the verifier can recompute each historical
    /// chain_head locally. Earlier entries are append-only — the verifier
    /// rejects any trace that drops or reorders past batches relative
    /// to the head.
    pub history: Vec<AnchorBatch>,

    /// blake3 of the canonical chain serialisation. Fail-fast field;
    /// the verifier recomputes and compares.
    pub head: String,  // hex
}

pub struct AnchorBatch {
    pub batch_index: u64,                  // 0, 1, 2, ...
    pub integrated_time: i64,              // Unix seconds, matches AnchorEntry.integrated_time
    pub entries: Vec<AnchorEntry>,         // the entries that made up this batch
    pub previous_head: String,             // hex of chain_head_{n-1}; "00..." for batch 0
}
```

### Storage

- **`data/{workspace}/anchors.json`** continues to hold the LATEST batch (current behaviour preserved).
- **NEW: `data/{workspace}/anchor-chain.jsonl`** — append-only log of every batch ever issued. Each line is one `AnchorBatch`. Atomic append via tmp-and-rename → no half-written line at a crash boundary.
- `exportWorkspaceBundle` reads BOTH files: latest batch goes in `trace.anchors` (unchanged), full history goes in `trace.anchor_chain.history`. The export computes `trace.anchor_chain.head` once.
- **Rotation discipline**: `anchor-chain.jsonl` is NEVER truncated except by an explicit operator action (`atlas-signer rotate-chain --confirm <workspace>`). Rotation produces a new chain root (genesis batch references the old chain's final head as `previous_head` so an auditor with the old trace + new trace can verify continuity).

### Verifier check (new)

In `atlas-trust-core::verify::verify_trace`:

1. If `trace.anchor_chain` is `None`: pass under lenient mode, fail under `require_anchor_chain`.
2. If present: walk `history[]` in order. For each `AnchorBatch[i]`:
   a. Recompute `chain_head_i` from `(batch_entries, previous_head)` using `blake3` with the `atlas-anchor-chain-v1:` prefix.
   b. Assert `history[i+1].previous_head == chain_head_i` (chain continuity).
   c. Assert `batch_index == i` (no gaps, no reorder).
   d. For the last batch, assert the recomputed head == `trace.anchor_chain.head`.
3. Optionally assert `trace.anchors == history[last].entries` (current snapshot matches chain tip).

### Issuer flow (changed)

In `atlas-signer/src/anchor.rs::anchor_batch`:

1. Read `anchor-chain.jsonl` to determine the next `batch_index` and the current `previous_head`. (Empty file → batch_index=0, previous_head="00…".)
2. Issue the new batch as today (mock or Sigstore).
3. Compute the new `chain_head` = blake3 of canonical bytes.
4. Append the new `AnchorBatch` to `anchor-chain.jsonl` atomically.
5. Write the latest batch to `anchors.json` (unchanged).

The signer is the only writer of `anchor-chain.jsonl`. The MCP server hands the workspace path to the signer; the signer does the read-modify-append. This keeps single-source canonicalisation.

### Schema discipline

- New types live in `crates/atlas-trust-core/src/trace_format.rs` with `#[serde(deny_unknown_fields)]`.
- The canonical-bytes computation for `chain_head` lives in **one** function in `atlas-trust-core::anchor`, called by both issuer and verifier.
- A pin test asserts `chain_head_for(empty_genesis_batch) == "<known hex>"` so silent drift in the canonicalisation breaks CI.
- `VERIFIER_VERSION` bump (V1.6 → V1.7).

### Backwards compatibility

- `AtlasTrace.anchor_chain: Option<AnchorChain>` is `None` for V1.5/V1.6 bundles → lenient pass.
- `VerifyOptions` gets a `require_anchor_chain: bool` (default `false`); strict mode demands a present, valid chain.
- Existing 13 golden_traces + 5 sigstore_golden tests continue to pass without modification (they don't carry `anchor_chain`).
- New tests are added in `tests/anchor_chain_*.rs` and cover: round-trip, missing batch in middle, reordered batches, head mismatch, mismatched previous_head, fresh issuer-side chain extension.

## 5. V1.7 secondary — Sigstore log shard expansion

### Current limitation

`crates/atlas-trust-core/src/anchor.rs::verify_sigstore_rekor_v1` rejects any entry whose `tree_id != SIGSTORE_REKOR_V1_ACTIVE_TREE_ID`. Sigstore has historical shards signed by the same pubkey but with different tree IDs. After a future shard rotation, *all V1.6 anchors will reject* against the new active shard.

### V1.7 fix

Replace the single-tree-id constant with a set of trusted (origin, tree_id) tuples:

```rust
pub static SIGSTORE_REKOR_V1_TRUSTED_SHARDS: &[(&str, i64)] = &[
    ("rekor.sigstore.dev", 1_193_050_959_916_656_506),  // current active
    // Add historical shards as Sigstore documents them.
];
```

Verifier: accept the entry if `(checkpoint_origin, tree_id)` matches *any* row. Pubkey is shared across shards (`SIGSTORE_REKOR_V1_PEM` unchanged).

This is a small, mechanical change and a natural fit alongside the anchor-chain work (both touch `verify_sigstore_rekor_v1`).

## 7. Footguns / hard-won learnings (carry forward + new)

### Carried from V1.6 (still load-bearing)

1. **JS `parseInt` truncates `tree_id`** — `1193050959916656506` exceeds `Number.MAX_SAFE_INTEGER`. Parse as STRING / `BigInt` in TS. Splice into fixture JSON via regex string replacement, not `JSON.stringify`.

2. **Base64 strictness on the Sigstore path** — Sigstore uses RFC 4648 §4 STANDARD base64 with `=` padding. The mock path uses URL_SAFE_NO_PAD. Don't cross-contaminate.

3. **Origin separators** — `canonical_checkpoint_bytes_sigstore` rejects origins containing `\n` or ` - ` or empty. C2SP separators; bypass = signed-body injection.

4. **C2SP keyID is BIG-endian, prepended** — `base64_decoded_signature[..4] == SIGSTORE_REKOR_V1_KEY_ID = SHA-256(DER SPKI)[:4]`. Defends against signature lines authored by a different log.

5. **`cargo` not in default PATH on this Windows machine** — every shell invocation needs `export PATH="/c/Users/nelso/.cargo/bin:$PATH"`. The `PowerShell` tool isn't available; use `Bash`.

6. **Smoke is at `apps/atlas-mcp-server`** — `pnpm --filter atlas-mcp-server smoke` from root, OR `cd apps/atlas-mcp-server && pnpm smoke`.

7. **Pinned hex values in tests** — `sigstore_leaf_hash_is_rfc6962_sha256`, `sigstore_canonical_checkpoint_bytes_pin`, `sigstore_log_pubkey_matches_pinned`, `atlas_anchor_pubkey_pem_is_pinned`, `mock_log_pubkey_matches_signer_seed`, plus the `sigstore_anchored_hash_for_*_is_pinned` derivation tests. If a refactor changes these without changing the algorithm, you broke determinism — investigate before updating the pins.

8. **Single canonicalisation source** — TS never re-implements canonical hashing or canonical bytes. Always shell to `atlas-signer`. The `bundleHashViaSigner` and `anchorViaSigner` functions enforce this. Keep it that way for `chain_head` too — TS never computes it; the signer does, and the verifier recomputes.

9. **Atlas's anchoring key vs the log's signing key** — TWO separate keys:
   - **Log signing key** (`SIGSTORE_REKOR_V1_VERIFYING_KEY`) — pinned, signs checkpoints, defines the trusted log identity.
   - **Atlas anchoring key** — used to sign the hashedrekord artifact, embedded in the entry body. The verifier doesn't validate it (Sigstore did at submit time). Pinned via `atlas_anchor_pubkey_pem_is_pinned`.

10. **HTTP loopback gate** — `crates/atlas-signer/src/rekor_client.rs::RekorClient::new` requires `https://` for non-loopback hosts. Plaintext `http://` is ONLY allowed for `localhost`/`127.0.0.1`/`[::1]` (wiremock testing). Don't loosen this.

### NEW for V1.7

11. **`anchor-chain.jsonl` is append-only and irreversible** — the format is "log of all anchor batches forever". Operators cannot truncate it without a rotation ceremony (`atlas-signer rotate-chain`) that produces a continuity-bridging genesis batch. If a workspace's chain file is lost, the trust property fails for that workspace from that point. Document this clearly in `apps/atlas-mcp-server/README.md`.

12. **Chain head canonicalisation must be byte-identical between issuer and verifier** — same single-source-in-Rust discipline as `signing_input_bytes` and `bundle_hash_bytes`. Add a byte-pinned golden so silent drift fails CI.

13. **Verifier may NOT trust the trace's `anchor_chain.head` field** — recompute from `history[]` and compare. The `head` field is a fail-fast convenience (if `head` doesn't match recomputed head, fail before walking). Do NOT short-circuit on the field.

14. **`batch_index` must be sequential, no gaps** — `history[i].batch_index == i`. Gaps → reject. This is structural; don't allow "skipped batches" even if the operator claims the workspace was idle. Idle workspaces just don't extend the chain.

15. **Rotation ceremony has a separate commit subject** — `feat(v1.7/rotation): atlas-signer rotate-chain --confirm` is its own contained change, NOT bundled with the chain implementation. The user reviews rotation tooling specifically. Keep it scoped.
