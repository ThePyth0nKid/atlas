### V1.9 capabilities (shipped)

#### Per-tenant Atlas anchoring keys

```rust
// crates/atlas-signer/src/keys.rs
info       = "atlas-anchor-v1:" || workspace_id
key_bytes  = HKDF-SHA256(salt = None, ikm = master_seed, info, len = 32)
signing    = ed25519_dalek::SigningKey::from_bytes(&key_bytes)
```

- Each workspace gets its own Ed25519 key. Compromise of one workspace's key does not compromise others.
- Public keys appear in `PubkeyBundle.keys` under kid shape `atlas-anchor:{workspace_id}`.
- Issuer-side HKDF info-prefix `atlas-anchor-v1:` is intentionally distinct from verifier-side kid prefix `atlas-anchor:` (cryptographic-domain tag decoupled from wire identifier).

#### Production gate

`atlas-signer::keys::production_gate()` refuses every per-tenant subcommand (`derive-key`, `derive-pubkey`, `rotate-pubkey-bundle`, `sign --derive-from-workspace`) when `ATLAS_PRODUCTION=1`. The `DEV_MASTER_SEED` is a source-committed dev constant; V1.10 closes the dev/prod gap by sealing the master seed in hardware.

#### Signer-internal derivation (no secret in Node heap)

The MCP hot path uses `atlas-signer sign --derive-from-workspace <ws>` — secret material is derived inside the signer process and never crosses the TS↔Rust boundary. Bundle assembly uses `atlas-signer derive-pubkey` (public key only). The `derive-key` subcommand (which emits the secret) is ceremony-only and gated.

#### Strict mode + lenient compat

`VerifyOptions::require_per_tenant_keys` (default `false`) is the V1.9 security boundary. Lenient mode accepts both legacy SPIFFE kids and per-tenant kids; strict mode demands every event's kid equal `format!("atlas-anchor:{trace.workspace_id}")` for the trace's own workspace_id.

#### Bundle rotation ceremony

`atlas-signer rotate-pubkey-bundle --workspace <ws>` reads a `PubkeyBundle` from stdin, inserts the per-tenant kid + pubkey, emits the updated bundle on stdout. Idempotent; refuses to overwrite mismatched existing pubkey. Atomic-replace + concurrency are operator-side responsibilities — see `docs/OPERATOR-RUNBOOK.md`.

## 2. V1.9 review findings — to address in V1.10 warm-up

The V1.9 commits passed parallel `code-reviewer` + `security-reviewer` runs. Findings below are **the hardening work to do before the HSM seal sprint begins.** None of these block V1.9 in production-equivalent dev workflows; all of them tighten the documented security story or close concrete CI risks. They are sized so a focused half-day session closes them all.

### CRITICAL — must fix first

1. **Test env-var data race (`crates/atlas-signer/src/keys.rs:421-438`).** Tests use `unsafe { std::env::set_var(...) }` / `remove_var(...)` to flip `ATLAS_PRODUCTION`. The comment claims sequential execution but cargo defaults to a thread pool. Two such tests racing in the same binary corrupt each other's env. **Fix:** add `serial_test = "3"` as a dev-dependency and annotate the affected tests with `#[serial]`, OR refactor `production_gate` to accept an injectable env reader (`fn production_gate_with(env: impl Fn(&str) -> Option<String>)` + the public `production_gate` calls it with `std::env::var`). The injection refactor is the better V1.10 design because the HSM-loader path will need to inject seed sources too.

2. **`production_gate` is a negative guard (`crates/atlas-signer/src/keys.rs:97-107`).** When `ATLAS_PRODUCTION` is unset, empty, or any value other than `"1"`, the per-tenant subcommands silently run with `DEV_MASTER_SEED`. This is the documented design but the runbook does not warn loudly enough that an unset variable is the dangerous state. The V1.10 inversion is: **positive opt-in** — refuse to start unless a sealed-seed source is configured (`ATLAS_MASTER_SEED_PATH`, `ATLAS_HSM_HANDLE`, etc.), with `ATLAS_DEV_SEED=1` as the only escape hatch for dev/CI. **Action:** plan this inversion as part of the V1.10 HSM design, not a V1.9 patch — but document the unset-env gap in `docs/OPERATOR-RUNBOOK.md` §1 prominently before V1.10 ships.

3. **Library-path bypass for `per_tenant_identity` / `derive_workspace_signing_key_default` (`crates/atlas-signer/src/keys.rs:180,206`).** Currently `pub fn` — gated only at the CLI dispatch layer. Today `atlas-signer` is binary-only so no external consumer exists, but V1.10 will add an HSM-backed loader (likely a sibling crate) and the temptation to call these functions directly is real. **Fix:** make both `pub(crate)` for now. The V1.10 HSM crate gets its own gated entry point; it does not bypass the binary's gate.

### HIGH — should fix in warm-up

4. **`atlas-verify-cli` does not surface `--require-per-tenant-keys` (`crates/atlas-verify-cli/src/main.rs`).** The flag exists in `VerifyOptions` but the CLI hardcodes `VerifyOptions::default()` (lenient). The documented "strict mode is the V1.9 security boundary" story is not reachable from the auditor-facing tool. **Fix:** add `#[arg(long)] require_per_tenant_keys: bool` (and the matching `require_anchors` / `require_anchor_chain` flags if missing) to the `VerifyTrace` subcommand. ~30 lines including docstrings + a CLI integration test.

5. **Discriminated-union escape hatch in `signer.ts:90` (`apps/atlas-mcp-server/src/lib/signer.ts`).** `isDerive = "deriveFromWorkspace" in args && args.deriveFromWorkspace !== undefined` works at runtime but escapes the type via `as string` casts at lines 92/94. **Fix:** use proper TS narrowing:
   ```typescript
   if ("deriveFromWorkspace" in args && args.deriveFromWorkspace !== undefined) {
     // args.deriveFromWorkspace: string here, no cast needed
   }
   ```

6. **`Base64UrlNoPad` schema does not enforce Ed25519 32-byte length (`apps/atlas-mcp-server/src/lib/schema.ts:27`).** A signer regression that emits a truncated pubkey passes the schema and silently lands a junk key in the bundle. **Fix:** add a 43-char-length variant `Base64UrlEd25519PubkeySchema = z.string().length(43).regex(/^[A-Za-z0-9_-]+$/)` and use it in `DerivedIdentitySchema.pubkey_b64url` and `DerivedPubkeySchema.pubkey_b64url`.

7. **HSM-incompatible signature: `derive_workspace_signing_key(&[u8; 32], ...)` (`crates/atlas-signer/src/keys.rs:165-175`).** A real HSM key handle is not a raw 32-byte array. V1.10 cannot extract the seed bytes to call this function (defeats sealing). **Design action — settle this before V1.10 implementation begins.** Two options:
   - **(a)** Trait abstraction: `trait MasterSeedHkdf { fn derive(&self, info: &[u8], out: &mut [u8; 32]); }`, with one impl backed by `DEV_MASTER_SEED` (today) and another backed by an HSM/TPM PKCS#11 derive operation. This is the cleanest path because most HSMs natively support HKDF as a key-derivation operation that returns a derived handle without exposing ikm.
   - **(b)** External KDF service: signer submits HKDF-derive requests to a sidecar that owns the seal. Higher operational cost; reserve for cloud-KMS deployments where local HSM is not present.
   Recommend (a) for V1.10 with (b) reserved as an extension point.

### MEDIUM — opportunistic

8. **Workspace-id has no length cap (`crates/atlas-signer/src/keys.rs:131-143`).** A 10 MB workspace_id is valid input today. Adds a 10 MB key into `PubkeyBundle`; cheap DoS. **Fix:** cap at 256 bytes (or document the chosen ceiling) in `validate_workspace_id`.

9. **Missing adversary tests:**
   - Tampered `pubkey_bundle_hash` with valid per-tenant kid (force the bundle-hash check to fire before strict-mode).
   - Zero-event trace under `require_per_tenant_keys = true` (`mismatched.is_empty()` passes vacuously — assert this is the intended behaviour).
   - Cross-bundle per-tenant kid reuse (workspace A's kid in workspace B's bundle).

10. **`PerTenantIdentity` derives `Debug` with `secret_hex` (`crates/atlas-signer/src/keys.rs`).** Any `dbg!()` or `tracing` call on the struct prints the secret. **Fix:** manual `Debug` impl that emits `secret_hex: <redacted>`. Same applies to TS `DerivedIdentity` — add a custom `[Symbol.for('nodejs.util.inspect.custom')]`.

11. **`rotate-pubkey-bundle` is silent on idempotent re-run.** Operator scripts cannot distinguish "added" from "already present." **Fix:** `eprintln!("rotate-pubkey-bundle: kid {} already present with matching pubkey — no change", identity.kid)` in the no-op path.

12. **`HKDF_INFO_PREFIX` is private (`crates/atlas-signer/src/keys.rs:150`).** When V1.10 adds an HSM-backed loader that re-implements derivation externally, the prefix string must be authoritative. **Fix:** make `pub` with security rationale doc.

13. **`SECURITY-NOTES.md` does not document seed-rotation invalidates historical hashes.** Rotating the master seed changes every per-tenant pubkey, which changes every `pubkey_bundle_hash`, which invalidates every historical trace's claimed hash. **Fix:** add a paragraph under "Master-seed exfiltration is full compromise."

### LOW — nits, drop in passing

14. Dead placeholder check in `keys.rs:313-318` (`if DEFAULT_PUBKEY_B64URL == "__DEFAULT_PIN_PLACEHOLDER__"`). Remove.
15. Smoke test verifies per-tenant trace in lenient mode only — add a strict-mode call once the verify-cli flag exists.
16. Consider renaming `DeriveKeyArgs` → `DeriveWorkspaceArgs` since it's now shared between `derive-key` and `derive-pubkey`.
17. Annotate `HKDF_INFO_PREFIX` with "future domain prefixes must not produce info strings that are a prefix of this one" (defends against future cross-domain collisions).

## 3. V1.10 — primary scope: HSM/TPM master-seed sealing

### Goal

Move the master seed from a source-committed `[u8; 32]` constant to a hardware-sealed handle. After V1.10:

- The per-tenant Ed25519 derivation runs against a sealed master key. Memory-disclosure on the signer host does **not** leak the master seed.
- Production deployments require a positive-proof startup check: refuse to begin signing operations if no sealed seed source is reachable.
- Dev / CI keep working via an explicit opt-in (`ATLAS_DEV_SEED=1` or similar) — never the default.

### Why HSM/TPM, why now

V1.9 documented "master seed compromise = full compromise" as residual risk. That is the largest single defensive gap left in the system. Every other V1.10 candidate (witness wiring, multi-issuer Sigstore, WASM publishing, live-Rekor CI) is incremental. HSM sealing is foundational — it changes the attack model for every workspace at once, and it makes the V1.9 per-tenant story actually load-bearing in adversarial deployments.

### Implementation seams (use what V1.9 left in place)

- **HKDF function signature (HIGH #7).** Must change from `&[u8; 32]` ikm to a trait abstraction. Recommend a `trait MasterSeedHkdf` with a `dev` impl (current `DEV_MASTER_SEED` behind a feature flag) and an `hsm` impl (PKCS#11 / TPM 2.0 / cloud-KMS derive).
- **Production gate inversion (CRITICAL #2).** Flip `production_gate` from "refuse if `ATLAS_PRODUCTION=1`" to "refuse unless a sealed-seed source is configured." `ATLAS_DEV_SEED=1` is the explicit dev escape hatch.
- **Loader injection (HIGH #7 + CRITICAL #1).** Refactor every entry point in `main.rs` to take an injectable `Box<dyn MasterSeedHkdf>` (or static dispatch via generic). Tests pass a deterministic dev impl; production passes the HSM-backed impl.
- **Workspace-id ingress (MEDIUM #8).** Cap length before HSM derive — HSMs may have stricter info-length limits than local HKDF, and you want the failure mode to be "operator error" not "HSM rejected."

### Implementation plan (suggested order)

1. **Settle HSM choice with user.** PKCS#11 (broad HSM compatibility) vs TPM 2.0 (laptop-class) vs cloud-KMS (AWS KMS, GCP Cloud KMS, Azure Key Vault). Recommend PKCS#11 as the primary because most production HSMs (Thales, AWS CloudHSM, Yubico, SoftHSM2 for testing) support it. TPM 2.0 as a second target for edge deployments. Cloud-KMS as a v1.11 extension.
2. **Address V1.9 review findings** as the warm-up sprint (§2 above, especially CRITICAL #1 and HIGH #7).
3. **`crates/atlas-signer/src/master_seed.rs`** — `MasterSeedHkdf` trait + dev impl (gated behind `ATLAS_DEV_SEED=1`).
4. **`crates/atlas-signer/src/hsm/`** (NEW in-tree module behind the `hsm` Cargo feature) — PKCS#11 impl using the `cryptoki` 0.12 crate. *(Originally planned as a sibling `crates/atlas-signer-hsm` crate; folded back into `atlas-signer` because Cargo's no-cycles rule prevented the binary from dispatching to a sibling that itself depended on `atlas-signer` for the `MasterSeedHkdf` trait. Same operator opt-in semantics, no Cargo cycle. The feature flag is the operator-visible boundary: builds without `--features hsm` ship a stub that fails closed with `MasterSeedError::Unavailable`.)*
5. **Plumb the trait through** `keys.rs::derive_workspace_signing_key`, `per_tenant_identity`, all subcommand handlers.
6. **Production gate inversion** — refuse to start without `ATLAS_DEV_SEED=1` or a configured HSM source.
7. **Adversary tests** — HSM unreachable at startup → refuse; HSM derive returns wrong-length output → refuse; concurrent HSM derive races → serialise.
8. **Smoke test** — extend to use SoftHSM2 (open-source PKCS#11 impl) so CI can exercise the full HSM path.
9. **Operator runbook** — sealed-seed import ceremony, key-handle backup, recovery-from-HSM-failure.
10. **Run all tests + clippy + smoke + parallel `code-reviewer` + `security-reviewer`** — block on CRITICAL/HIGH.
11. **Docs** — `ARCHITECTURE.md` §7.5 "Sealed master seed (V1.10 — shipped)", `SECURITY-NOTES.md` rewrite the master-seed paragraph from "residual risk" to "addressed", `OPERATOR-RUNBOOK.md` HSM ceremony.

### Backwards compatibility

- V1.5–V1.9 bundles continue to verify (same kids, same canonical bytes — the V1.10 change is issuer-side only).
- V1.10 issuer with HSM seed produces byte-identical pubkeys to V1.9 dev seed **only if the HSM-sealed seed equals the V1.9 dev seed**. Migration is a fresh-rotation event, not a transparent upgrade. Operators run `rotate-pubkey-bundle` on every workspace once after the HSM is seeded.
- If V1.10 ships with a different sealed seed than the V1.9 dev seed, every existing per-tenant trace becomes unverifiable against the new bundle. **Document this loudly in OPERATOR-RUNBOOK.md.**

### Testing

- **Unit:** `MasterSeedHkdf` dev impl produces V1.9-compatible output. HSM impl produces deterministic output across signer restarts.
- **Adversary:** HSM unreachable → refuse to start (not silently fall back to dev). HSM returns wrong-length output → refuse. Concurrent derive serialised.
- **Smoke:** end-to-end with SoftHSM2 produces ✓ VALID bundle.
- **Cross-workspace isolation property** preserved: derive(seed, "alice") ≠ derive(seed, "bob") regardless of seed source.

## 4. Footguns / hard-won learnings

### Carried forward (still load-bearing in V1.10)

1. **`anchor-chain.jsonl` is append-only and irreversible** — sole writer is the issuer. (V1.7)
2. **Chain head canonicalisation must be byte-identical between issuer and verifier.** Pinned golden tests. (V1.7)
3. **Verifier may NOT trust `trace.anchor_chain.head` field** — recompute. (V1.7)
4. **Sigstore Rekor v1 shard roster is a source change** — adding tree-IDs requires crate version bump. (V1.7)
5. **Every signer-stdout boundary in TS uses `parseAnchorJson`** (lossless-json). Don't add a new `JSON.parse(stdout)` site without re-deriving why. (V1.8)
6. **`cargo` not in default PATH on Windows** — `/c/Users/nelso/.cargo/bin/cargo.exe`.
7. **Smoke at `apps/atlas-mcp-server`** — `pnpm --filter atlas-mcp-server smoke` from root.
8. **Single canonicalisation source — TS never re-implements anything cryptographic.** Always shell to `atlas-signer`. (V1.5+)
9. **HKDF info parameter is the trust boundary for per-tenant key separation.** Always domain-prefix. (V1.9)
10. **Backward compat lenient mode is a trust trade-off, not a free win.** Strict mode is the security boundary. (V1.9)
11. **Don't re-derive a pubkey at verify time.** The verifier reads from the bundle; never knows the master seed. (V1.9)

### NEW for V1.10

12. **HSM key handles are not byte arrays.** Any V1.9 function signature taking `&[u8; 32]` for seed material must be redesigned before V1.10 implementation begins. The trait abstraction belongs in V1.10's design phase, not its implementation phase.

13. **Production gate must be a positive opt-in, not a negative guard.** `ATLAS_PRODUCTION=1` (V1.9) was correct as a kill-switch but wrong as a permission model. V1.10 inverts: refuse to start unless explicitly opted-in (HSM configured, or `ATLAS_DEV_SEED=1` for dev/CI).

14. **`std::env::set_var` in tests races on multi-core CI.** `serial_test` annotation or env-injection refactor — never rely on cargo defaults to be single-threaded.

15. **Strict mode unreachable from `atlas-verify-cli` makes the security story fictional.** Every documented security boundary must be exercisable from the auditor-facing tool. Add the flag.

16. **`Base64UrlNoPad` is not enough — Ed25519 pubkeys are exactly 43 chars.** Length-pinning at the schema boundary catches signer-side regressions before they pollute the bundle.

17. **Seed rotation invalidates all historical traces.** Document this in the runbook before an operator discovers it the hard way. V1.10 HSM sealing makes seed rotation rare but not impossible.
