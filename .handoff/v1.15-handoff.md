1. **Browser-runtime hardening for `apps/wasm-playground/`** (HIGH if hosting decision lands). Currently mitigated by the §5 escalation framing in CONSUMER-RUNBOOK + the §6 reproduce-from-source fallback.
3. **Auto-verify CI action for downstream consumers** (MEDIUM). A reusable GitHub Action (`@atlas-trust/verify-wasm-pin-check@v1`) that runs `npm ci --ignore-scripts` + `npm audit signatures` + (optional) lockfile-integrity-format-validation as a single composite step, so downstream consumers don't have to compose the steps themselves. Lowers the bar from "read CONSUMER-RUNBOOK + write 4 lines of CI YAML" to "single-line `uses:` clause". Best surface for the runbook to live as code, not just docs.
4. **Two-job split for `wasm-publish.yml`** (LOW — Welle B residual). Split into build-and-publish under `id-token: write, contents: read` + release-upload under `contents: write, id-token: none`. Currently `contents: write` is co-located with `id-token: write` for the full job duration — a compromised step in the middle of the job has both write surfaces. Workflow-architecture refinement, not a bug; documented in the workflow YAML comments.
5. **Backup-channel registry-API equivalence** (LOW — Welle B residual). GH-Releases doesn't expose `npm view <pkg>@<v> dist.integrity` API equivalent; consumers fall back to the `tarball-sha256.txt` manifest. Could be remediated by publishing a `dist.json` with the `dist.integrity` SHA512 alongside the tarballs, so a programmatic recovery flow doesn't need to round-trip through `npm view`. Marginal benefit (operators are already running CONSUMER-RUNBOOK §6 quick-reference flow); revisit if a recovery-tooling effort is greenlit.
6. **Tag-signing enforcement** (MEDIUM — V1.16 if not deferred to V2). Currently V1.15 relies on tag-protection rules (only specific actors can push `v*` tags); ed25519 tag-signing (`git tag -s`) would close the residual gap of a compromised maintainer push. Documented in CONSUMER-RUNBOOK §6 step 2 as a future hardening path.
7. **Const-time policy expansion to `BTreeMap`/`BTreeSet` lookups on KIDs** (LOW — Welle A residual). `BTreeMap`/`BTreeSet` use `Ord`-based comparison which is not constant-time; explicitly documented as out-of-scope in `ct.rs` boundaries-not-covered. Mitigation candidate for V1.16: a wrapper type `KidMap` that uses linear-scan + `ct_eq_str` for keys, with a benchmark-driven cutover. Current scope: production code does not use `BTreeMap`-on-KID hot paths.

For the V1.16 session opener: if no specific scope is requested, default to **#1 (browser-runtime hardening)** — it's the only HIGH candidate and unblocks the playground hosting question that has been deferred since V1.14 Scope E. ## 3. Welle B — Backup distribution channel (SHIPPED — GitHub Releases, not GitHub Packages)

**Original plan: GitHub Packages mirror. Actual ship: GitHub Releases backup channel.** See post-Welle-B status §0 above for the pivot rationale (scope-ownership constraint: GitHub Packages requires npm scope to match GitHub repo owner, `@atlas-trust` ≠ `ThePyth0nKid` blocked direct mirror).

**Scope.** Today `@atlas-trust/verify-wasm` ships only via npmjs.org. If npm has an outage, is compromised at the registry layer, or is account-locked under a future maintainer-attack scenario, every downstream auditor that pulls the package fresh on a CI run loses the verifier. A second, independent distribution channel (GitHub Releases: byte-identical `npm pack` tarballs uploaded as release assets on every `v*` tag push) closes this single-point-of-failure without requiring a scope rename or consumer-side `.npmrc` change.

### Deliverables (as shipped)

- **B-a — `wasm-publish.yml` extension.** New step `Upload tarballs to GitHub Release (V1.15 Welle B backup channel)` runs alongside the existing `npm publish` on `refs/tags/v*` push. Copies pkg-web/pkg-node tarballs to suffix-disambiguated names (`-web.tgz` / `-node.tgz` — both pkg-* dirs share the same `package.json` name, so default `npm pack` filenames collide on a single GH-Release), generates `tarball-sha256.txt` via `sha256sum`, then `gh release view` → either `gh release upload --clobber` or `gh release create`. Uses `${{ github.token }}` (per-run, no rotation) — no new long-lived credential.
- **B-b — SHA256 manifest + SHA512 cross-verify.** `tarball-sha256.txt` (SHA256 hex) is generated in the same workflow run as the tarballs and uploaded as a release asset. The auditor runs `sha256sum --check` for transport-integrity (in-flight tamper detection). For origin integrity, the auditor cross-verifies the SHA512 against `npm view @atlas-trust/verify-wasm@<version> dist.integrity` — npm uses SHA512 base64, the manifest uses SHA256 hex, both covering the same bytes at different hash levels. The npm-side OIDC-signed SLSA L3 provenance attestation is the ultimate trust root and verifiable against either channel's bytes by recomputing the SHA512.
- **B-c — `OPERATOR-RUNBOOK.md §12`.** New "WASM verifier — backup-channel install via GitHub Releases (V1.15 Welle B)" section: when to fall back, `gh release download` flow (or `curl -fsSLO` without gh CLI), MANDATORY `sha256sum --check` for transport integrity, MANDATORY `npm view … dist.integrity` cross-verify for origin integrity, `npm install --ignore-scripts ./local.tgz` defence-in-depth on the install path, when to fall through to verifier-side rebuild from source. Compose-with-§11 step table.
- **B-d — `SECURITY-NOTES.md scope-b`.** Trust property (none new — distribution resilience only); residual risks (both-channels-on-GitHub correlation, GH-Release-only tamper without `NPM_TOKEN`, no registry-API equivalence, operator-driven not transparent failover, both channels could be tampered simultaneously); what NOT covered (complete registry mirror, lock-file pinning recommendations → Welle C, browser-runtime hardening → V1.16+). `scope-e` retroactively retrofitted to point at the Welle B fix for the latent step-level `id-token: write` placement bug.

### Latent V1.14 Scope E bug fixed in Welle B

The V1.14 Scope E workflow placed `id-token: write` at *step* level. GitHub Actions silently drops step-scoped `permissions:` keys (they are not a recognised YAML field at that scope; narrowing IS supported at job level by overriding workflow defaults, but step level is dropped entirely). `npm publish --provenance` would have failed at runtime on the first `v*` tag push because the OIDC token mint was never authorised — the bug was latent because no `v*` tag had been cut at V1.14 close. Welle B moves both `id-token: write` (for OIDC provenance) and `contents: write` (for `gh release upload`) to workflow level and removes the dead step-level block. A future Welle could split into a build-and-publish job (id-token: write, contents: read) + release-upload job (contents: write, id-token: none) to eliminate the OIDC + write-content co-location entirely; documented in the workflow YAML comment as a deferred refinement.
