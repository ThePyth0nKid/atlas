## 3. Welle B — Backup-registry mirror (`@atlas-trust/verify-wasm` to GitHub Packages)

**Scope.** Today `@atlas-trust/verify-wasm` ships only via npmjs.org. If npm has an outage, is compromised at the registry layer, or is account-locked under a future maintainer-attack scenario, every downstream auditor that pulls the package fresh on a CI run loses the verifier. A second, independent registry (GitHub Packages: `@atlas-trust/verify-wasm` under the `atlas-trust` GitHub org's npm scope) closes this single-point-of-failure.

### Deliverables (sketch — detail in V1.15 Welle B handoff, written when A ships)

- **B-a — `wasm-publish.yml` extension.** Existing job `publish-wasm-pkg-web` and `publish-wasm-pkg-node` extended to publish to BOTH registries. Use `actions/setup-node@<sha>` with `registry-url: https://npm.pkg.github.com/` for the second leg. Authentication via `GITHUB_TOKEN` (per-workflow, no rotation) for GitHub Packages; `NPM_TOKEN` (existing, granular) for npm. Both publishes use `--provenance`. Provenance attestations are registry-independent; the same SLSA L3 build claim covers both artefacts.
- **B-b — Smoke step extended.** The existing dual-target Node smoke runs against the artefacts BEFORE publish. After publish, add a small step that fetches the GitHub Packages tarball via `npm view @atlas-trust/verify-wasm@latest --registry=https://npm.pkg.github.com/` and asserts byte-identity (SHA256) against the npm tarball — both registries serve the same bytes, signed by the same SLSA-L3 attestation. Catches a registry-side sign-or-tamper mid-flight.
- **B-c — OPERATOR-RUNBOOK §11 alongside witness commissioning.** New §12 "WASM verifier consumption — backup-registry path". Steps for an auditor: (1) prefer npmjs.org, (2) fall back to GitHub Packages with the `.npmrc` snippet, (3) verify byte-identity locally if both registries are reachable.
- **B-d — Doc sweep.** SECURITY-NOTES.md `## scope-b — Backup-registry mirror (V1.15 — shipped)`: trust property (none new — distribution resilience only), residual risks (both registries simultaneously compromised — argue for verifier-side reproducibility from source as the ultimate fallback), what NOT covered (vendoring the WASM artefact directly into auditor repos — V1.16+ candidate).

### Open design question

GitHub Packages requires the consuming `.npmrc` to reference the `@atlas-trust` scope at `npm.pkg.github.com`. This means consumers need a deliberate `.npmrc` change to opt into the backup registry — it is NOT a transparent failover. Document this trade-off explicitly in OPERATOR-RUNBOOK §12: the fallback is operator-driven, not automatic. An automatic-failover proxy is V2-territory (it would need to be a hosted service at `npm.atlas-trust.io`).

## 4. Welle C — Lock-file pinning runbook for downstream consumers

**Scope.** A consumer of `@atlas-trust/verify-wasm` who runs `npm install @atlas-trust/verify-wasm@latest` on every CI build is silently trusting that npmjs.org served the same bytes today as yesterday. SLSA provenance + tarball artefact + tag-protection close registry-side compromise; the **last** layer is consumer-side lock-file pinning.

### Deliverables

- **C-a — `docs/CONSUMER-RUNBOOK.md`** (new file). Sections: *(1) Why pin?*, *(2) `package-lock.json` pinning by version + integrity hash*, *(3) `pnpm-lock.yaml` pinning*, *(4) `bun.lockb` pinning*, *(5) verifying SLSA provenance with `npm audit signatures`*, *(6) reproducing the published artefact from source if both registries are unreachable* (`cargo install wasm-pack` + `wasm-pack build --target web/nodejs` against the tagged commit).
