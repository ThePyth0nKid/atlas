## Was als nächstes BLOCKING ist

### npm 403: Two-Factor Authentication required

Letzte wasm-publish run (`25678588888` re-run, 2026-05-11 19:14 UTC):

```
npm error code E403
npm error 403 Forbidden - PUT https://registry.npmjs.org/@atlas-trust%2fverify-wasm
  Two-factor authentication or granular access token with bypass 2fa enabled is
  required to publish packages.
```

Provenance attestation **wurde** vor dem 403 zu Sigstore Rekor geschickt — logIndex `1510551161`. Das ist eine orphan attestation (matches gepacktes wasm but kein published package).

**Root cause:** Nelson's npm Account 2FA-Mode = `Auth & write` (strict). Diese mode blockt **alle** publish-actions ohne 2FA-prompt, including automation via granular access token. Granular tokens haben kein "bypass 2fa" flag (das ist nur für Classic automation tokens, deprecated).

**Fix:** 2FA-Mode auf `Auth only` reduzieren (web-UI, https://www.npmjs.com/settings/[username]/profile → Two-Factor Authentication → mode wählen). `Auth only` bedeutet 2FA für interactive logins, aber tokens dürfen publishen.

## Reminders / Gotchas / Anti-Footguns

1. **Workflow_dispatch on tag uses the workflow file FROM the tag commit, not from master.** Lesson aus today: `gh workflow run --ref refs/tags/v1.0.0` runs the wasm-publish.yml as it existed at commit `e97c025`, not the version on master HEAD. So workflow modifications in master only take effect via rerun-on-tag-push (Option A) or a fresh v1.0.1 tag.

2. **`github.ref_type == 'tag'` ist robuster als `startsWith(github.ref, 'refs/tags/v')` für `workflow_dispatch`.** GitHub's evaluation of `github.ref` on dispatch context appears unreliable (head_branch shows tag name unprefixed). Use `ref_type` canonical field.

3. **gh CLI `-f` vs `-F` für workflow inputs:** `-f` sendet form data als STRINGS; `-F` (capital) sendet RAW typed values. Für boolean inputs (`dry_run`): beide funktionieren wenn das workflow input mit `type: boolean` deklariert ist (GA coerces). Aber bei expression `!inputs.dry_run`: wenn input ist string `"false"`, `!"false"` = `!truthy` = `false`. Use `-F inputs[dry_run]=false` für boolean correctness wenn nesting via dispatches API call.

4. **npm 2FA mode `Auth & write` blockt ALLE automation tokens** für publish (granular OR classic). Only `Auth only` mode allows tokens publish without TOTP prompt. Documented in npm docs but easy to miss.

5. **Provenance attestation kann unabhängig published werden vom package.** Today's 403-failed publish DID push provenance to Sigstore Rekor (logIndex 1510551161). Das ist eine orphan attestation — when next publish succeeds, npm matches by content hash and likely re-uses (Sigstore content-addressed). Wenn nicht: orphan bleibt for posterity, harmless.

6. **NPM_TOKEN ist im chat geleakt** (today). Token aktuell live, used für today's failed publish. **Rotate immediately after successful publish lands** — bevor jemand mit transcript access publishes a malicious v1.0.1.

7. **4 wasm-publish runs auf v1.0.0 in history** als Audit-Trail:
   - `25678588888` — original tag-push (2026-05-11 15:07 UTC), publish skipped (NPM_TOKEN missing); re-run @ 19:14 UTC FAILURE (403 2FA)
   - `25690728625` — dispatch attempt 1 mit `--ref v1.0.0` (19:00 UTC), publish skipped (old workflow if-check)
   - `25690967890` — dispatch attempt 2 mit `--ref refs/tags/v1.0.0` (19:01 UTC), same skip
   - `25691569738` — dispatch attempt 3 post-fix-merge (19:13 UTC), same skip (workflow still from tag commit)
   - **Tomorrow's rerun of 25678588888** sollte succeed (tag-push event → original old workflow if: matches → npm publish fires → no 2FA block).

8. **Concurrency-group in modified workflow:** Welle 14a feat added `concurrency: group: wasm-publish-${{ github.ref }} / cancel-in-progress: false`. NICHT auf v1.0.0 tag-commit präsent (kam mit feat 577a407, post-tag). Effect: today's re-runs nicht serialisiert. Morgen's rerun also nicht (uses tag-commit workflow). Sobald 14a docs PR mergt, future v1.0.1+ tags würden den concurrency-block nutzen.
