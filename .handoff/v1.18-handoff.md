**Four root causes (one squash commit, `7d0a40f`):**

1. **Awk parser regression (initial commit `8122c43`, all three workflows).** The runner image picked up a SoftHSM2 build (regression observed on master 2026-05-04) that pads the Label field with trailing whitespace to a fixed column width; the prior `Label:[[:space:]]+...$` regex stopped matching once the padding appeared. **Fix shape:** awk now passes the label via `-v label=...` (so a future label string with awk-special characters cannot break the parser), captures slot id from `^Slot N` headers, strips leading `^[[:space:]]*Label:[[:space:]]+` + trailing `[[:space:]]+$` from any `Label:`-containing line, then exact-compares `line == label`. **Defence-in-depth (folded in same commit):** SLOT_ID is validated by `[[ "${SLOT_ID}" =~ ^[0-9]{1,10}$ ]]` (10-digit bound = max CK_SLOT_ID = 4294967295) before being written to `$GITHUB_ENV` — a tampered `softhsm2-util` binary emitting a Slot header line with an embedded newline can no longer inject arbitrary `key=value` pairs into the step environment. The FATAL diagnostic no longer echoes the raw SLOT_ID value (defence against log-injection by adversarial output).

2. **`pnpm/action-setup` ERR_PNPM_BAD_PM_VERSION (`hsm-wave3-smoke.yml`).** `with: version: 9` conflicted with the root `package.json` `packageManager: pnpm@9.12.0` field (Corepack convention); v4 of the action explicitly errors when both are set, to prevent silent dev-machine vs CI version drift. **Fix shape:** removed `with: version: 9`; the `packageManager` field (Corepack-resolved) is now the single source of truth. Inline comment in the workflow explains why the conflict exists so a future editor doesn't re-add the explicit version pin.

3. **cryptoki `Ulong: From<CK_ULONG>` cross-platform cast (`crates/atlas-signer/src/hsm/pkcs11.rs` `derived_key_template()` line 482).** `CK_ULONG` is the C `unsigned long` type; its width is platform-defined per the PKCS#11 spec — Windows is LLP64 → 32-bit on both 32-bit and 64-bit Windows; Unix-likes follow pointer width (LP64 → 64-bit on 64-bit Linux/macOS, ILP32 → 32-bit on 32-bit Linux / armv7 / wasm32). The cryptoki crate's `Ulong: From<CK_ULONG>` impl therefore resolves to `From<u32>` on `{ any-Windows, 32-bit Unix }` and `From<u64>` on 64-bit Unix. The old code `Attribute::ValueLen((DERIVED_LEN as u32).into())` worked on Windows but failed the trait bound on Linux CI. **Fix shape:** cfg-gated cast driven by `target_pointer_width` (NOT just `windows`/`not-windows` — that would be wrong on 32-bit Linux): `#[cfg(any(windows, target_pointer_width = "32"))] let derived_len_ck = DERIVED_LEN as u32;` + `#[cfg(all(not(windows), target_pointer_width = "64"))] let derived_len_ck = DERIVED_LEN as u64;`. Build-verified on Windows in 3.73s; CI validates Linux end-to-end.

4. **`pkcs11-tool --pin-source` removal (`hsm-witness-smoke.yml` 3 sites).** Newer `opensc-pkcs11-tool` (the build in the ubuntu-latest runner image as of 2026-05-04) dropped the `--pin-source env:VAR` flag entirely. **Investigation outcome:** evaluated `--pin-file <path>` as the natural successor (it's the equivalent of `softhsm2-util --pin-file` used elsewhere in the lane) — **upstream OpenSC `pkcs11-tool` does NOT expose `--pin-file` at all** (verified against `OpenSC/OpenSC` master `src/tools/pkcs11-tool.c` 2026-05-06; only `--pin`, `--so-pin`, `--puk`, `--new-pin` accept arg-line values, with the source explicitly warning "if used in scripts: careful!"). So `--pin "${USER_PIN}"` direct is the only available shape. **Threat-model analysis:** the PIN now appears on argv readable via `/proc/<pid>/cmdline` for the lifetime of the one-shot pkcs11-tool process. This is the **same threat-model class** as the existing `softhsm2-util --init-token --pin "${SOFTHSM2_PIN}"` call in the prior step — both are ephemeral one-shot processes on a single-tenant CI runner with a throwaway literal PIN ('1234' fallback when secrets are unset, e.g. fork PRs). `set +x` at the top of the step is the load-bearing guard against the PIN landing in the run log; the cleanup-block comment now also documents the scope of `set +x` so a future editor splitting the cleanup into its own `- name:` step knows to re-add `set +x` as the absolute first line. **Acceptable here**, NOT acceptable for production HSM (OPERATOR-RUNBOOK §2 already mandates `ATLAS_HSM_PIN_FILE` for production deploys; this exposure is structurally bounded to the single-tenant ephemeral CI runner case).

## 7. V1.18 Welle A (2026-05-06) — Multi-Issuer Sigstore Tracking ADR

**State summary:** the first Architecture Decision Record under `docs/ADR/` is shipped. ADR-Atlas-006 formally documents Atlas's three structurally distinct Sigstore touchpoints, the upstream multi-issuer landscape, and the explicit decision to **track (not adopt)** until concrete adoption triggers fire. Status field: `Tracking (not adopted)`. Date: 2026-05-06. Wave: V1.18 Welle A. PR: #5 (squash-merged as `c6aed22`).

**Why one ADR, not three (§1.2 of ADR):** all three touchpoints terminate at the same Rekor v1 production instance run by the Sigstore Foundation. A single Rekor incident degrades all three simultaneously; the decision-space (track, prepare, adopt, fall back) is identical for all three; splitting would force three near-duplicate decision records that drift over time.

**Three touchpoints documented (§1.1):**

- **Touchpoint A — Rekor v1 anchor verification (Atlas trust-core).** `crates/atlas-trust-core/src/anchor.rs` pins `SIGSTORE_REKOR_V1_PEM` (P-256 SPKI pubkey at line 83) and a 3-element `SIGSTORE_REKOR_V1_TREE_IDS` roster (line 138: active production shard `1_193_050_959_916_656_506` + two historical shards `3_904_496_407_287_907_110` and `2_605_736_670_972_794_746`). `SIGSTORE_REKOR_V1_ACTIVE_TREE_ID` (line 98) is the *issuer-side* anchor (atlas-signer always posts to the active shard); the *verifier-side* acceptance check `is_known_sigstore_rekor_v1_tree_id` does roster membership against the full slice so historical anchors captured under earlier shards still verify. Atlas accepts an anchor as Sigstore-verified if (a) its tree_id is a roster member, (b) the C2SP-format origin line `"rekor.sigstore.dev - {tree_id}\n"` reconstructs against that tree-ID, and (c) the anchor signature verifies against the pinned pubkey. Adding a new shard is intentionally a source change requiring a crate-version bump — silent acceptance of unknown tree-IDs is exactly the forgery primitive the roster forbids.
- **Touchpoint B — npm publish `--provenance` (`wasm-publish.yml`).** OIDC token from GitHub Actions ID-token service → Fulcio short-lived signing certificate → tarball SHA512 + workflow metadata signed → Rekor-attestation written. Single-issuer dependency: Fulcio + Rekor v1.
- **Touchpoint C — `npm audit signatures` round-trip (Welle A consumer-side).** `verify-wasm-pin-check@v1` Layer 3 self-test verifies the OIDC-Sigstore chain back to a configured trust root. Rekor outage during a consumer's CI run downgrades Layer 3 from active to passive — `docs/SECURITY-NOTES.md` **scope-k** covers this as a known degradation, including the air-gapped/offline-mirror note that explicitly bounds Sigstore network dependencies.

**Adoption triggers (§4):**

- **Trigger A** — npm trust root gains a second Fulcio CA OR a second Rekor log entry. Verification procedure: inspect npm bundled `trusted_root.json` at `$(npm root -g)/npm/node_modules/sigstore/store/trusted_root.json` + cross-reference upstream `sigstore/root-signing` + foundation announcement. **Known reproducibility weakness** acknowledged in-ADR: until npm exposes the trust root as first-class CLI output, the trigger is observable but not single-command-checkable.
- **Trigger B** — Rekor v2 issued for `@atlas-trust/verify-wasm`. Adoption work: extend `anchor.rs` for v2 entries via `SIGSTORE_REKOR_V2_PEM` + `SIGSTORE_REKOR_V2_TREE_ID` pair, update `crates/atlas-signer/src/rekor_client.rs` to negotiate v2 endpoints, port golden-test fixtures, validate cross-format mixed-mode (v1+v2 anchors in same chain). Estimate ~1 week.
- **Trigger C** — Sigstore Public Good extended incident (≥24h availability/integrity loss, or unscheduled `SIGSTORE_REKOR_V1_PEM` rotation, or documented integrity breach). Atlas response: **operator-runbook activation** (switch consumer guidance to Welle B SSH-tag-only verification), with adoption planning starting in parallel. **Documentation gap acknowledged in-ADR**: CONSUMER-RUNBOOK has no §10 "Sigstore Public Good incident — extended degradation protocol" section. Filling this gap is an **explicit prerequisite for Trigger C readiness** — recorded as a V1.18 Welle B follow-on task (`docs(consumer-runbook)/sigstore-incident-protocol`) to ship before the next quarterly ADR refresh.
- **Trigger D** — Atlas-side compliance requirement (customer audit, regulatory framework, SOC-2 line item) explicitly demands multi-issuer redundancy. Differs from A/B/C in that response is requirement-driven, not upstream-readiness-driven; may force a partial / alpha-quality multi-issuer verifier ship to meet a contractual deadline.

**Triggers that explicitly do NOT fire adoption (§4):** foundation announcements without deployed second issuer, non-npm trust-root additions, internal-corporate-only deployments, academic schemes without registry adoption. Principle: track production-deployable redundancy, not architectural readiness.

**Mitigations recognised (§6) — why "tracking, not adopting" is acceptable today:**

- Welle B SSH-tag primary trust root + Welle C trust-root mutation defence cover the ship-side trust root independently of Sigstore state.
- Compile-time pinned Rekor v1 anchor in `anchor.rs` (false-negative-safe under key rotation; operator response is to ship a verifier-core update with the new pin, golden-test verified).
- TOCTOU-pinned GH-Release via `--target ${VERIFIED_SHA}` (V1.17 Welle B): Sigstore-side compromise of npm provenance cannot retroactively rewrite the GH-Release-distributed tarball binding.
- Per-algo minimum base64 payload length checks in `verify-wasm-pin-check@v1` (V1.17 Welle A post-review): empty-hash bypass blocked even if a Sigstore-level signature were somehow valid for an empty payload.
- A Sigstore Public Good incident degrades **one** of three Atlas verification surfaces; **two** of three (Welle B+C, anchor-pinning) remain fully active.

**Atlas-side preparation (§5) — three follow-ons recorded:**

- **§5.1** — refactor `anchor.rs` to a `RekorIssuer { name, pem, active_tree_id, tree_id_roster }` registry pattern + `&[&RekorIssuer]` slice. Mechanical, pure-internal-API, no semantics change. Lowers future Rekor-v2-adoption cost. **Status:** deferred to V1.18 Welle B+ as a candidate.
- **§5.2** — document the inline-pin-update protocol in `OPERATOR-RUNBOOK.md` §15 (TBD): PR review requirements, golden-test fixture regeneration, cross-version-anchor compatibility test. Currently implicit. **Status:** deferred to V1.18 Welle B as a `docs(v1.18/welle-b)` task.
- **§5.3** — `SIGSTORE_DEGRADED_OK` flag for `verify-wasm-pin-check@v1`. **OPEN PROBLEM (NOT solved by an obvious quick fix).** The naive defence — "require a fresh liveness probe to `https://rekor.sigstore.dev/api/v1/log/publicKey` before accepting degraded mode, with a per-repo per-week cap" — is **defeatable by the same network-positioned adversary class the feature is meant to exclude**. A targeted DNS or BGP hijack against `rekor.sigstore.dev` for the duration of one probe causes the probe to return 200 (because the probe and the audit round-trip both resolve to the same domain via the same path) while the attacker simultaneously serves a forged "no attestation" response or a stale-but-syntactically-valid attestation. The probe and the protected operation share a fate. The per-week cap mitigates opportunistic abuse but not a targeted attacker. Adequate independence requires probing through a route that does NOT share a fate with the audit round-trip — three candidates sketched: (i) cross-log consistency check against an independent Sigstore-compatible log (only possible after multi-issuer adoption — circular), (ii) Sigstore Foundation status page over a separately-anchored TLS certificate chain (different CA, different DNS authoritative servers), (iii) Atlas-operated witness recording canonical Rekor checkpoints out-of-band, itself anchored. None of (i)/(ii)/(iii) is trivially correct. **Decision:** record intent + naive defence + why-it-fails + defer the actual design to a future ADR-Atlas-00X addressing fate-shared-channel verification specifically. **Do NOT ship `SIGSTORE_DEGRADED_OK` until that ADR lands.**

## Standing protocol (re-read each session)

1. **Implement → parallel review → fix → single coherent commit.** For any non-trivial change, run `code-reviewer` and `security-reviewer` agents in parallel against the staged diff. Fix all CRITICAL and HIGH findings in-commit (do not split). Selectively fix MEDIUMs that don't expand scope. Document the fix triage in the commit message body. Single `feat(v1.X/welle-Y)` or `docs(v1.X/welle-Y)` commit, then a separate `docs(v1.X/handoff)` commit prepending the new status block above §0 in this file.

2. **PR-then-merge, not direct push.** Any change to `master` goes through a PR. The Welle C `verify-trust-root-mutations` gate runs on every PR; trust-root-surface changes require a commit signed by a key in the *prior* `.github/allowed_signers`. Pure-docs PRs (this file, ADRs, runbooks outside the protected list) flow through cleanly.

3. **Anti-drift gates before any V1.18+ edit.** Run all four (Welle A 40/40, Welle B 14/14, Welle C 17/17, playground CSP) and confirm green BEFORE touching anything. If any gate is red on a clean tree, the harness itself has regressed — stop and investigate before adding new work.

4. **German in chat, English in code/comments/commits.** User preference is German in conversation; project artefacts (commit messages, PR bodies, ADRs, code comments) stay English for downstream-consumer compatibility.

5. **Standing directive:** *"Geh mit A und hol es bestmöglich aus dem Produkt raus. Entscheide du, was jetzt das Bestmögliche ist anhand deiner Informationen, die du hast. Immer beste Sicherheit und beste Codequalität."* — autonomous design choices when explicit, security-first, codequality-first; report the decision tree in the commit message rather than asking for explicit approval on each design dimension.
