### V1.7 capabilities (shipped)

#### Anchor-chain tip-rotation

Every `atlas_anchor_bundle` invocation now extends an append-only anchor-chain log (`data/{workspace}/anchor-chain.jsonl`). Each line is one `AnchorBatch`:

```rust
pub struct AnchorBatch {
    pub batch_index: u64,          // 0, 1, 2, ...
    pub integrated_time: i64,      // Unix seconds
    pub entries: Vec<AnchorEntry>, // the anchors themselves
    pub previous_head: String,     // hex of chain_head_{n-1}; "00..." for batch 0
}

pub struct AnchorChain {
    pub history: Vec<AnchorBatch>,
    pub head: String,              // blake3(canonical batch serialization)
}
```

Chain head is blake3 with domain prefix `atlas-anchor-chain-v1:`. Issued to `trace.anchor_chain` on export (optional field; old bundles pass under lenient mode).

**Verifier side:**
- `verify_anchor_chain(&AnchorChain)` walks history, validates batch_index monotonicity, recomputes each head, checks previous_head continuity.
- `VerifyOptions::require_anchor_chain` (default false) enforces chain presence in strict mode.
- `anchor_chain_coverage` cross-check: every entry in `trace.anchors` (current snapshot) appears byte-identical in some history batch. Catches mixed-mode workspaces.

**Issuer side:**
- `atlas-signer anchor --chain-path <file>` reads JSONL (find next batch_index and previous_head), appends new batch atomically (tmp + fsync + rename).
- Sole writer of the chain file.
- Mock-issuer path only; Sigstore path gated (see constraint below).

#### Sigstore Rekor v1 shard roster

Replaced strict tree-ID equality check with roster membership. Three shards, all signed by the same pinned ECDSA P-256 pubkey:

```rust
pub const SIGSTORE_REKOR_V1_TREE_IDS: &[i64] = &[
    1_193_050_959_916_656_506,   // active (current production)
    3_904_496_407_287_907_110,   // historical shard
    2_605_736_670_972_794_746,   // historical shard
];
```

`is_known_sigstore_rekor_v1_tree_id(tree_id)` replaces equality check. No cross-shard replay (C2SP origin embeds tree_id; mismatch fails signature verify). Issuer still posts to active shard only (intentional asymmetry).

### What V1.7 cannot do yet (the gap V1.8 should close)

**Two constraints in V1.7:**

1. **Anchor-chain extension is gated to the mock-issuer path.** Sigstore tree_id values (~2^60) exceed `Number.MAX_SAFE_INTEGER` (~2^53) in JavaScript. JSON round-trip through the MCP server silently loses the low digits of tree_id, corrupting the entry. V1.7 disables chain extension when `rekor_url` is set; the issuer skips appending to anchor-chain.jsonl on the Sigstore path.

2. **The verifier accepts Sigstore entries as part of a mock-built chain only.** If you anchor via Sigstore, `trace.anchor_chain` is undefined (or left as is from a prior mock-phase). The verifier's `anchor_chain_coverage` check will fire if you then export because Sigstore entries are absent from the mock chain. Mixed-mode anchoring is loud at audit time.

**That's V1.8.**

## 4. V1.8 primary — precision-preserving JSON parser

### Goal

JSON parsing in the MCP server must preserve Sigstore Rekor tree_id values (~2^60 integers) without truncation to JavaScript `Number.MAX_SAFE_INTEGER` (~2^53).

Current behaviour:
- `JSON.parse("{ \"tree_id\": 1193050959916656506 }")` silently loses the low digit.
- The entry is built with the corrupted tree_id, making it unverifiable on re-export.

### Solution: `lossless-json` library

The `lossless-json` npm package implements RFC 7493 (I-JSON, Interoperable JSON) parsing that preserves large integer bit patterns:

```typescript
import { parse, stringify } from 'lossless-json';

const entry = parse('{"tree_id": 1193050959916656506}');
// entry.tree_id is a LosslessNumber object with full precision

const reencoded = stringify(entry);
// "1193050959916656506" (exact match)
```

- Drop-in replacement for `JSON.parse` / `JSON.stringify` in specific code paths (anchor entry parsing).
- Leaves the rest of the MCP server untouched (normal JSON for traces, bundles, etc.).
- Zero runtime cost if used selectively.

### Implementation plan

1. Add `lossless-json` to `apps/atlas-mcp-server/package.json`.
2. Wrap `AnchorEntry` parsing (in `lib/signer.ts` and `tools/anchor-bundle.ts`) to use `parse()` instead of `JSON.parse()`.
3. On re-serialization (export), use `stringify()` to emit exact tree_id values.
4. Update the `AnchorEntrySchema` Zod validator to accept `LosslessNumber` in the tree_id field (or convert to `BigInt` internally).
5. Smoke test must pass with full chain extension on Sigstore path.

### Testing

- Unit test: parse/round-trip a real Sigstore entry with tree_id `1193050959916656506`. Assert exact bit match.
- Integration test: issue an anchor via live Sigstore (wiremock), extend chain, export, verify. Anchor-chain must include the Sigstore entry with correct tree_id.
- No precision loss in any test.

## 5. V1.8 secondary — extend chain coverage to Sigstore entries

### Current limitation

`anchor_chain_coverage` in `verify.rs` fires if `trace.anchors` contains entries not found in `trace.anchor_chain.history`. For mock-path anchors, the entry is always in the history. For Sigstore-path anchors (V1.7), the entry is absent (chain extension was deferred), so the check fires: mixed-mode rejected.

### V1.8 expansion

Update `anchor_chain_coverage` to:

1. If `trace.anchor_chain` is present: walk `history[]` and collect all entries.
2. For each entry in `trace.anchors` not found in history:
   - If it's a **Sigstore entry** (`log_id` matches `SIGSTORE_REKOR_V1_LOG_ID`), it's okay — Sigstore path extension was deferred at issue time.
   - If it's a **mock entry** (`log_id` matches mock), that's an error — mock entries must be in the chain if the chain exists.
3. If the entry is neither, fail: unknown log_id (shouldn't happen; already rejected at anchor verify time).

This allows:
- Full mock-path audit trails (all entries in chain).
- Mixed-mode bundles that started with mock, transitioned to Sigstore later (chain has only mock entries; Sigstore entries are accepted separately because they are from the active shard at that time).
- Rejection of mixed-mode bundles that reorder or drop past mock entries.

### Testing

- Test case: workspace anchors via mock (2 batches, 4 entries), then switches to Sigstore (1 entry). Export → anchor_chain has 2 batches, trace.anchors has 5 entries (4 mock + 1 Sigstore). Verify: passes under lenient + `anchor_chain_coverage` check accepts the Sigstore entry.

## 7. Footguns / hard-won learnings (carry forward + new)

### Carried from V1.7 (still load-bearing)

1. **`anchor-chain.jsonl` is append-only and irreversible** — sole writer is the issuer. Loss of the file breaks future chain extension for that workspace. Operator needs rotation ceremony to bridge to a new chain. Documented in README.

2. **Chain head canonicalisation must be byte-identical between issuer and verifier** — same single-source-in-Rust discipline. Pinned golden tests.

3. **Verifier may NOT trust the trace's `anchor_chain.head` field** — recompute from `history[]` and compare.

4. **`batch_index` must be sequential, no gaps** — `history[i].batch_index == i`. Gaps → reject.

5. **Sigstore Rekor v1 shard roster is a source change** — adding new tree-IDs requires a crate version bump. Silent acceptance is forbidden.

6. **JS `parseInt` and `Number()` truncate large integers** — use `lossless-json::parse()` for Sigstore entries. Splicing into fixture JSON via regex string replacement, not `JSON.stringify()`.

7. **`cargo` not in default PATH on Windows** — every shell invocation needs `export PATH="/c/Users/nelso/.cargo/bin:$PATH"`.

8. **Smoke is at `apps/atlas-mcp-server`** — `pnpm --filter atlas-mcp-server smoke` from root.

9. **Pinned hex values in tests** — anchor-chain head pins (genesis batch, known hash). If refactored, algorithm didn't change: investigate before updating pins.

10. **Single canonicalisation source** — TS never re-implements chain head computation. Always shells to `atlas-signer` or verifier. Keep it that way.

### NEW for V1.8

11. **`lossless-json` is a selective parsing gate.** Don't parse all JSON through it; it's slower and unnecessary for most payloads. Use only where precision matters: Sigstore entries. Profile to ensure no performance regression in smoke tests.

12. **`LosslessNumber` serialization:** When re-emitting, use `stringify()` from `lossless-json` to preserve the `LosslessNumber` type's bit pattern. Normal `JSON.stringify()` on a `LosslessNumber` object will stringify the wrapper, not the value. Watch for accidental precision loss at the boundary.

13. **Mixed-mode audit trail complexity** — `anchor_chain_coverage` with Sigstore carve-out is more nuanced than V1.7. Add comments in `verify.rs` explaining the three cases (in-chain mock, deferred Sigstore, unknown). Reviewers will ask for clarity.

14. **Wiremock doesn't produce valid Sigstore checksums.** If you test chain extension with wiremock + lossless-json, the entry tree_id will be exact (good), but the inclusion proof and signature verification are wiremock-mocked. That's fine for testing "the chain extended"; real e2e (live Rekor) is optional.
