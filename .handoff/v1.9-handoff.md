### V1.8 capabilities (shipped)

#### Precision-preserving anchor JSON pipeline

The MCP server routes every signer-stdout boundary (`signEvent`, `anchorViaSigner`, `chainExportViaSigner`) and on-disk anchor parse (`anchors.json`, `anchor-chain.jsonl`, `trace.json` export) through `lossless-json` via the helper `apps/atlas-mcp-server/src/lib/anchor-json.ts`:

```typescript
export const INTEGER_LITERAL_REGEX = /^(?:0|[1-9]\d*)$/;  // non-negative

function parseSafeNumberOrLossless(raw: string): number | LosslessNumber {
  if (INTEGER_LITERAL_REGEX.test(raw) && isSafeNumber(raw)) {
    return Number(raw);                  // safe-range integer → native
  }
  return new LosslessNumber(raw);        // oversized / float / sci → wrap
}
```

Integer literals in safe range stay native `number`. Oversized integers (Sigstore Rekor v1 `tree_id` ~2^60), fractionals, and scientific-notation literals wrap as `LosslessNumber` whose `.value` preserves the source string. `stringifyAnchorJson` re-emits wrappers as the exact digit string. Schema-side: `LosslessIntegerSchema` requires the digit string to match the regex AND fit within 19 decimal digits (i64 magnitude ceiling).

The V1.7 Sigstore-path chain-extension gate is removed (`apps/atlas-mcp-server/src/tools/anchor-bundle.ts`). Both paths now extend `anchor-chain.jsonl`. The Rust signer remains the sole writer.

#### Sigstore-deferred coverage carve-out

`anchor_chain_coverage` in `crates/atlas-trust-core/src/verify.rs` classifies each entry in `trace.anchors`:

- **covered** — entry is byte-identical to a row in `trace.anchor_chain.history`.
- **sigstore-deferred** — entry's `log_id == SIGSTORE_REKOR_V1_LOG_ID` and is absent from chain history. Accepted because Sigstore Rekor v1's own publicly-witnessed log gives equivalent monotonicity, and per-entry `verify_anchors` (runs unconditionally before coverage) still validates the full inclusion proof and checkpoint signature against the pinned PEM.
- **uncovered** — any other absence. Coverage fails (preserves V1.7 anti-rewrite for the mock path).

V1.7-issued bundles (Sigstore anchors not in chain) keep verifying. Mixed-mode batches (mock + Sigstore in same batch, recorded in chain) chain-verify properly.

### What V1.8 cannot do yet (the gaps V1.9 might close)

1. **Single-tenant Atlas anchoring key.** All workspaces sign with one Atlas anchoring keypair, derived from a single deterministic seed (`atlas-signer/examples/seed_bank_demo.rs`). Compromise of that one key compromises every workspace's anchor chain. There is no per-tenant isolation.
2. **No HSM/TPM sealing.** The Atlas anchoring secret is in process memory at sign time. A memory-disclosure attack on the signer host exfiltrates the master key.
3. **Single-issuer dependency.** Only one Sigstore Rekor v1 instance is trusted (the production one). If it goes offline, all live-anchoring stops.
4. **No external witness over chain head.** The chain head is signed implicitly via the entries Sigstore witnessed, but the chain BATCH structure (which entries grouped, which `integrated_time`) is purely Atlas-claimed. A second independent witness (timestamping authority, witness coalition) would reduce trust-in-issuer.
5. **WASM verifier is build-only.** `crates/atlas-verify-wasm` compiles but isn't packaged for browser/Node distribution. An auditor must use the native CLI.
6. **Verifier output is human-prose.** No machine-readable JSON output for CI integration.
7. **No live-Rekor CI smoke.** Live-Sigstore is exercised via wiremock only. Network drift would not be caught until manual operator runs.

## 2. Prime directive — the trust property

> An auditor with the trace bundle and a fresh `atlas-verify-cli` binary can establish, **fully offline**, that every claim was bit-identical to what was signed and anchored, **and that the anchor history is monotonic** — no past anchor batch has been silently rewritten between T₀ and T_now.

V1.8 closes the precision-loss gap. V1.9 should close the next-most-load-bearing gap. **The strongest single V1.9 candidate is per-tenant Atlas anchoring keys**, which removes the "single key compromise = all workspaces compromised" blast radius.

Every V1.9 design decision filters through:

- The verifier may NOT make a network call. Multi-tenant key isolation must be visible offline (each workspace's pubkey appears in its bundle).
- Single canonicalisation source in Rust. Key derivation must live in the signer crate, never duplicated in TS.
- **Forwards compatibility:** V1.5/V1.6/V1.7/V1.8 bundles signed with the single-tenant seed must continue to verify. V1.9 introduces a new key path additively.
- Operator ergonomics: the operator must not need to manually manage keys. Derivation must be automatic and deterministic from a workspace identity + master seed.

## 4. V1.9 primary — per-tenant Atlas anchoring keys

### Goal

Each workspace's anchor entries are signed with a workspace-specific Ed25519 keypair derived deterministically from a master seed + workspace_id. Compromise of one workspace's key does not compromise others. The master seed itself remains a single secret; multi-master-seed HSM sealing is V1.10.

### Why HKDF-SHA256

- Standard, audit-friendly (RFC 5869). Security analysis is mature.
- Deterministic — same `(master_seed, workspace_id)` → same keypair across signer restarts. Required for chain head reproducibility.
- Domain-separated via the `info` parameter; no length-extension or context-confusion attacks.
- Already a primitive in `ring` / `hkdf` crates; one dependency.

```rust
// crates/atlas-signer/src/keys.rs (new module)
pub fn derive_workspace_keypair(
    master_seed: &[u8; 32],
    workspace_id: &str,
) -> ed25519_dalek::SigningKey {
    let hk = Hkdf::<Sha256>::new(None, master_seed);
    let mut key_bytes = [0u8; 32];
    hk.expand(
        format!("atlas-anchor-v1:{workspace_id}").as_bytes(),
        &mut key_bytes,
    ).expect("HKDF expand: 32 bytes is well within max output");
    ed25519_dalek::SigningKey::from_bytes(&key_bytes)
}
```

### Bundle representation

`PubkeyBundle.keys` becomes a map keyed by `kid`:

- Legacy single-tenant kid (`atlas-anchor:default`) — present for backwards compat. Continues to sign workspaces that opt out.
- Per-tenant kids (`atlas-anchor:<workspace_id>`) — derived on first anchor in that workspace; cached in bundle.

Anchor entries reference the `kid` they were signed with. `verify_anchors` looks up the pubkey by `kid` from the bundle (already does — `PubkeyBundle` is the trust root for offline verify).

### Trust property under per-tenant keys

- Compromise of workspace A's key forges workspace A's anchor chain only. Workspace B's chain is independent.
- The master seed compromise still compromises everything (HKDF is one-way, but the seed → all keys). HSM sealing in V1.10 closes this. Document the residual risk in `SECURITY-NOTES.md`.
- `verify_anchors` does not need new logic; it already dispatches by `kid`. The change is mostly issuer-side + bundle-construction.

### Backwards compatibility

- V1.5/V1.6/V1.7/V1.8 bundles with `kid = atlas-anchor:default` continue to verify.
- `VerifyOptions::require_per_tenant_keys` (default false) tightens: every anchor must use a per-tenant kid in matching workspace_id format. V1.5-V1.8 fail under strict; V1.9 passes.

### Implementation plan

1. Add `hkdf = "0.12"` and `sha2` to `crates/atlas-signer/Cargo.toml`.
2. New module `crates/atlas-signer/src/keys.rs` with `derive_workspace_keypair`.
3. Update `atlas-signer sign --workspace …` to derive the keypair via HKDF when a per-tenant flag is set; default to legacy single-tenant keypair for backward compat.
4. Update `atlas-signer anchor` similarly: derive the issuer keypair per workspace.
5. Update `PubkeyBundle::deterministic_hash` to handle the new kid shape (no change if the bundle already serializes via canonical sort).

8. Update MCP server to surface a per-tenant kid (or default) per workspace.

### Testing

## 7. Footguns / hard-won learnings (carry forward + new)

### Carried from V1.5–V1.8 (still load-bearing)

1. **`anchor-chain.jsonl` is append-only and irreversible** — sole writer is the issuer. Loss of the file breaks future chain extension for that workspace. Operator needs rotation ceremony.

2. **Chain head canonicalisation must be byte-identical between issuer and verifier** — same single-source-in-Rust discipline. Pinned golden tests.

3. **Verifier may NOT trust the trace's `anchor_chain.head` field** — recompute from `history[]` and compare.

4. **`batch_index` must be sequential, no gaps** — `history[i].batch_index == i`. Gaps → reject.

5. **Sigstore Rekor v1 shard roster is a source change** — adding new tree-IDs requires a crate version bump. Silent acceptance is forbidden.

6. **JS `parseInt` and `Number()` truncate large integers** — every signer-stdout boundary in TS uses `parseAnchorJson` (lossless-json). Don't add a new `JSON.parse(stdout)` site without re-deriving why.

7. **`cargo` not in default PATH on Windows** — every shell invocation needs `/c/Users/nelso/.cargo/bin/cargo.exe` or `export PATH=…`.

8. **Smoke is at `apps/atlas-mcp-server`** — `pnpm --filter atlas-mcp-server smoke` from root, or `pnpm run smoke` from inside.

9. **Single canonicalisation source** — TS never re-implements chain head computation, key derivation, or canonical-bundle hashing. Always shells to `atlas-signer`. Keep it that way.

10. **`lossless-json` parser is non-negative-integer only.** The regex `INTEGER_LITERAL_REGEX = /^(?:0|[1-9]\d*)$/` deliberately excludes leading minus. Don't relax this without re-deriving the float-vs-integer trap from V1.8 (where `1.193e18` was an integer-valued double that passed `z.number().int()` if returned native).

11. **`LosslessIntegerSchema` has a 19-digit ceiling.** Any tree_id past i64 magnitude fails at the Zod boundary, not later in Rust deserialization. If you ever need a `u64` field at the boundary (one decimal digit more), audit this constant.

12. **The Sigstore-deferred coverage carve-out is gated on `log_id == SIGSTORE_REKOR_V1_LOG_ID`.** That log_id is SHA-256(DER SPKI of the pinned key) — forging it is preimage resistance. The carve-out does not bypass per-entry inclusion-proof or checkpoint-signature verification. If you ever add a second trusted log family, decide explicitly whether it gets the same carve-out.

### NEW for V1.9

13. **HKDF info parameter is the trust boundary for per-tenant key separation.** Using just `workspace_id` without the `atlas-anchor-v1:` prefix would let an attacker who controlled `workspace_id` re-derive the same key used in a different domain (e.g. a future `atlas-policy` derivation). Always domain-prefix.

14. **Master seed handling is the residual single-point-of-failure.** Document this loud in `SECURITY-NOTES.md`. Do NOT claim per-tenant isolation without the caveat. HSM/TPM sealing in V1.10 closes this; until then, master seed compromise = full compromise.

15. **Backward compat lenient mode is a trust trade-off, not a free win.** A V1.9 verifier in lenient mode accepts both legacy single-tenant bundles AND per-tenant bundles. An attacker who can downgrade a workspace's bundle to legacy form bypasses per-tenant isolation. Strict mode (`require_per_tenant_keys = true`) is the real security boundary; document the gap.

16. **Bundle deterministic_hash must be stable across kid additions.** When `rotate-pubkey-bundle` adds a per-tenant pubkey, the bundle hash changes — that's expected and the trace's `pubkey_bundle_hash` will reflect the new bundle. But the canonical sort order must be deterministic so two operators running rotate-pubkey-bundle on the same workspace at the same wall time produce byte-identical bundles. Verify via golden test.

17. **Don't re-derive a pubkey at verify time.** The verifier reads pubkeys from the bundle. Re-deriving from `master_seed` would force the verifier to know the master seed, which is the opposite of what we want. Per-tenant keys are derived ONCE at issuer side; the verifier sees only pubkeys.

18. **Smoke test must exercise a non-default workspace.** The default `WORKSPACE = "ws-mcp-smoke"` constant in `apps/atlas-mcp-server/scripts/smoke.ts` would mask any "default workspace bypasses derivation" bug. Add at least one additional workspace_id to smoke or to a new e2e test.
