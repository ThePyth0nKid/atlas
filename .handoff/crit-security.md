# Crit: security on Atlas V2 Vision

> **Reviewer role:** adversarial security reviewer. Primary targets: Doc B (`v2-vision-knowledge-graph-layer.md` v0.5) + Doc C (`v2-risk-matrix.md` v0). Secondary: claims in Docs A, D, E that touch trust property.
>
> **Stance:** V1's trust property is sacred: Ed25519 + COSE_Sign1 over deterministic CBOR, blake3 hash chain, Sigstore Rekor v1 + RFC 6962 inclusion proofs, witness cosignatures, HKDF per-workspace keys, offline WASM verifier — all live, all pinned by byte-determinism tests (`signing_input_byte_determinism_pin`, `bundle_hash_byte_determinism_pin`, `mock_log_pubkey_matches_signer_seed`, `atlas_anchor_pubkey_pem_is_pinned`, `SIGSTORE_REKOR_V1.active_tree_id`). The job of this crit is to find every place V2 *almost* breaks that property — structurally, operationally, or legally.
>
> **Convergence check:** ≥5 strukturelle Punkte (Stärken) + ≥3 konkrete Edits → both met (11 Stärken / 8 konkrete Edits below).

## Stärken (was ist gut, sollte bleiben)

1. **Doc B §2.1 / §2.5 — Three-Layer Invariant is the right structural commitment.** Layer 1 (events.jsonl) authoritative; Layers 2 (FalkorDB) + 3 (Mem0g) declared explicitly non-trust-authoritative, derivative, rebuildable. This is the **single most important property of V2** and Doc B says it loud enough. The trust path still bottoms out in the offline WASM verifier against `events.jsonl` — V1's invariant survives by construction if this discipline holds.

2. **Doc B §2.5 — Mem0g malice scenario explicitly modelled.** "What if Mem0g returns false results" → answered with "cite event_uuid back to Layer 1, verifier resolves discrepancy." This is exactly the right framing: cache is *evidence-pointing*, never *evidence-producing*. The agent must always cite back to a signed event-uuid that the WASM verifier can validate independently.

3. **Doc B §2.7 — Per-workspace HKDF (V1.9) + per-agent DID treated as orthogonal, not replacing.** Event carries both `kid` (workspace-anchor key) and `author_did` (agent identity). Cross-workspace-replay defence (V1 property — `workspace_id` is folded into signing input per `crates/atlas-trust-core/src/verify.rs`) is preserved exactly. Two-layer identity is structurally clean.

4. **Doc B §2.10 — Federated witness uses the V1.13 cosignature primitive without modification.** Regulator-witness is just another `WitnessSig` entry in the existing roster — same `witness_signing_input`, same `WitnessFailureWire`, same threshold-strict path. No new cryptographic primitive, no new attack surface in `atlas-trust-core`. This is genuinely additive in the right way.

5. **Doc B §3.3 — Content/hash separation as the GDPR architecture.** `content_hash` signed; `content_pointer` resolves to a deletable blob. Tombstone-on-delete preserves "event existed, signed by author_did at time T" while letting the bytes go. This is the only architecturally sound resolution of GDPR Art. 17 vs immutability.

6. **Doc C R-A-01 — Projection Determinism CI Gate.** The proposed `projector-state-hash` invariant (replay events.jsonl → fresh FalkorDB → canonical hash → compare against pinned `.projection-integrity.json`) is the right *shape*: it mirrors V1's `signing_input_byte_determinism_pin` discipline. The pattern (CI tripwire that fires before code reaches a customer browser) is exactly the V1 anti-drift mechanism extended to V2.

7. **Doc C R-A-03 — Agent Identity Key Compromise explicitly flagged as Detectability: LOW, Reversibility: LOW.** Most agent-identity designs hand-wave revocation; Doc C names the structural problem: events signed in the compromise-window verify as valid, and they can't be unsigned. This is honest.

8. **Doc C R-C-01 — Post-Quantum migration hooks already exist in V1.** `Algorithm` enum is `#[non_exhaustive]` (per `SEMVER-AUDIT-V1.0.md` §1.1), so adding ML-DSA-65 is a SemVer-minor *additive* operation. Hybrid Ed25519 + ML-DSA-65 co-sign design is the right NIST-aligned path. The schema-version-v2 transition is the right insertion point.

9. **Doc C R-L-01 — Legal opinion gated as a blocker.** GDPR-counsel opinion on whether content-hash separation actually satisfies Art. 17 listed as pre-V2-α gate. Engineering analysis alone is correctly judged insufficient.

10. **Doc B §2.6 — Atlas Memory Skill compromise model.** Skill is a thin client; DID private key lives in skill-state, not in skill code. A malicious npm/PyPI publish of the skill cannot forge events under the original DID. This is a clean separation and Doc B states it explicitly.

11. **Doc C §"V2-α blocking risks" — Four blockers named.** R-A-01 (determinism CI gate), R-L-01 (GDPR architecture), R-L-02 (FalkorDB SSPL legal opinion), R-A-03 (revocation mechanism). Calling these "blocking" gives the V2-α plan a tested gate rather than a fuzzy good-intentions checklist.

## Probleme (was muss adressiert werden — by severity)

### CRITICAL

- **C-1 (Doc B §2.7, Doc C R-A-03): Revocation lag has no quantified bound.** Doc B §2.7 says: "Attacker can sign events under the DID until the revocation event is published… window bounded by anchor latency (typically minutes)." This is wrong on two counts:
  - (a) **Anchor latency does not bound attacker capability.** Sigstore Rekor anchor latency only constrains *how late we observe* the events — it does **not** prevent the events from existing and verifying ✓ VALID under the compromised key. An attacker who learns of the compromise and the proposed revocation channel before the operator does can sign events *after* the revocation event with an earlier `event.timestamp` (the `event.timestamp` field is set by the agent, not by Rekor). The only thing Rekor pins is the time of *anchor submission*, not the claimed event time.
  - (b) **The revocation event itself must be signed by the compromised key (per §2.7).** If the attacker controls the key, the attacker controls revocation publication. Doc B §2.7 acknowledges this ("What if the operator can't rotate? DID becomes 'abandoned'") but doesn't address: what if the *attacker* publishes a revocation event to lock out the legitimate operator? The legitimate operator is now in the same state as the attacker — locked out — but the attacker has had a head start.
  - **The Insurance-Pricing demo (Doc E §4 / Doc A §4.2) makes this CRITICAL.** A compromised insurer-attesting agent that writes fraudulent "no-violation" attestations during the revocation window is real money. Doc C R-A-03's HIGH-impact classification under-states it given the Doc A §4.2 monetisation thesis.
  - **Demand:** an out-of-band revocation channel **not** signed by the compromised key (e.g., a separate operator-rooted Ed25519 key, or a threshold-of-workspace-witnesses event that any witness can initiate). Plus a `signed_at_rekor_inclusion_time` field — an event whose `event.timestamp` is more than Δ before its Rekor inclusion time is flagged.

- **C-2 (Doc B §2.10, Doc C R-A-02, Doc C Q5): Witness federation roster is itself an unmodelled trust root.** Doc B §2.10's `PubkeyBundle.witness_keys` is updated via "V1.18 protected-surface PR path — bundle change is itself a trust-root mutation, signed by the operator's existing Atlas tag-signing key." Two problems:
  - (a) **Single-key trust root for federation enrolment is a Sybil/rogue-witness vector.** If a regulator-witness can be added by one operator signature (per V1.18 SSH-Ed25519 trust-root-mutation), an attacker who compromises the operator's tag-signing key can add a witness that *attests anything*. That witness's cosignatures then look legitimate to downstream verifiers using `--require-witness-kid`. The V1.18 protected-surface model assumes one operator; V2's federation expands the blast radius without escalating the gate.
  - (b) **No M-of-N threshold for federation roster mutation.** Doc B §2.10 doesn't mention threshold-signed bundle updates. Compare to The Update Framework (TUF) root-rotation: M-of-N threshold for any change to the root. V2 federation should require similar.
  - (c) **Doc C Q5 (supply-chain) doesn't cover this** — the witness federation supply chain (regulator HSM provisioning, key handoff, OOB validation per V1.18) is not enumerated as a separate attack surface.
  - **Demand:** explicit threshold-signed bundle-mutation protocol for federation-roster additions, distinct from single-operator V1.18 path. Plus a `federation_enrolment_event` kind in events.jsonl so the addition is itself part of the verifiable record, not just a PR commit.

- **C-3 (Doc B §3.3, Doc C R-L-01, Doc C Q2): Hash-as-personal-data is a real GDPR risk and Doc B's defence is too breezy.** Doc B §3.3 asserts: "Hash exists, anchor exists, signature exists → 'this event existed at time T, this content_hash existed at time T, signed by author_did'." But:
  - GDPR Art. 4(1) defines personal data as anything that can identify a natural person *directly or indirectly*. **A blake3 hash combined with auxiliary knowledge (e.g., the deleted plaintext was a known person's medical record at a known clinic at a known time) can identify the subject indirectly.** This is the position EU DPAs have taken in past cases (e.g., Hamburg DPA on hashed email addresses, 2020s).
  - The `author_did` + `timestamp` + `workspace_id` triple is itself potentially personal data when the agent is acting on behalf of an identifiable individual. Doc B §3.3 does not redact the author identity from the post-deletion record.
  - **"Signature attests this person did X at T"** survives content deletion — and is itself processing of personal data. The signature is *about* the data subject's interaction, not just the content.
  - Doc C Q2 names this question. Doc C R-L-01 Mitigation gates it on legal opinion. **But Doc B §3.3 reads as if the answer were already "hash is not PII", which is over-confident.**
  - **Demand:** Doc B §3.3 must explicitly state "this architecture is *plausibly* GDPR-compatible pending counsel opinion; if counsel rules hash-as-PII, we need a 'redact-event-entirely' mode that breaks the hash chain at that event and replaces it with a `redacted-event` placeholder whose continuity is attested by the workspace anchor, not by parent_hash linkage." Doc B should sketch this fallback mode now, not punt it.

### HIGH

- **H-1 (Doc B §2.7, Doc B §2.5, cross-cutting): Per-tenant HKDF vs per-agent DID — combined derivation not specified.** V1's HKDF derives one Ed25519 keypair per workspace (`atlas-anchor:<workspace_id>`, prefix-pinned). V2's `did:atlas:<pubkey-hash>` is per-agent. Doc B §2.7 says they're "complementary, not competing" and an event carries both `kid` (workspace) + `author_did` (agent). But it doesn't say:
  - Is the agent DID key derived via HKDF from a per-agent master secret, or generated independently per instance?
  - If derived: what's the HKDF info-string? `atlas-agent:<agent_kind>:<instance_id>`? `atlas-agent:<workspace_id>:<agent_did>`?
  - Cross-workspace agent identity (an agent that writes in workspaces A and B) — does it use the same DID across workspaces (good for passport) or different DIDs (good for per-workspace isolation)? Doc B §2.7 implies "same DID across workspaces" via `used_by_workspaces` field but doesn't address what happens to the V1 per-workspace HKDF property when the same agent key signs across workspace boundaries.
  - **Demand:** §2.7 needs a sub-section specifying the agent-key derivation function, the cross-workspace-replay defence for agent-DID-signed events (cross-workspace replay defence must remain — `workspace_id` is still folded into the signing input by V1 design, so the event hash is workspace-bound regardless of whose key signs it; this should be stated explicitly).

- **H-2 (Doc B §2.8, §2.9, §3.13): Cypher passthrough = injection + DoS surface.** `POST /api/atlas/query` accepts Cypher subset with TBD whitelist, MCP tool `query_graph` does the same. Documented restrictions: read-only verbs, query timeout 5s/30s hard cap, row count cap 1000/10000, `require_anchored: true` rewrite. Insufficient:
  - (a) **Cypher parameter injection** — the body is `{cypher: "MATCH …", params: {…}}`. If params are interpolated naively into the Cypher string anywhere in the path (server-side rewriting, audit logging, FalkorDB driver), that's a SQLi-class bug. Need explicit prepared-statement parameter binding through to FalkorDB driver, never string concat.
  - (b) **Resource exhaustion via cartesian explosions / unbounded path expansion.** A 5s timeout doesn't stop a Cypher query that triggers `MATCH (a)-[*1..]-(b)` from consuming all GraphBLAS memory before the timeout fires. Need depth caps enforced at parse-time, not at execution-time.
  - (c) **Cross-workspace data leakage via multi-graph queries.** Doc B §3.10 recommends FalkorDB native multi-graph. But Cypher passthrough on a multi-graph backend — is the `:WorkspaceA` graph isolated from the `:WorkspaceB` graph at the parse layer or only at the driver layer? FalkorDB's documentation on this is thin.
  - (d) **`require_anchored: true` rewrite is itself an injection vector** if the rewrite splices into the user's Cypher string. Must be done as an AST-level transform, not text substitution.
  - **Demand:** §2.8 explicit policy: AST-validate every Cypher query against a server-side allow-list of patterns; whitelist procedures (no `apoc.*`, no `CALL db.*`); parse-time depth cap; explicit per-graph workspace gating that does not rely on Cypher author hygiene.

- **H-3 (Doc B §2.5, Mem0g cache as a content side-channel): Embedding leakage of redacted content.** Doc B §3.3 deletes blob storage on GDPR erasure and reseeds Layers 2 + 3. But:
  - Mem0g (Layer 3) caches *embeddings* of facts (vector representations). Embeddings are not full plaintext but they are *retrievable* — published research (e.g., Morris et al. 2023 "Text Embeddings Reveal (Almost) As Much As Text") shows ~92% reconstruction of short text from embeddings using nearest-neighbour attacks against embedding-vocabulary corpora.
  - If Mem0g's embeddings of a person's medical record are cached and the cache is *reseeded* (rebuilt from Layer 2 after content deletion), the **old embeddings might persist in Mem0g's pre-deletion snapshot, backups, or cache-warm-up logs**.
  - Doc B §3.3 says "reseed Layer 3 from Layer 2" but doesn't specify: (a) what happens to the old Mem0g cache files (overwritten? archived? swapped?); (b) what happens to the embedding model's training/fine-tune corpus if Mem0g uses any embedding fine-tuning.
  - **Demand:** §3.3 GDPR procedure must include explicit secure-deletion of Layer 3 embeddings (overwrite, not just unlink); plus an audit-trail event for "embedding-erased" parallel to the content tombstone. Plus an explicit "no embedding-fine-tune on tenant data" workspace policy.

- **H-4 (Doc B §2.5, §3.2; Doc A §4.1): Private witness federation as confidentiality control — privacy property under-specified.** Doc B §3.14 says "ship Sigstore-public as default + private-federation as opt-in". Doc C R-A-02 mitigation says enterprise gets a private witness tier. But Doc B doesn't analyse:
  - (a) How many witnesses are needed in a private federation for the *cryptographic security* (vs the *confidentiality*) to be at parity with public Sigstore? Sigstore Rekor is one log with global public observation = high tamper-detection. A 3-witness private federation gives *cryptographic* multi-sig but no *public* observability — a coordinated attack on 2-of-3 witnesses suffices.
  - (b) Sybil resistance in private federation — who runs the witnesses? Same enterprise = single-trust-domain (defeats the purpose). External consortium = needs governance.
  - (c) What if a private-federation workspace is later forced (subpoena, regulator request) to *prove* its log is consistent with public Sigstore? Doc B says "verifier accepts either" — but cross-validation between private and public modes is not specified.
  - **Demand:** §3.14 needs a sub-section "Private federation security model" with explicit threshold (e.g., t=2/3 minimum, t=3/5 recommended for high-confidentiality), governance requirement (must not be all-same-organisation), and a "private-to-public bridge" mode (private log can periodically anchor its head into public Sigstore for cross-validation without leaking content).

- **H-5 (Doc B §2.10, Doc C Q-RW-2): Coerced witness cosignature has no defence — and Doc C admits it.** Doc C Q-RW-2: "What happens if a regulator's witness service is compelled (subpoena) to cosign a *false* event? Likely no defence." Doc B §2.10 says "regulator can publish their witness key + revocation history alongside their public regulatory register, making the trust chain end-to-end auditable" — but a compelled cosignature doesn't trigger revocation; the witness is *operating normally*, just signing what they were ordered to. This is a fundamental limit of the trust model and Doc B should state it. The mitigation isn't cryptographic — it's that the audit trail of *which events the regulator cosigned* is itself public and observable, so a pattern of suspicious cosignatures can be exposed.
  - **Demand:** Doc B §2.10 add a "Limitations of the trust model" subsection that explicitly names: coerced cosignature is policy-valid even if morally invalid; the only defence is post-hoc public scrutiny of cosignature patterns; Atlas does not claim to defend against state-actor coercion of a witness party.

- **H-6 (Doc C Q5, Doc B all §2.x dependencies): Supply chain — V2 adds three new code dependencies (FalkorDB, Mem0/Mem0g, Graphiti optional). SLSA L3 boundary for the verifier doesn't automatically extend.**
  - V1 ships `@atlas-trust/verify-wasm` with SLSA Build L3 provenance (per v1.19 handoff). The verifier is trust-load-bearing and SLSA-attested.
  - V2 introduces the **Atlas Projector** (new crate) as the bridge from authoritative Layer 1 to derivative Layer 2. The projector is **partially trust-load-bearing** — it does not produce trust evidence (signatures live in events), but a maliciously-compiled projector could silently drop or duplicate events when projecting. If Layer 2 is what users see in the UI, and Layer 2 was projected by a backdoored projector, the user sees the wrong graph.
  - The projector therefore must be SLSA-L3-attested and the FalkorDB/Mem0g/Graphiti supply chains must be pinned (cargo deps locked, npm deps locked, SHA-pinned releases).
  - Doc B §2 doesn't say "projector ships with SLSA L3." Doc C R-V-02 (Mem0 vendor risk) doesn't enumerate supply-chain attack on Mem0g specifically. Doc C Q5 names it but mitigation is "requires a dependency security audit of the V2 stack analogous to npm audit" — that's right but it should be a V2-α blocker, not an open question.
  - **Demand:** Add `atlas-projector` to the SLSA L3 build matrix as a V2-α requirement. Pin all FalkorDB / Mem0g / Graphiti versions in Cargo.toml / package.json with SHA-256 integrity hashes. Add `npm audit signatures` equivalent for the Rust supply chain (`cargo audit` + `cargo deny`).

### MEDIUM

- **M-1 (Doc B §2.5, Mem0g embedding model versioning): "Eventually rebuildable" Layer 3 is a weaker invariant than Doc B admits.** Doc B §2.5 says Mem0g cache is "rebuildable given pinned model versions" but acknowledges "embedding models drift; LLM summarisation is non-deterministic across model versions." Practical implication: an enterprise that rebuilds Layer 3 after 18 months may get *different* retrieval results than at write time. The agent's *retrieval* of a fact may differ from the original — even though the fact itself (Layer 1) verifies identically. **For high-stakes attestations (compliance, insurance pricing), this means an attestation made at T=0 may not be re-retrievable in the same form at T=18mo.** Doc B should be more honest that Layer 3 is "best-effort retrieval"; trust-load-bearing queries should always route to Layer 1 (with `query_provenance` returning event_uuid, then verifier).

- **M-2 (Doc B §2.8 `POST /api/atlas/query`, §2.9 `query_graph`): No rate limiting specified.** Tool surface exposed to MCP-host agents — a runaway agent loop can issue 10K Cypher queries/sec and DoS the FalkorDB. No per-DID, per-workspace, or per-IP rate limit specified. V1's write surface already has `per-workspace mutex` (per Doc B §2.6 atlas-web diagram) — read surface needs an equivalent.

- **M-3 (Doc A §4.3, Doc B §2.7, Doc C R-A-03 + §3.16 Sybil resistance): Sybil resistance is acknowledged but mitigation is non-cryptographic.** Doc B §3.16 says: "Reputation requires witness-cosignature coverage from independent witness keys" — but doesn't define how the verifier decides which witness keys are "independent". Two witnesses operated by the same entity are not independent. The system has no notion of operator-identity-of-witness. Practical implication: a sophisticated Sybil attacker spins up 10 DIDs + 10 "witnesses" (also their own keys) and cross-cosigns everything → reputation score inflates. Doc B §3.16 recommendation (c) ("reputation explicitly not a global scalar") is the right hedge but it's hand-waving — marketplaces *will* compute a scalar. The Agent Passport demo (Doc E §3) shows "1247 facts, 0 retractions, 12 unique witness cosigners" as cards on a marketplace UI — that's exactly the gameable number.

- **M-4 (Doc B §2.6 Atlas Memory Skill, Hermes-side trust): Skill-side DID private key storage is hand-waved.** Doc B §2.6 says skill "holds the Hermes-instance DID keypair locally (skill-state dir)." Skill-state dir = filesystem path = readable by any process with the same user privileges. For an agent running in a multi-tenant deployment (e.g., shared K8s pod), this is insufficient. The HSM integration (V1.10 Wave-2) exists for operator keys — extend it to agent DID keys for high-stakes use cases. Doc C R-A-03 Mitigation proactive #2 names this ("HSM integration provides infrastructure for this — extend it to agent keypairs") but Doc B §2.6 doesn't reference it.

- **M-5 (Doc B §2.10, regulator-witness Sybil within federation): The "auditor-deloitte" pattern in §2.10 example.** Doc B §2.10's example bundle lists "atlas-witness:regulator-bafin" *and* "atlas-witness:auditor-deloitte" as both federation witnesses. But these are very different trust types — one is a regulator (state authority, public-interest binding), one is a commercial auditor (paid by the operator, fiduciary duty to operator). Treating them as equal `witness_kids` blurs the trust model. A workspace operator can pay an auditor to cosign anything (commercial relationship); they cannot pay a regulator. The strict-mode `--require-witness-kid` flag at the verifier doesn't distinguish these trust classes. Suggest: separate `witness_class` field (`regulator` | `auditor` | `peer` | `internal`) so downstream tools can filter by trust class.

- **M-6 (Doc C R-O-01 Performance Overhead, security implications): Tiered anchoring "hot tier" creates an anchor-delay race for short-lived attacks.** Doc C R-O-01 proposes "hot tier — every write Ed25519-signed + hash-chained in <5ms (no Rekor submission); warm tier — batch Rekor anchoring every N seconds." But: between the "hot" signature and the "warm" Rekor batch, the event is signed but *not anchored*. An attacker who compromises a key in this window can produce events that look legitimate (signature verifies) but are not yet Rekor-anchored. The verifier in lenient mode passes them. Doc C R-A-03 (revocation lag) compounds with this: if revocation lag and anchor lag overlap, the attacker's window is doubled. Strict mode (`require_anchors`) is the defence, but Doc B doesn't require it for high-stakes use cases (insurance pricing, regulator attestation). **Demand:** insurance-pricing and regulator-attestation use cases default to strict-mode at the verifier and have a max-anchor-lag SLO (e.g., 60 seconds) below which a workspace cannot operate.

### LOW

- **L-1 (Doc B §2.5, bi-temporal mapping): `t_invalid` computation requires forward-scanning the entire event log for retraction events.** This is an O(N) per-event projector cost. At 100M events (Doc C R-O-03 scale), this is operationally expensive. Doc B should note that incremental projection with retraction-index maintenance is needed, not derived afresh each rebuild.

- **L-2 (Doc B §2.8 `GET /api/atlas/passport/:did`): Passport endpoint leaks agent activity patterns even to anonymous callers.** Doc B doesn't specify authentication on the passport endpoint. An attacker can scrape passports to discover which agents are active in which workspaces (the `used_by_workspaces` field). For confidential workspaces, this is a side-channel. Add: passport endpoint requires DID-holder consent or operator policy to disclose `used_by_workspaces`; default to redacted.

- **L-3 (Doc A §4.2 "AI-Liability Insurance Pricing", Doc E §3): The insurance demo will be filmed before V2-α ships. The numbers (1247 facts, 12 cosigners) will be real but cherry-picked from a test workspace. Doc E §3 acknowledges this. Acceptable for storytelling but the underlying property (Sybil-resistant reputation) is unproven at scale — see M-3.**

- **L-4 (Doc B §3.5 reference agent, Doc D §"Hermes"): Hermes adoption is a soft dependency, not a hard one — but the docs treat it as both at different moments.** Doc B §2.6 invests heavily in the Hermes-skill integration; Doc C R-V-03 says "MCP path compounds across framework changes." Consistent — but Doc E §1 (Multi-Agent Race demo) explicitly shows "Hermes-Agent purple accent color" in the demo. Demo invests visual brand in Hermes; should it pivot if Hermes adoption stalls? Not a security issue but a positioning fragility.

## Blinde Flecken (was wird in den docs gar nicht angesprochen)

- **B-1: Confidential agent reasoning vs. verifiable provenance trade-off.** When an agent writes "I conclude X from internal reasoning steps Y₁..Yₙ", does the chain-of-thought get hashed too, or only the conclusion? If only the conclusion, attribution to "this agent claimed X" is weakened. If the reasoning, the chain-of-thought leaks confidential prompts, system messages, and possibly user PII into the (anchored or witnessed) event. Doc B has no model for "the agent's reasoning is part of the trust evidence but is privacy-sensitive." This bleeds into GDPR (the reasoning may contain PII) and into trade secret (the operator's prompts may be proprietary).

- **B-2: Audit logs of the audit log.** Who can see *who* queried `/api/atlas/audit/:event_uuid` or `/api/atlas/passport/:did`? If a regulator's witness service is silently querying the passport of an agent before deciding whether to cosign, that's a meta-trust property that Atlas does not currently log. Compare to GitHub's audit logs of who-accessed-which-secret. Doc B §2.8 mentions audit responses but not audit-of-audit-requests.

- **B-3: Verifier side-channel resistance.** The WASM verifier runs in the auditor's browser against an untrusted page (atlas-trust.dev — operator-controlled). Doc B treats the verifier as the trust anchor but does not address: (a) timing side-channels in the WASM (does it use `subtle::ConstantTimeEq` everywhere, as V1 native does? — needs confirmation in WASM target); (b) browser-cache poisoning of the WASM bundle (the published WASM is signed via SLSA L3 attestation, but does the browser actually re-verify against the attestation on every load, or trust HTTPS+CDN?); (c) `@atlas-trust/verify-wasm` updates — what's the auditor's discipline for verifying they're running the canonical WASM?

- **B-4: Time-of-check vs time-of-use for Sigstore trust-root rotation.** SECURITY-NOTES pins `SIGSTORE_REKOR_V1.active_tree_id = 1_193_050_959_916_656_506` and `SIGSTORE_REKOR_V1.pem`. Doc C R-A-02 covers privacy concerns but Doc C R-Sigstore-Sunset (call it that) is unenumerated: what happens if Sigstore Rekor rotates its production key? V1's pin is a single point of failure for all historic anchors. Doc C Q10 names this but classifies the risk inadequately — the V1 trust property *depends on* a Linux Foundation project's continued operation, not on math. Mitigation exists (offline Merkle proof + checkpoint signature data is in every anchor) but Doc B / Doc C should make this an explicit risk row.

- **B-5: Multi-author event semantics — co-signing the same event.** Doc B §2.1 / §2.7 model "every event has one author_did." But Demo 1 (Doc E §1) shows "Hermes wrote it" and "Claude wrote it" producing two events about the same entity. What about a *single* event co-authored by two agents (negotiation outcome, multi-agent consensus, AI-human pair-programming)? The current schema doesn't support multi-author events. This is fine if we never need it, but Doc B should state the limitation.

- **B-6: Replay attacks on the read side.** A consumer presents the same provenance bundle (event_uuid, rekor_idx, signature) twice — for example, to claim "this fact is still valid as of today" when it was actually retracted. The verifier validates the signature (still valid — signature is signature). But the *current* state of the fact (retracted in a later event) is not part of the bundle. How does a downstream consumer detect "this evidence is stale"? Doc B §2.8 `audit/:event_uuid` returns retraction descendants — but the consumer has to ask. Anonymous read-side calls don't get this automatically. **Demand:** every read-side response should include a "freshness window" (the timestamp the projector last verified this fact wasn't superseded), or a freshness-token signed by the projector that the consumer can check.

- **B-7: Workspace-deletion semantics.** GDPR (R-L-01) and content-hash separation handle per-event redaction. But what about *workspace deletion*? When a customer churns, the operator wants to delete the workspace's events.jsonl + blobs + FalkorDB graph + Mem0g embeddings + agent passports for DIDs that wrote only in that workspace. Is the workspace-deletion event itself signed and chained? Where does it live (in some root workspace?). Doc B has no model for this.

- **B-8: Anchor-chain forking across workspaces.** V1.7 ships anchor-chain tip-rotation. V2 introduces multi-workspace (per Doc B §3.10 multi-graph). Does each workspace have its own anchor chain (clean isolation) or do they share a global chain (cross-workspace tip-correlation defence)? Doc B doesn't say. If shared, a workspace can affect another workspace's anchor head ordering. If isolated, cross-workspace federation queries can't use the chain head as a global timeline.

- **B-9: Schema evolution of the Algorithm enum is one-way only.** V1's `Algorithm` is `#[non_exhaustive]` — adding ML-DSA-65 is SemVer-minor, good. But: post-quantum verifier needs to handle "events signed with Ed25519 before the migration, events signed with both during transition, events signed with ML-DSA-65 only after migration." Doc C R-C-01 mitigation #2 ("hybrid sign Ed25519 + ML-DSA-65 co-signature") is correct, but Doc B §2 doesn't specify the verifier's policy for "I see only Ed25519 — is that OK?" (yes if event is pre-migration, no if post-migration). The verifier needs an `algorithm_policy_version` aware of the workspace's PQ transition timeline. This isn't in any doc.

- **B-10: Trust of the `projector_version` field.** Layer 2 nodes stamp `projector_version`. The verifier uses this to "reject nodes whose projector_version doesn't match" (R-A-01 mitigation #2). But `projector_version` is just a string property of a graph node — it's not cryptographically bound to anything in events.jsonl. A malicious or buggy Layer 2 can lie about the projector_version. The trust property here is "projector_version + events.jsonl + projector binary determines graph state" — but the user verifies *one* of those (events.jsonl) and the other two are taken on faith.

- **B-11: Witness federation when the workspace itself goes hostile.** Doc B §2.10 models "regulator-witness is added by the operator." What if the operator decides to *remove* the regulator-witness from the bundle (e.g., to escape oversight)? V1.18 protected-surface PR path means the operator can amend the bundle. The regulator's only signal is "I'm no longer being asked to cosign." There's no cryptographic property forcing the operator to keep the regulator in the roster once added. Doc B should state this is fundamentally a contractual/legal binding, not a cryptographic one (analogous to H-5).

## Konkrete Vorschläge (specific edits/additions)

- **Doc B §2.7 (Agent DID + Revocation):** Add a sub-section "**Out-of-band revocation channel**" that specifies (a) a separate operator-root Ed25519 key (not the compromised agent key) authorised to publish revocation events, OR (b) a witness-initiated revocation event that requires t-of-n witness threshold from the workspace bundle. Plus add a verifier policy `--max-anchor-lag <duration>` that rejects events whose `event.timestamp` is more than Δ before their Rekor inclusion time (defends against attacker backdating). Reference: C-1 above.

- **Doc B §2.10 (Federated Witness):** Add a sub-section "**Federation roster mutation protocol**" requiring M-of-N threshold-signed bundle updates for *witness-roster additions* (distinct from the V1.18 single-operator-key path for non-witness bundle changes). Plus introduce a `federation_enrolment_event` event kind so the addition is itself part of the events.jsonl record and any downstream verifier sees "this witness was added at time T by these M-of-N operator keys." Plus a `witness_class` field (`regulator` | `auditor` | `peer` | `internal`) on each witness entry. Reference: C-2, M-5.

- **Doc B §3.3 (GDPR):** Add a fallback sub-section "**If hash-as-PII is upheld by counsel**" sketching a `redacted-event` placeholder mode: when an event must be erased entirely (not just its content), replace the event with a `redacted-event` placeholder of the same `event_hash` (preserving the chain) but with a flag `redacted=true` and all author/timestamp/workspace fields scrubbed. The chain remains verifiable but the linkability evidence is gone. State that this is a *worst-case mitigation* that operators must explicitly opt into for high-PII workspaces. Reference: C-3.

- **Doc B §3.3 + §2.5 (GDPR + Mem0g):** Add explicit procedure for **secure embedding deletion** during GDPR erasure: (a) overwrite Mem0g cache file regions (not unlink); (b) emit `embedding-erased` event in the audit trail parallel to the content tombstone; (c) prohibit Mem0g embedding fine-tuning on tenant data by default. Reference: H-3.

- **Doc B §2.8 + §2.9 (Read-side API + MCP V2):** Add a sub-section "**Cypher safety policy**": all Cypher queries are AST-parsed and validated before driver dispatch; parameter binding is prepared-statement-only (no string interpolation); per-graph workspace isolation is enforced at the parse layer; per-DID and per-workspace rate limits (default: 100 queries/min/DID, 10K queries/min/workspace); allow-list of procedures (no `apoc.*` by default); explicit depth/expansion caps enforced at parse time, not execution time. Reference: H-2, M-2.

- **Doc C R-A-03 (Agent Identity Compromise):** Escalate severity to **CRITICAL combined** (Q4 in Doc C open questions already asks this — answer it): with LOW detectability + LOW reversibility, MEDIUM probability is sufficient for CRITICAL. Reference: C-1 + Doc C Q4.

- **Doc C: New row R-V-04 — Sigstore Rekor Trust-Root Sunset.** Description: V1's anchor verification depends on `SIGSTORE_REKOR_V1.pem` and `active_tree_id` pins. If Sigstore rotates keys, sunsets v1, or the Linux Foundation discontinues the service, all historic anchors require *offline-Merkle-proof-only* verification (which Atlas already supports per SECURITY-NOTES V1.8 mixed-mode). But operationally, new anchors stop. Probability: LOW (Linux Foundation projects are durable). Impact: HIGH (write side blocked for public anchors). Mitigation: ensure offline Merkle proof + checkpoint signature is captured in every anchor at issuance time (already done per V1.6 design); document the fallback procedure; tighten Doc C Q10 from "open" to "answered: offline proof data is in every anchor." Reference: B-4.

- **Doc C: New row R-V-05 — Atlas Projector Supply Chain.** Description: `atlas-projector` is partially trust-load-bearing (a backdoored projector could silently drop/duplicate events). Demand: SLSA Build L3 attestation parity with `@atlas-trust/verify-wasm`; pinned SHA-256 dependency integrity on FalkorDB, Mem0g, Graphiti via Cargo / npm lockfiles; `cargo audit + cargo deny` in CI; signed releases for the projector crate. Make this a V2-α blocker. Reference: H-6, Doc C Q5.

## Offene Fragen für Phase 3

- **Q-SEC-1: Should the out-of-band revocation channel for agent DID compromise (C-1) be a separate operator-root key, a t-of-n workspace-witness threshold, or both?**
  Context: §2.7 currently allows only same-key revocation, which fails closed against attacker-controlled keys. Two-channel revocation increases security but operational complexity.
  Recommendation: **accept both**. Default = same-key revocation (V1 path); operator can opt into a separate operator-root revocation key OR a workspace-witness-initiated revocation in higher-stakes workspaces. Specify in `docs/specs/did-method-atlas.md` (drafted in V2-α).

- **Q-SEC-2: Does the `federation_enrolment_event` (C-2) need to be threshold-signed by existing federation members, or is operator-signature + out-of-band public announcement sufficient?**
  Context: The first regulator-witness added to a workspace has no prior federation to threshold-sign against. Bootstrapping requires *some* initial trust gate.
  Recommendation: **defer to Phase 3 with TUF-spec-inspired design**. The Update Framework's "root-of-trust rotation" model handles this: first key is bootstrapped from out-of-band trust, subsequent changes are threshold-signed. Adapt to Atlas.

- **Q-SEC-3: Is "hash + author_did + timestamp + workspace_id" personal data under GDPR Art. 4(1)?**
  Context: Doc B §3.3 implicitly says no; EU DPAs may say yes. The answer determines whether content-hash separation is sufficient or whether `redacted-event` mode is required.
  Recommendation: **research/defer to legal counsel** (already Doc C R-L-01 blocker). Phase 3 should commission this opinion before V2-α writes any signed event under EU jurisdiction. Atlas's fallback `redacted-event` mode (per concrete edit above) should be implemented regardless, as a safety net.

- **Q-SEC-4: Should Cypher passthrough be removed entirely from the public read-side API and reserved for an operator-admin endpoint?**
  Context: H-2 enumerates Cypher injection / DoS / cross-graph leakage risks. The opinionated REST surface (entities, related, timeline, audit, passport) covers ~90% of agent use cases. Cypher passthrough is power-user expressiveness vs a real attack surface.
  Recommendation: **accept restriction** — public read-side ships REST only; Cypher passthrough is operator-admin endpoint with separate auth (mTLS, API-key, OIDC). Phase 3 confirms before V2-β.

- **Q-SEC-5: Does Mem0g need to ship with content-hash deletion hooks before Atlas's V2-α GDPR-erasure procedure works end-to-end?**
  Context: If Atlas deletes content + reseeds Mem0g, but Mem0g doesn't expose a "delete-this-embedding" API, we have no programmatic way to enforce H-3 secure-deletion of embeddings.
  Recommendation: **research** — open an issue on Mem0g GitHub before committing to V2-α GDPR claim. If Mem0g lacks the hook, we either (a) pin a fork that adds it, (b) rebuild Layer 3 entirely from scratch on erasure (slow but safe), or (c) accept a Mem0g-side GDPR caveat in the workspace docs.

- **Q-SEC-6: Should the `projector_version` field be cryptographically bound to the events it projects, not just stamped as a property?**
  Context: B-10 — `projector_version` on graph nodes is currently a trust-by-string. A cryptographic binding (signed `(projector_version, events.jsonl head_hash) → graph_state_hash` tuple, published per projector run) would tighten the invariant. Cost: a per-projector-run signature ceremony.
  Recommendation: **accept for V2-α** — projector runs emit a signed `ProjectorRunAttestation` event into events.jsonl asserting "I, projector_version X, ran against events.jsonl head H and produced graph_state_hash G at time T." The verifier can then check graph_state matches the attestation. This makes the determinism CI gate (R-A-01) part of the trust chain rather than out-of-band CI hygiene.

- **Q-SEC-7: Is the WASM verifier's bundle distribution adequately tamper-evident in browser-based verification?**
  Context: B-3 — the auditor running the WASM in a browser receives the WASM over HTTPS+CDN from atlas-trust.dev (operator-controlled). SLSA L3 provenance protects npm distribution but not browser-side CDN distribution. A malicious CDN could serve a tampered WASM.
  Recommendation: **research/defer** — investigate Subresource Integrity (SRI) hashes for the WASM script tag; investigate a "verify-your-verifier" page that compares the loaded WASM hash against the SLSA-attested npm hash. May be a Welle 14b deliverable.

- **Q-SEC-8: Should high-stakes workspaces (insurance-pricing, regulator-attestation per Doc A §4.1/§4.2) have a hard max-anchor-lag SLO below which write-side accepts no events?**
  Context: M-6 — tiered anchoring's hot-tier creates an unanchored-window. For high-stakes use cases, this window must be bounded.
  Recommendation: **accept for V2-β** — workspace policy field `max_anchor_lag_seconds: 60` (or operator-chosen); writes accepted only if the most recent anchor is within this window. Below threshold, writes queue or reject. Verifier in strict mode requires every event to be anchored within this SLO.

- **Q-SEC-9: Multi-author event support — defer or design now?**
  Context: B-5 — multi-agent consensus / negotiation outcome / human-AI pair-programming all want "one event, two signing authorities." Doc B §2.7's schema is single-author. Adding multi-author later is a schema break.
  Recommendation: **defer to Phase 3 / V2-γ** but reserve the schema field. `event.author_dids: [DID, DID, ...]` instead of `event.author_did: DID` in the V2 schema-version-v2 transition. SemVer-major break is the right time to introduce this.

- **Q-SEC-10: Workspace-deletion semantics — where does the deletion event live?**
  Context: B-7 — when a workspace is deleted, where is the "workspace deleted" attestation recorded? Some operator-level "root workspace"? A meta-log?
  Recommendation: **research/defer** — Phase 3 should sketch an operator-level audit log (per-Atlas-instance, not per-workspace) that records workspace lifecycle events (created, deleted, federation-roster-changed). This is a V2-γ scope item.

- **Q-SEC-11: Post-quantum verifier behaviour during the migration window — what's the policy for events with Ed25519-only signatures after the PQ deadline?**
  Context: B-9 — V1 events are Ed25519-only; V2 events during transition are hybrid; V2 post-migration are ML-DSA-only. The verifier needs a per-workspace `pq_migration_policy` field: "lenient" (accept Ed25519 indefinitely), "strict-pq-required-after-T" (Ed25519-only events post-T are rejected), "strict-pq-always" (only hybrid + PQ-only accepted).
  Recommendation: **accept for V2-γ design, defer implementation** — Doc C R-C-01 says "not a V2-α-blocker." Agreed. But the verifier's algorithm-policy field needs to be designed alongside the Algorithm enum extension to avoid a SemVer break later.
