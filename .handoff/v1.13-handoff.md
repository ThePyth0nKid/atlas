#### V1.13 Scope C Wave 1 — independent witness cosignature attestor (commits `a06e5ab` + `8410064`)

A second, independent attestor (the "witness") signs over the recomputed Atlas anchor-chain head with its own Ed25519 keypair. The verifier resolves the witness signature against `ATLAS_WITNESS_V1_ROSTER` (a pinned slice of `(witness_kid, [u8; 32])` pubkeys living in `crates/atlas-trust-core/src/witness.rs`). Trust-domain separation: the witness runs as the `atlas-witness` binary in a separate process from the `atlas-signer` — this is the entire point. A witness in the same process as the signer is "the same machine signed twice", not an independent attestation.

**New crate `atlas-witness`** (`crates/atlas-witness/`):
- `src/lib.rs` — `Witness` trait (dyn-safe, `Send + Sync`, `String` errors at boundary per V1.11 footgun #20).
- `src/ed25519_witness.rs` — `Ed25519Witness` impl. `Zeroizing<[u8; 32]>` secret wrapper; pubkey derived lazily on each call (no permanent in-memory copy of verifying-key bytes).
- `src/main.rs` — `atlas-witness` binary with two subcommands: `gen-key` (writes 32-byte raw secret + hex pubkey) and `sign-chain-head` (reads secret, signs hex chain head, prints `WitnessSig` JSON). Secret file created **atomically** with `OpenOptions::create_new(true).mode(0o600)` on Unix to close the TOCTOU window between `fs::write` (default umask 0o644) and operator-run `chmod 0400`. `create_new(true)` also fail-closes on accidental overwrite of an existing key file.
- `Cargo.toml` — depends on `atlas-trust-core` only (no cycle, V1.10 footgun #13).

**`atlas-trust-core` additions** (commit `a06e5ab`):
- `src/witness.rs` — `WitnessSig { witness_kid, signature }` (URL_SAFE_NO_PAD base64, matches `EventSignature.sig` dialect project-wide). `ATLAS_WITNESS_DOMAIN = b"atlas-witness-v1:"` for domain separation. `ATLAS_WITNESS_V1_ROSTER` ships **genesis-empty** — first commissioning bumps the crate version per the V1.7 boundary rule. `verify_witness_against_roster` inlines `VerifyingKey::verify_strict` directly (does NOT route through `crate::ed25519::verify_signature`, which hard-wires `BadSignature { event_id, .. }` semantics — the witness trust domain stays self-contained from the first failure point). `ct_eq_str` for kid lookup. Explicit length check on signature bytes before `Signature::from_bytes`. `verify_witnesses_against_roster` returns a `WitnessVerifyOutcome { presented, verified, failures }` so the future strict-mode caller can apply an M-of-N threshold without re-running verification.
- `src/anchor.rs` — `AnchorBatch.witnesses: Vec<WitnessSig>` (backward-compat via `serde(default, skip_serializing_if = "Vec::is_empty")`). `canonical_chain_batch_body` now runs through a `ChainHeadInput` projection that **excludes** `witnesses` from the head — preventing infinite regress (the witnesses sign the head, so the head must not depend on them). Compile-time enforcement: exhaustive destructure + `#[deny(unused_variables)]` so a future `AnchorBatch` field forces a compile-time decision (include in head or `: _` it explicitly).
- `src/verify.rs` — `witness_evidence_for_chain` walks `chain.history`, recomputes each batch's head via `chain_head_for`, runs `verify_witnesses_against_roster` against `ATLAS_WITNESS_V1_ROSTER`, and emits a `witnesses` evidence row aggregated across batches. **Lenient mode** (V1.13 wave C-1): failures surface as `ok=false` evidence rows but do NOT push to `outcome.errors` — the trace stays valid. Empty roster + no presented witnesses passes as no-op; presented-but-unverified-because-uncommissioned-roster surfaces with the per-failure breakdown.

**Wave C-1 follow-up additions** (commit `8410064`):
- `verify_witnesses_against_roster` — duplicate-`witness_kid` defence. Pre-pass counts kid occurrences in a `BTreeMap`; every occurrence of a duplicated kid (including the first) is rejected. Without this, an issuer (or attacker controlling batch serialisation) could satisfy a wave-C-2 3-of-3 threshold by attaching the same valid signature three times under one commissioned key.
- `VerifyOutcome::has_witness_failures()` — named helper returning `true` iff a `witnesses`-keyed evidence row carries `ok=false`. Wave C-2 strict-mode threshold-check must look at the witnesses evidence row, NOT at `errors` (lenient mode does NOT push to errors). Named helper > inline filter scattered at the strict-mode site.

**Trust property (lenient mode):** a valid `WitnessSig` in a trace is evidence that a second, vom Atlas-Signer unabhängiger Process the exact chain-head bytes gesehen und unterzeichnet hat. Domain-prefix prevents cross-system replay (anchor-chain sigs cannot be replayed as witness sigs and vice versa, pinned by test). Roster-manipulation at runtime is impossible (`pub const`, no env var, no file read, no cfg-gate).

**Defence-in-depth tests pinned (37 tests in `witness.rs` + 13 in `verify.rs`):** dialect mismatch (STANDARD-padded b64 hard-rejects under URL_SAFE_NO_PAD); explicit length check (63-byte sig rejected before reaching dalek); empty kid rejected; domain disjointness (`ATLAS_WITNESS_DOMAIN` ≠ `ANCHOR_CHAIN_DOMAIN`, neither is prefix of the other); duplicate-kid rejection; `has_witness_failures` is witness-specific (other failed evidence rows do not trip it).

1. **`WitnessVerifyOutcome.failures` is `Vec<String>` — should be structured.** For wave C-2's M-of-N threshold + auditor diagnostics, `Vec<(String, TrustError)>` (kid + structured error) would let the strict-mode caller filter by failure reason without parsing free-form strings. 2. **`chain_head_for` returns `String` (hex) — should be a `ChainHeadHex` newtype.** The verifier currently relies on a string-typed convention that the value is hex; a future refactor that switches to raw bytes would compile but be silently broken. 3. **`ATLAS_WITNESS_V1_ROSTER` doc should mark `verify_witness_against_roster` as test-only entry-point.** The function takes a free `roster` parameter for testability, but downstream consumers must always pass `ATLAS_WITNESS_V1_ROSTER` in the production path. Doc-only change. (`crates/atlas-trust-core/src/witness.rs`)
4. **`failures.join("; ")` in `witness_evidence_for_chain` detail is free-form.** Same argument as #1 but on the verifier side: a structured failure-list lands cleaner in JSON output for an auditor UI. (`crates/atlas-trust-core/src/verify.rs`)
5. **`gen-key` eprintln advice for the pubkey path is mildly mehrdeutig** — the operator might think `chmod 0400` is required on the pubkey too. The wave-C-2 commissioning runbook should clarify (the pubkey is non-sensitive; only the secret needs restrictive perms). (`crates/atlas-witness/src/main.rs:138`)

### Carried forward (still load-bearing in V1.13)

1. **`anchor-chain.jsonl` is append-only and irreversible** — sole writer is the issuer. (V1.7)
2. **Chain head canonicalisation must be byte-identical between issuer and verifier.** Pinned golden tests. (V1.7)
3. **Verifier may NOT trust `trace.anchor_chain.head` field** — recompute. (V1.7)
4. **Sigstore Rekor v1 shard roster is a source change** — adding tree-IDs requires crate version bump. (V1.7)
5. **Every signer-stdout boundary in TS uses `parseAnchorJson`** (lossless-json). (V1.8)
6. **`cargo` not in default PATH on Windows** — `/c/Users/nelso/.cargo/bin/cargo.exe`.
7. **Smoke at `apps/atlas-mcp-server`** — `pnpm --filter atlas-mcp-server smoke` from root.
8. **Single canonicalisation source — TS never re-implements anything cryptographic.** Always shell to `atlas-signer`. (V1.5+)
9. **HKDF info parameter is the trust boundary for per-tenant key separation.** (V1.9)
10. **Backward compat lenient mode is a trust trade-off, not a free win.** Strict mode is the security boundary. (V1.9)
11. **Don't re-derive a pubkey at verify time.** Verifier reads from the bundle. (V1.9)
12. **`std::env::set_var` in tests races on multi-core CI.** Use the V1.10 injection-form gates. (V1.10)
13. **Cargo no-cycles rule trumps clean module decomposition.** HSM lives in `crates/atlas-signer/src/hsm/` behind `--features hsm`, not in a sibling crate. (V1.10)
14. **PKCS#11 is verbose; cryptoki abstracts the syntax but not the semantics.** Test against SoftHSM2 with pinned-pubkey goldens before trusting any vendor's HKDF/EDDSA. (V1.10)
15. **Drop order matters for FFI resources.** `Mutex<Session>` BEFORE `_ctx: Pkcs11` so `C_CloseSession` runs before `C_Finalize`. (V1.10)
16. **Absolute-path guard at config-parse time, not at use time.** Platform-conditional test data on Windows vs Unix. (V1.10)
17. **Zeroize at every heap-resident copy of secret material.** PIN bytes, HKDF output, per-tenant seed array — all wrapped. (V1.10)
18. **Operator runbook ceremonies must use secrets-manager env vars, not literal PINs.** `umask 077`, source from secrets manager, `unset` at end. **wave-3 ceremony (V1.11) further upgrades this to `--pin-source env:VAR` for `pkcs11-tool` so even argv exposure is closed.** (V1.10/V1.11)
19. **Pre-flight test work pays for itself.** The V1.11 byte-equivalence test was instrumental in committing to wave-3's "explicit opt-in because pubkeys rotate" design. (V1.11)
20. **Trait dyn-safety is non-trivial and worth pinning explicitly.** `WorkspaceSigner` is `Send + Sync` + dyn-safe — no `async fn`. (V1.11)
21. **Layered fail-closed dispatchers need explicit refusal text per layer.** Each refusal site names the env var to fix. (V1.11)
22. **`derive-key` style "structurally refused" subcommands are a real defensive pattern.** Refuse at the CLI surface, not at the FFI layer. (V1.11)
23. **PIN-on-argv is unavoidable on some legacy CLIs (softhsm2-util's init-token).** Document honestly + recommend host-hardening. Don't conflate shell history with `/proc/<pid>/cmdline`. (V1.11)
24. **Section renumbering in long runbooks is reversible if anchors are textual, not numeric.** Prefer textual anchors over numeric in cross-reference comments. (V1.11)

### NEW for V1.13 (V1.12 lessons)

25. **TS-side env-var allow-lists must mirror the Rust side exactly.** When TS reads an env var that gates a Rust dispatcher, mirror the Rust truthy allow-list (`1`/`true`/`yes`/`on`, case-insensitive). A strict `=== "1"` check in TS while Rust accepts `true` produces a silent mode-mismatch where Rust activates one layer and TS logs a different layer — defeats audit signals. The `isTruthyEnv` helper in `apps/atlas-mcp-server/scripts/smoke.ts:65` is the load-bearing example. (V1.12 Scope B2)

26. **Fail-closed early at the closest boundary to the misconfiguration.** When wave-3 is opted in but the trio is partial, the Rust signer correctly refuses — but the TS smoke now also refuses, with a `(N/3 vars set)` diagnostic, BEFORE spawning any process. The closer the failure to the source of the misconfiguration, the cheaper the operator's debug loop. (V1.12 Scope B2 — `smoke.ts` partial-trio guard)

27. **CI lane promotion has a paths-filter discipline.** When promoting `workflow_dispatch`-only lanes to `pull_request:`, list every directory whose changes can affect the lane's behaviour — but no more. Over-broad filters waste CI cost and trigger spurious red on doc-only PRs. Under-broad filters miss real drift. The V1.12 Scope B1+B2 paths-filters are the reference: `crates/atlas-signer/**`, `crates/atlas-trust-core/**`, `Cargo.lock`, the workflow file itself, plus TS paths for the smoke lane. Test-only file changes are deliberately excluded — those are low-risk. (V1.12 Scope B1+B2)

28. **Fork-PR safety needs both a PIN fallback AND `permissions: contents: read`.** GitHub does not inject `secrets.*` into pull_request events from forks; the `${{ secrets.X || 'default' }}` pattern keeps the lane functional, but the literal default is not auto-masked, so `set +x` is the load-bearing PIN-leak guard. The `permissions: contents: read` block bounds the GITHUB_TOKEN scope so a fork-PR-injected workflow change cannot escalate to write capability. **Both must be present together** — neither alone is sufficient. (V1.12 Scope B1)

29. **Live-API CI lanes must be decoupled from PR triggers.** A nightly cron + manual dispatch is the right shape for any lane that hits an upstream API (Sigstore, etc.) where an outage on their side would otherwise block every PR. Concurrency group with `cancel-in-progress: true` prevents doubled submission if a manual dispatch races a scheduled run. (V1.12 Scope B2 — `sigstore-rekor-nightly.yml`)

### NEW for V1.13 wave C-2 (V1.13 wave C-1 lessons)

30. **Trust-domain separation is by-process, not by-code-path.** A second-attestor crate that runs in the SAME process as the signer is "the same machine signed twice", not an independent attestation. The whole reason `atlas-witness` is a separate binary (not a sibling module of `atlas-signer`) is so the witness signs from a host with disjoint compromise surface. A future "make it convenient" PR that pulls the witness back into the signer process silently destroys the trust property. Document at the impl boundary AND the operator-runbook ceremony. (V1.13 wave C-1 — `crates/atlas-witness/src/main.rs` doc-comment)

31. **One project-wide base64 dialect, no fallback.** `WitnessSig.signature` uses `URL_SAFE_NO_PAD` because `EventSignature.sig` already does — one dialect across the whole wire format. The first wave-C-1 draft mismatched (signer wrote STANDARD, verifier read URL_SAFE_NO_PAD); the silent-rejection bug was caught by a parallel reviewer not by the test suite. Pin a hard test that the OTHER dialect rejects (`verify_witness_padded_standard_b64_rejects`) so a future "transparently re-decode" PR cannot smuggle a fallback path back in. (V1.13 wave C-1 footgun, address in `crates/atlas-trust-core/src/witness.rs`)

32. **Inline crypto-primitive call to keep error semantics correct.** Routing `VerifyingKey::verify_strict` through a shared helper (`crate::ed25519::verify_signature`) hard-wires `BadSignature { event_id, .. }` semantics — the event-trust-domain identifier — into a witness-trust-domain failure. Even rewrapping the error after the fact leaks the wrong type-domain through one intermediate state. Inline the primitive at the witness boundary so the trust domain stays self-contained from the first failure point. (V1.13 wave C-1 — `verify_witness_against_roster` in `witness.rs`)

33. **Compile-time enforcement against silent infinite-regress.** `ChainHeadInput` is a projection struct that explicitly EXCLUDES `witnesses` (the witnesses sign over the head; if the head depended on them, the canonical bytes would be a fixed-point problem). Wave C-1 enforces this with exhaustive destructure + `#[deny(unused_variables)]` so a future `AnchorBatch` field forces a compile-time decision (include in head or `: _` it explicitly). Tests alone are insufficient — a developer adding a new field can write a passing test before noticing they need to handle ChainHeadInput. (V1.13 wave C-1 — `canonical_chain_batch_body` in `anchor.rs`)

34. **Lenient-mode failures must surface in `evidence`, not just `errors`.** Wave C-1's `witness_evidence_for_chain` emits `ok=false` evidence rows on witness failure but does NOT push to `errors` (lenient = trace stays valid; strict mode lands in C-2). The C-2 strict-mode threshold-check therefore must look at the witnesses evidence row, NOT at `errors`. Without a named helper, the strict-mode site would write fragile inline-filter code (`evidence.iter().any(|e| e.check == "witnesses" && !e.ok)`) — easy to wire against the wrong source. Wave C-1 ships `VerifyOutcome::has_witness_failures()` as the single named API for this exact question. Use it in C-2; do not write the inline filter. (V1.13 wave C-1 — `verify.rs`)

35. **Per-batch dedup is the right place for kid-uniqueness.** A wave-C-2 M-of-N threshold check that doesn't dedup `witness_kid` can be satisfied by repeating one valid signature N times under one commissioned key. Catching this at the per-batch verifier (wave-C-1 `verify_witnesses_against_roster` rejects every occurrence of a duplicated kid, including the first) keeps lenient and strict modes consistent and means the C-2 threshold check can trust `verified` already counts only kid-distinct cosignatures. The first impl draft missed this — it was only caught by the post-commit reviewer ahead of the C-2 threshold rollout. Always think one wave ahead when designing primitive trust checks. (V1.13 wave C-1 follow-up — `8410064`)

36. **File-backed secret material needs atomic create-time perms (Unix).** `fs::write(secret_path, &bytes)` opens the file with the umask default (typically 0o644) and a TOCTOU window exists where any local user can read the secret before the operator runs `chmod 0400`. The fix: `OpenOptions::create_new(true).mode(0o600).open(secret_path)` — atomic. `create_new(true)` also fail-closes on accidental overwrite of an existing key file (a typo `--out same/path` won't silently destroy a previously-generated witness key). Windows side cannot set ACLs from `std::fs` alone — use `create_new(true)` + an advisory eprintln + a runbook caveat. (V1.13 wave C-1 — `atlas-witness/src/main.rs::write_secret_file`)
