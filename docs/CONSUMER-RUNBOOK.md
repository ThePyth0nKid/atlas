# Atlas — Consumer Runbook (V1.15 + V1.17)

This document is the reference for **downstream consumers** of
`@atlas-trust/verify-wasm` — auditor tooling, in-house verification
services, CI lanes, regulator portals, and any other npm consumer
that imports the WASM verifier. It is written for the engineering
team that *uses* the package, not for the Atlas operator team that
publishes it (see [OPERATOR-RUNBOOK.md](OPERATOR-RUNBOOK.md)) and
not for engineers building Atlas itself (see
[ARCHITECTURE.md](ARCHITECTURE.md)).

The verifier is licensed Apache-2.0 and ships under two distribution
channels (V1.14 Scope E primary npmjs.org + V1.15 Welle B backup
GitHub Releases). The trust property is *byte-identical verification
regardless of input shape* and is locked in by the
byte-determinism anti-drift tests in `crates/atlas-trust-core/`.
Consumer-side reproducibility — the topic of this runbook — is the
last layer that makes that property load-bearing for *your* install,
not just for ours.

---

## 1. Why pin?

`npm install @atlas-trust/verify-wasm` without a lockfile entry is
silently a trust statement: *"npmjs.org served the same bytes today
as yesterday, and the maintainer has not pushed a new minor version
that I haven't audited."* Both halves of that statement can fail
benignly (a maintainer pushes a legitimate `1.15.1` and your CI now
runs against unaudited code) or maliciously (a compromised maintainer
token publishes a `1.15.99` that adds a backdoor). Lockfile pinning
closes both gaps.

Atlas's trust story has three integrity layers. Each layer protects
against a different threat:

| Layer | Mechanism | Threat closed |
|---|---|---|
| 1. Version pin | `package.json` `"@atlas-trust/verify-wasm": "1.15.0"` (exact, no `^` or `~`) | Maintainer pushes a higher version with valid bytes but unaudited code |
| 2. Lockfile integrity | `package-lock.json` `"integrity": "sha512-…"` (auto-generated by `npm install --save-exact`) | Registry-side compromise replaces published bytes for an existing version |
| 3. SLSA L3 provenance | `npm audit signatures` against the OIDC-signed attestation | Forged-but-byte-different tarball uploaded under the same version (provenance won't match the canonical commit SHA) |

All three are necessary; none alone is sufficient. Pinning version
without lockfile-integrity lets a registry-side attack succeed.
Lockfile-integrity without provenance lets a long-lived compromised
token publish bytes that your lockfile will then pin permanently.
Provenance without version pinning still drifts as new versions land.

For the threat model in detail see
[SECURITY-NOTES.md](SECURITY-NOTES.md) §scope-b (V1.15 Welle B —
backup distribution channel and dual-channel residual risks) and
§scope-e (V1.14 Scope E — npm publish lane and SLSA L3 provenance).

## 2. npm — `package-lock.json` pinning by version + integrity hash

### Install with exact-version pin

```bash
# Use --save-exact to write `"1.15.0"` (no caret, no tilde) into
# package.json. Without this, npm writes `"^1.15.0"` by default,
# which silently allows minor and patch upgrades on the next
# `npm install` (the lockfile pins the install you have, but the
# package.json range allows future drift).
npm install --save-exact @atlas-trust/verify-wasm@1.15.0
```

Resulting `package.json`:

```json
{
  "dependencies": {
    "@atlas-trust/verify-wasm": "1.15.0"
  }
}
```

Resulting `package-lock.json` fragment (npm v9+ lockfile format):

```json
{
  "packages": {
    "node_modules/@atlas-trust/verify-wasm": {
      "version": "1.15.0",
      "resolved": "https://registry.npmjs.org/@atlas-trust/verify-wasm/-/verify-wasm-1.15.0.tgz",
      "integrity": "sha512-…",
      "license": "Apache-2.0"
    }
  }
}
```

The `integrity` field is the load-bearing hash. It is computed by
npm at `install` time from the registry-served tarball bytes and
written into the lockfile. Subsequent installs against the same
lockfile validate the freshly-fetched tarball against this hash and
**refuse** if they don't match (with `npm ci` — see below — or with
`npm install` against an existing lockfile).

### CI: use `npm ci`, not `npm install`

```bash
# In CI: npm ci is the integrity-strict install. It:
#   - refuses if package.json and package-lock.json are out of sync
#   - validates every fetched tarball against the lockfile integrity
#     hash (sha512)
#   - fails the build on any mismatch (registry-side tamper, lockfile
#     drift, or version-skew between repo and lockfile)
# npm install is permissive and will rewrite the lockfile on
# divergence — never use it in CI.
npm ci
```

For high-assurance pipelines (financial, regulated, healthcare),
add `--ignore-scripts` to skip `preinstall`/`postinstall` hooks on
all packages — these are the canonical supply-chain attack surface
and `@atlas-trust/verify-wasm` itself does not declare any. If
your project (or another dependency) does require a `prepare`
script, run it explicitly after install:

```bash
npm ci --ignore-scripts
npm run prepare    # only if your project genuinely needs it
```

### Backup-channel install caveat

If you fall through to OPERATOR-RUNBOOK §12 (V1.15 Welle B GitHub-
Releases backup channel) and run:

```bash
npm install --ignore-scripts ./atlas-trust-verify-wasm-1.15.0-web.tgz
```

your `package-lock.json` records the local-file install:

```json
{
  "packages": {
    "node_modules/@atlas-trust/verify-wasm": {
      "version": "1.15.0",
      "resolved": "file:atlas-trust-verify-wasm-1.15.0-web.tgz",
      "integrity": "sha512-…"
    }
  }
}
```

The integrity hash is still SHA512 of the same bytes (npmjs.org and
GitHub Releases serve byte-identical tarballs by design — see
[SECURITY-NOTES.md](SECURITY-NOTES.md) §scope-b). When npm reachable
again, re-pin to the registry source by re-running:

```bash
npm install --save-exact @atlas-trust/verify-wasm@1.15.0
```

The lockfile updates back to the `https://registry.npmjs.org/...`
URL with the same SHA512 integrity, so `npm ci` runs identical
either way. Commit the relock after the outage clears.

## 3. pnpm — `pnpm-lock.yaml` pinning

### Install with exact-version pin

```bash
# pnpm respects --save-exact identically to npm.
pnpm add --save-exact @atlas-trust/verify-wasm@1.15.0
```

Resulting `pnpm-lock.yaml` fragment:

```yaml
'@atlas-trust/verify-wasm@1.15.0':
  resolution:
    integrity: sha512-…
    tarball: https://registry.npmjs.org/@atlas-trust/verify-wasm/-/verify-wasm-1.15.0.tgz
```

pnpm uses the same SHA512-base64 integrity hash format as npm, so
the hash value is identical between the two lockfile types. A team
migrating from npm to pnpm (or vice versa) can sanity-check the
hash hasn't changed across the migration.

### CI: `pnpm install --frozen-lockfile`

```bash
# --frozen-lockfile is pnpm's strict-install flag. It:
#   - refuses if pnpm-lock.yaml is missing or out of sync with
#     package.json
#   - validates every package against the lockfile integrity hash
#   - fails the build on any drift (no auto-rewrite)
pnpm install --frozen-lockfile
```

### pnpm content-addressable store

pnpm's global package store at `~/.pnpm-store/` is content-addressable
by SHA512 — installs are deduplicated globally by hash, and the
integrity check is implicit on every read from the store. This is a
second integrity layer beyond the lockfile: even if the lockfile and
the registry agree on bytes that turn out to be malicious, an existing
known-good copy in the store will still resolve from cache and skip
the registry fetch entirely.

For air-gapped or registry-isolated CI: `pnpm fetch` followed by
`pnpm install --offline --frozen-lockfile` lets the install run
against the pre-populated store with zero registry traffic.

## 4. Bun — `bun.lockb` pinning

### Install with exact-version pin

```bash
# Bun pins exact versions by default — no --save-exact equivalent
# needed. The package.json line lands as "1.15.0" (no caret).
bun add @atlas-trust/verify-wasm@1.15.0
```

`bun.lockb` is a binary lockfile, not human-readable. To inspect
the resolved dependency tree:

```bash
bun pm ls
# my-project@0.0.0 /path/to/cwd
# └── @atlas-trust/verify-wasm@1.15.0
```

`bun pm ls` shows the resolved version but not the integrity hash
or registry URL — those are stored in the binary lockfile but not
surfaced by the tree printer. To audit integrity-hash equivalence
against an npm or pnpm install, the cleanest path is to do the
canonical pin in npm (which writes a human-readable
`package-lock.json` with the SHA512), record the integrity value,
then sanity-check against a Bun-side install — Bun and npm both
fetch from `registry.npmjs.org` and the SHA512 of the served
tarball is identical regardless of which client wrote the
lockfile.

### CI: `bun install --frozen-lockfile`

```bash
# Strict-install flag, same semantics as npm ci / pnpm
# --frozen-lockfile.
bun install --frozen-lockfile
```

### Caveat — Bun npm-registry tarball URL

As of Bun 1.x, `bun.lockb` records `tarball:` URLs against
`registry.npmjs.org`. If you run `bun install` from a private
registry mirror, the recorded URL will point at the mirror — which
means the lockfile is mirror-locked, not registry-locked. For
maximum portability between primary and backup channels, run the
canonical `bun add` against npmjs.org (not a corporate mirror) so
the lockfile entry matches what other consumers will see.

## 5. Verifying SLSA L3 provenance with `npm audit signatures`

### What the attestation proves

Every `@atlas-trust/verify-wasm@<version>` published from V1.14
Scope E onwards carries an OIDC-signed Sigstore-rekor attestation
linking the published bytes to:

- the GitHub repository (`github.com/ThePyth0nKid/atlas`),
- the commit SHA the workflow ran against (`<sha>`),
- the workflow file path (`.github/workflows/wasm-publish.yml`),
- the workflow run ID,
- the GitHub Actions runner identity.

A consumer who verifies the attestation gets cryptographic proof
that the tarball your `node_modules` resolved from was built from
the canonical Atlas repo at a known commit on a known runner — not
from a typosquat, a hijacked maintainer machine, or a long-lived
compromised token.

### Run the verification

```bash
# Verifies every package in the lockfile that has a publish-time
# provenance attestation. Atlas's @atlas-trust/verify-wasm is one
# of them; other dependencies in your tree may or may not be.
# Requires npm >= 9.5.
npm audit signatures
```

Successful output:

```
audited 1 package in <duration>

1 package has a verified registry signature
1 package has a verified attestation
```

Failed output (paraphrased — npm prints the specific failure):

```
audited 1 package in <duration>

1 package has a missing or invalid attestation
  @atlas-trust/verify-wasm@1.15.0
```

A missing-or-invalid attestation on `@atlas-trust/verify-wasm`
means: STOP, do not deploy. Possible causes (in decreasing
likelihood):

1. You're running an older npm version (pre-9.5) that doesn't
   speak the attestation API. Upgrade npm and re-run.
2. The version you pinned predates V1.14 Scope E (provenance was
   added in that scope). Upgrade to a post-Scope-E version.
3. **Transient outage of the Sigstore Rekor transparency log or
   the npm attestation endpoint.** `npm audit signatures` requires
   round-trips to both; a partial outage of either surface can
   produce a missing-or-invalid result against an artefact that is
   in fact correctly signed. Re-run after a few minutes; check
   <https://status.sigstore.dev> and <https://status.npmjs.org> if
   the failure persists. This is the most common false-alarm
   path — rule it out before escalating.
4. A registry-side compromise replaced the tarball but cannot
   forge the OIDC attestation against the canonical repo's
   identity.

**Only after causes 1-3 are ruled out**, treat the failure as a
security incident: escalate to security@your-org and to
nelson@ultranova.io (Atlas operator side). Don't downgrade the
package or skip the verification — fall through to the §6
reproduce-from-source path while the incident is investigated.

### When to verify

| Trigger | Why |
|---|---|
| Every CI run (recommended) | Catches a registry-side tamper window between your last verification and now. Cheap (~1s on cached lockfile). |
| On lockfile bump (mandatory) | Catches an attacker-pushed version that landed without a corresponding repo commit. |
| Quarterly audit (mandatory) | Defends against rare cases where a verified package was silently unpublished or re-published with a different attestation. |

For high-assurance pipelines (financial, regulated, healthcare —
the audit context Atlas is built for), running `npm audit signatures`
on every CI install is the right default. It is fast, has zero false
positives in normal operation, and a real failure is the canonical
class of supply-chain incident.

## 6. Reproduce-from-source fallback

When **both** distribution channels are unreachable (npmjs.org +
GitHub Releases simultaneously down or compromised — a rare but
possible scenario, especially under coordinated-attack threat
models), the ultimate trust root is verifier-side reproducibility
from source. The `@atlas-trust/verify-wasm` build is fully
deterministic from a tagged commit + the pinned `wasm-pack`
version: cloning the repo and running `wasm-pack build` locally
yields a byte-identical tarball to the published artefact.

This is the trust root that does not depend on either registry.

### Steps

```bash
# 1. Clone the repo at the tagged release commit. Replace ${VERSION}
#    with the tag matching the WASM verifier version you intend to
#    install (e.g. v1.15.0, v1.17.0, …).
VERSION=v1.17.0   # ← set to the release you want to verify
git clone https://github.com/ThePyth0nKid/atlas.git
cd atlas
git checkout "${VERSION}"

# 2. Verify the tag is signed by an authorised maintainer key
#    (V1.17 Welle B onwards — tag-signing enforcement, SECURITY-NOTES
#    §scope-l). The repo ships a trust root at `.github/allowed_signers`
#    and a verification helper that returns 0 iff the tag is
#    cryptographically signed by an SSH key in that file.
bash tools/verify-tag-signatures.sh "${VERSION}"
# Expected output (with ${VERSION} substituted):
#   Verifying 1 tag(s) against .../.github/allowed_signers ...
#   ---
#     PASS: ${VERSION}
#   ---
#   PASS: 1 / 1    FAIL: 0 / 1
#
# Cross-check: also inspect the commit SHA and confirm against the
# SLSA provenance attestation if you have a previously-verified
# install on file.
git log -1 --format=%H

# 3. Install the pinned wasm-pack version from crates.io with
#    --locked (uses the workspace's Cargo.lock for reproducible
#    transitive deps). The pin lives in
#    `.github/workflows/wasm-publish.yml` as the `WASM_PACK_VERSION`
#    env var.
WASM_PACK_VERSION=$(grep "WASM_PACK_VERSION:" .github/workflows/wasm-publish.yml \
  | head -1 | cut -d"'" -f2)
echo "Pinned wasm-pack version: ${WASM_PACK_VERSION}"
cargo install wasm-pack --version "${WASM_PACK_VERSION}" --locked

# 4. Build both targets exactly as the workflow does. Same flags,
#    same target, same release profile.
wasm-pack build crates/atlas-verify-wasm \
  --target web --release --out-dir pkg-web
wasm-pack build crates/atlas-verify-wasm \
  --target nodejs --release --out-dir pkg-node

# 5. Apply the same npm scope + publishConfig as the published
#    package. Note the quoted `="public"` form — this matches what
#    `.github/workflows/wasm-publish.yml` does. Without quotes,
#    older npm versions can write the value as a bare token rather
#    than the JSON string `"public"`, which causes `npm publish` /
#    pack to misinterpret access scope. Always quote.
for dir in crates/atlas-verify-wasm/pkg-web crates/atlas-verify-wasm/pkg-node; do
  (cd "${dir}" && npm pkg set name="@atlas-trust/verify-wasm" \
                           publishConfig.access="public")
done

# 6. Pack and compare. The resulting tarballs are byte-identical to
#    the published artefacts (provided the toolchain matches —
#    Rust stable + the pinned wasm-pack).
(cd crates/atlas-verify-wasm/pkg-web && npm pack)
(cd crates/atlas-verify-wasm/pkg-node && npm pack)

# 7. SHA256 these locally-built tarballs and compare to a
#    previously-verified install's lockfile integrity (after base64
#    decoding the SHA512 — different algorithms but same bytes), or
#    to a previously-downloaded tarball-sha256.txt manifest from a
#    GH-Release. Bytes equal ⇒ your local rebuild proves the same
#    artefact you've been running. Bytes differ ⇒ either the
#    toolchain pin diverged (re-check WASM_PACK_VERSION + rust-
#    toolchain) or the previously-trusted artefact was tampered.
sha256sum crates/atlas-verify-wasm/pkg-*/*.tgz

# 8. Install from the local tarball — same as the GH-Release
#    backup-channel flow. --ignore-scripts as defence-in-depth.
npm install --ignore-scripts \
  /path/to/atlas/crates/atlas-verify-wasm/pkg-web/atlas-trust-verify-wasm-1.15.0.tgz
```

### Caveats

- **Toolchain reproducibility is not absolute.** `wasm-pack` build
  output can vary across host architectures, libc versions, and
  Rust toolchain patch releases — even with `--locked`. The
  workflow runs on `ubuntu-latest` with
  `dtolnay/rust-toolchain@<sha> toolchain: stable`. Note: the
  `@<sha>` SHA pin pins *the action implementation*, not the Rust
  compiler — `toolchain: stable` resolves to whatever rustc release
  is current on rust-lang.org at workflow run time, and floats
  forward across Rust releases. For an exact compiler match, the
  consumer side must use the same `toolchain: <exact-version>` (e.g.
  `1.83.0`) the publish run used; the published tarballs do not
  carry the rustc version in their metadata, so this requires
  cross-referencing the workflow run logs (visible at
  `https://github.com/ThePyth0nKid/atlas/actions`). A byte-for-byte
  comparison against the published tarball is the strongest check;
  a SHA512-of-the-VerifyOutcome-on-the-bank-demo-trace check is a
  weaker but still useful check (the verifier output is structurally
  byte-deterministic per the V1.5
  `signing_input_byte_determinism_pin`, even if the WASM module
  itself differs in toolchain-dependent ways).
- **`cargo install wasm-pack --locked` depends on crates.io.**
  The reproduce-from-source path needs `crates.io` reachable for
  the `wasm-pack` install (and transitively for any
  `wasm-pack`-pulled deps). If both npm + GH-Releases + crates.io
  are simultaneously down, the trust root collapses to a
  pre-cached `wasm-pack` binary on the consumer machine. High-
  assurance pipelines should cache the resolved `wasm-pack`
  binary alongside the offline Atlas clone (next bullet).
- **This path requires the repo to be reachable.** If
  `github.com/ThePyth0nKid/atlas` is itself unreachable (e.g.
  GitHub down + an attacker has DDoS'd a mirror), you need a
  pre-cloned local copy. A security-conscious team running
  high-assurance pipelines should keep an offline clone of the
  Atlas repo at every consumed tag, refreshed on each lockfile
  bump. The repo is small (~10 MiB) and trivial to mirror.
- **The Apache-2.0 license is the foundation.** Reproducing from
  source is a load-bearing trust property *because* the verifier
  crates are Apache-2.0 — no Sustainable Use License obligations
  travel with the verifier. See
  [SECURITY-NOTES.md](SECURITY-NOTES.md) "Crate boundary".

## 9. Auto-Verify CI Action (V1.17 Welle A — `verify-wasm-pin-check@v1`)

Sections 1–8 describe the consumer-side trust ceremony as a sequence
of manual `npm install --save-exact` + `npm audit signatures`
commands. That ceremony is correct, but it has a structural failure
mode: a consumer follows it once at install time, then forgets to
re-verify on every CI run, leaving a silent window where a
maintainer-token-compromise can land between two installs and only be
caught at the next manual quarterly audit.

V1.17 Welle A closes that gap with a one-line GitHub Action:

```yaml
- uses: ThePyth0nKid/atlas/.github/actions/verify-wasm-pin-check@v1.17.0
```

### What the action asserts on every CI run

The action runs all three §1 trust layers automatically and fails the
build on any drift:

1. **Layer 1 — Version pin.** Reads `package.json`, rejects any
   `^1.15.0`, `~1.15.0`, `>=`, `||`, `*`, `latest`, `next`, `x`,
   `workspace:`, `file:`, `link:`, `git+`, `github:`, `http:`, or
   `https:` value. Bare semver only (`1.15.0` / `1.15.0-rc.1` /
   `1.15.0+sha.abc`). Checks `dependencies`, `devDependencies`,
   `peerDependencies`, and `optionalDependencies`.
2. **Layer 2 — Lockfile integrity.** Reads the lockfile (auto-detected:
   `package-lock.json` / `pnpm-lock.yaml` / `bun.lockb` / `bun.lock`)
   and asserts every entry for the package has an `integrity` field
   whose hash prefix is `sha512-` / `sha384-` / `sha256-`. `sha1-` and
   `md5-` are HARD FAIL (collisions are practical and don't defend
   against registry-side replacement). Validates the `resolved` URL
   origin (canonical `registry.npmjs.org` is silent-OK; corporate-
   mirror HTTPS hosts emit a warning so the integrity hash being
   mirror-served is visible; `file:` is the V1.15 Welle B
   backup-channel state — WARN by default, HARD FAIL with
   `fail-on-local-file: true`).
3. **Layer 3 — SLSA L3 provenance.** Runs `npm audit signatures` and
   asserts (i) the command exits 0, (ii) the output reports a
   `verified attestation` line, and (iii) the package is not
   mentioned in any failure context. Retries on transient Sigstore
   Rekor / npm attestation endpoint outages with exponential backoff
   (10s / 30s / 90s, ~2 min total on 3 retries).

### Minimal usage

```yaml
# .github/workflows/ci.yml
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'   # ships npm >= 10, well past the npm >= 9.5 attestation threshold
      - run: npm ci             # consumer's own install step — action does not install anything itself
      - uses: ThePyth0nKid/atlas/.github/actions/verify-wasm-pin-check@v1.17.0
```

### Full input surface

```yaml
- uses: ThePyth0nKid/atlas/.github/actions/verify-wasm-pin-check@v1.17.0
  with:
    package-name: '@atlas-trust/verify-wasm'   # default
    expected-version: '1.15.0'                  # if set, must match exactly
    package-manager: 'auto'                     # auto | npm | pnpm | bun
    working-directory: '.'                      # default — repo root; set per-package for monorepos
    skip-provenance: 'false'                    # set 'true' ONLY for npm < 9.5 or scheduled outage acceptance
    fail-on-local-file: 'false'                 # set 'true' to refuse V1.15 Welle B backup-channel installs once npm is reachable
    provenance-retries: '3'                     # exponential backoff: 10s / 30s / 90s on transient Sigstore outages
```

### Recommended pinning posture (consumer side)

For high-assurance pipelines, pin the action itself to a **commit
SHA**, not the moving `@v1.17.0` tag — this is the same defence-in-
depth the action enforces on the package it verifies:

```yaml
- uses: ThePyth0nKid/atlas/.github/actions/verify-wasm-pin-check@<full-40-char-sha>
```

### Composition with `npm ci` / `pnpm install --frozen-lockfile`

The action **does not** run `npm install` itself. It is a read-only
post-install verifier. Run your own install step first (typically
`npm ci` / `pnpm install --frozen-lockfile` / `bun install
--frozen-lockfile`) and then invoke the action. The separation is
deliberate: install steps are user-side concerns (caching, registry
config, lockfile regeneration), and bundling install with
verification would obscure responsibility on failure.

### Why a composite action (not a TypeScript or Docker action)

The action ships as a pure-bash composite (every step `shell: bash`
against per-script files in `.github/actions/verify-wasm-pin-check/scripts/`),
not a TypeScript bundle. This is deliberate: a TypeScript action would
become its own dependency tree that itself needs auditing — a supply-
chain-hardening action that introduces supply-chain risk would defeat
the point. The composite is reviewable in one scroll, with no opaque
`dist/index.js` and no transitive npm deps.

### Self-test workflow

`.github/workflows/verify-wasm-pin-check-self-test.yml` exercises the
action on every push / PR that touches it, plus a weekly cron at
`Mon 06:17 UTC` to catch live Sigstore regressions independent of
code changes. Four jobs:

1. `fixture-unit-tests` — 16 synthetic fixtures, Layers 1 + 2 only,
   no network (~30s).
2. `action-fixture-invocation` — invokes the composite action via
   `uses:` against each Layer-2-passing fixture; validates input
   plumbing.
3. `action-negative-cases` — invokes the action against bad fixtures,
   asserts each fails with the expected exit code (`continue-on-error:
   true` + outcome assertion in the next step).
4. `live-install-layer-3` — `npm install --save-exact
   @atlas-trust/verify-wasm@latest` then runs the full action with all
   three layers and a real Sigstore round-trip.

### When to skip a layer

Don't, in production CI. Three legitimate reasons exist:

- `skip-provenance: true` for npm < 9.5 (no attestation API). Upgrade
  npm instead — `actions/setup-node@v4` ships ≥10.
- `skip-provenance: true` for a known-scheduled Sigstore /
  rekor.sigstore.dev outage. Re-enable as soon as the outage closes.
- The `fail-on-local-file: true` toggle is the OPPOSITE — it
  *increases* strictness, not decreases it (rejects backup-channel
  installs after npm reachability is restored).

## Reporting issues

Verifier vulnerabilities — bypasses, signature-acceptance bugs,
canonicalisation drift, side-channel leaks, supply-chain incidents
involving the npm package or the GH-Release backup channel — should
be disclosed privately to nelson@ultranova.io. We will respond
within 48 hours.

A vulnerability that lets a forged trace verify as `valid: true` is
the worst possible class of bug for this project; we take such
reports seriously and will publish a fix + advisory on a
co-ordinated timeline.
